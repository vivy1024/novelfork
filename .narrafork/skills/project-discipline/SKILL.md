# 项目纪律 — 从 NovelFork 82 个 spec 的教训中提炼

> 本 skill 在以下场景触发：开始新功能、创建新 spec、讨论重构方案、准备发版时。
> 目的：防止重复犯已知错误。

---

## 六条铁律

### 1. 不做骨架，只做完整功能

**错误模式**：先做 UI 骨架/API 骨架，"后面再接真实实现"。结果假实现被当作完成，永远不会被填充。

**规则**：
- 一个功能要么完整实现（前端+后端+验证），要么不开始
- 禁止"先占位后填充"的开发方式
- typecheck 通过 ≠ 功能完成。必须 Browser 验证或 curl 实测
- 如果一个功能需要 3 天才能完整实现，就花 3 天，不要花 1 天做骨架然后标记完成

### 2. 需求驱动，不对标驱动

**错误模式**：发现一个参考产品就全面对标，产生一批新 spec，上一批还没做完。

**规则**：
- 参考产品只用来学习设计思路，不产生"对齐清单"
- 新功能必须回答：用户（薛小川）现在写小说时遇到了什么具体问题？
- 禁止创建"XXX-parity"类型的 spec
- 一次只做一个方向，做完再看下一个

### 3. 修复优于重写

**错误模式**：遇到问题就重写整个模块。前端重写 4 次、Provider 重写 4 次、管线重写 3 次。

**规则**：
- 重写的门槛：必须证明修复成本 > 重写成本（用具体代码行数和影响范围论证）
- 如果一个模块已经重写过 1 次，第 2 次重写需要用户明确批准
- 重写前必须列出：旧代码哪些部分可以保留、哪些必须删除、新代码的完成标准是什么
- "这次方向对了"不是重写的理由——每次重写都这么说

### 4. 一个 spec 做完再开下一个

**错误模式**：同时有 17 个活跃 spec，每个完成 30-70%，没有一个真正完成。

**规则**：
- 活跃 spec 最多 3 个（1 个主线 + 2 个小修复）
- 新 spec 创建前，必须先归档或关闭一个现有 spec
- spec 的完成标准在创建时就写死，不允许中途扩大范围
- 如果一个 spec 超过 2 周没有进展，要么关闭要么降级为"远期"

### 5. 版本号有意义

**错误模式**：一周发 20 个版本，语义化版本失去意义。

**规则**：
- patch：修复 bug，不改行为
- minor：新功能，用户可感知
- major：破坏性变更
- 不为"内部重构"单独发版
- 一天最多发 1 个版本（除非是紧急 hotfix）

### 6. 完成 = 用户能用

**错误模式**：typecheck 通过就标记完成。实际打开软件用不了。

**规则**：
- 前端改动：必须 Browser 截图验证
- 后端改动：必须 curl 或实际对话验证
- 工具改动：必须在对话中让 Agent 调用一次验证
- "完成"的定义：用户（薛小川）打开软件，按正常流程操作，功能正常工作

## 决策检查清单

开始任何新工作前，过一遍：

- [ ] 这个功能解决的是用户的真实问题，还是"对标"产生的需求？
- [ ] 我能在一个会话内完整实现并验证吗？如果不能，拆分到什么粒度可以？
- [ ] 现有代码能修复吗？还是必须重写？如果重写，旧代码删干净了吗？
- [ ] 当前有几个活跃 spec？超过 3 个了吗？
- [ ] 这个改动完成后，我怎么验证它真的能用？

## 技术债清单（已知，待处理）

| 债务 | 来源 | 影响 | 处理优先级 |
|------|------|------|-----------|
| CLI 包（packages/cli） | InkOS 遗产 | 无人使用，增加维护成本 | 低（不影响功能） |
| PipelineRunner 类 | InkOS 遗产 | HTTP API 依赖它，和 Agent 工具层重复 | 中（功能重复但不阻塞） |
| core/registry/builtin-tools.ts | InkOS 遗产 | Pipeline agent loop 专用，对话模式不用 | 低 |
| 60+ 归档 spec | 历史积累 | 占磁盘空间，偶尔造成混淆 | 低 |
| 旧前端残留组件 | 多次重写遗留 | 增加 bundle size | 低 |

## 何时触发本 skill

- 用户说"做一个新功能"→ 检查清单
- 用户说"重构/重写"→ 铁律 3
- 用户说"对标 XXX"→ 铁律 2
- 准备创建新 spec → 铁律 4
- 准备发版 → 铁律 5
- 标记任务完成 → 铁律 6
