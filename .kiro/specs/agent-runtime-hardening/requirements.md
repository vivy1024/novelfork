# Agent 运行时加固 — 需求文档

## 概述

为 NovelFork Studio 的 Agent 运行时增加四项生产级加固能力，使长时间写作会话更稳定可靠。

## 背景

当前 Agent 运行时已具备基本的 turn 循环、工具执行、权限管道和上下文压缩能力。但在以下场景中仍有明显不足：

1. 长写作会话的上下文溢出处理不够智能
2. YOLO 模式下缺乏对危险操作的自动安全判断
3. 服务器重启或用户中断后无法恢复正在执行的 turn
4. Agent 陷入循环或空转时缺乏有效检测和干预

## 功能需求

### FR-1：上下文压缩增强

#### FR-1.1：级联压缩（自动）

- **FR-1.1.1**：当历史消息总 token 超过摘要模型上下文窗口的 80% 时，自动触发级联压缩
- **FR-1.1.2**：将历史消息按时间分块，每块不超过摘要模型窗口的 60%
- **FR-1.1.3**：从最旧的块开始，逐块调用摘要模型生成摘要
- **FR-1.1.4**：每块摘要替换原始消息，作为后续块的前置上下文
- **FR-1.1.5**：最近的消息保持原样不压缩
- **FR-1.1.6**：压缩过程异步执行，不阻塞当前 turn
- **FR-1.1.7**：压缩失败时回退到现有的紧急截断机制
- **FR-1.1.8**：复用现有 full-compact 的摘要 prompt 格式

#### FR-1.2：分段压缩（手动）

- **FR-1.2.1**：用户可在对话框右键某条消息，选择"压缩到此消息前"
- **FR-1.2.2**：将该消息之前的所有历史压缩为一条摘要消息
- **FR-1.2.3**：调用当前 session 的摘要模型执行压缩
- **FR-1.2.4**：压缩后原始消息标记为 collapsed，不物理删除
- **FR-1.2.5**：支持 undo 操作（恢复原始消息，移除摘要）
- **FR-1.2.6**：压缩过程中前端显示进度指示

### FR-2：YOLO 模式 + 安全反思

#### FR-2.1：YOLO 模式

- **FR-2.1.1**：用户可在设置中开启 YOLO 模式（全自动执行）
- **FR-2.1.2**：YOLO 模式下，safe 级别工具（Read/Glob/Grep/WebSearch/WebFetch）自动放行
- **FR-2.1.3**：YOLO 模式下，write 级别工具（Write/Edit/Bash 非危险命令）自动放行
- **FR-2.1.4**：YOLO 模式下，dangerous 级别工具触发安全反思流程
- **FR-2.1.5**：非 YOLO 模式下行为与现有权限管道一致

#### FR-2.2：安全反思

- **FR-2.2.1**：当 YOLO 模式下遇到 dangerous 操作时，使用当前对话模型（带完整上下文）进行二次确认
- **FR-2.2.2**：反思 prompt 包含：操作详情 + 当前任务上下文 + 要求判断是否合理
- **FR-2.2.3**：模型判定 APPROVE → 自动执行，不打扰用户
- **FR-2.2.4**：模型判定 REJECT → 暂停，推送 safety-pause 事件到前端，由用户最终决定
- **FR-2.2.5**：反思调用超时（15s）时，暂停让用户决定
- **FR-2.2.6**：安全反思开关可独立于 YOLO 模式关闭（关闭后 dangerous 操作直接暂停让用户确认）
- **FR-2.2.7**：危险命令模式列表包括但不限于：rm -rf、git push --force、DROP TABLE、git reset --hard、git clean -f

### FR-3：中断恢复

#### FR-3.1：服务器重启恢复

- **FR-3.1.1**：Agent turn 循环中，每完成一次工具执行后，将当前 turn 状态写入 SQLite（checkpoint）
- **FR-3.1.2**：Checkpoint 内容包括：sessionId、turnId、completedToolResults、pendingToolCalls、lastAssistantMessage、step
- **FR-3.1.3**：Checkpoint 写入异步执行，不阻塞主循环
- **FR-3.1.4**：Checkpoint 写入失败不影响正常执行
- **FR-3.1.5**：服务器启动时扫描未完成的 checkpoint
- **FR-3.1.6**：恢复时将已完成的工具结果作为历史消息注入，重新调用 LLM 继续
- **FR-3.1.7**：恢复失败时标记 session 为 interrupted 状态
- **FR-3.1.8**：Turn 正常完成后清除对应 checkpoint

#### FR-3.2：手动中断后继续

- **FR-3.2.1**：用户中断时，已完成的工具结果正常保存到消息历史
- **FR-3.2.2**：前端在中断后的消息末尾显示"继续"按钮
- **FR-3.2.3**：点击"继续"发送 session:continue 消息到后端
- **FR-3.2.4**：后端收到后，将中断时的部分结果作为上下文，重新启动 agent turn
- **FR-3.2.5**：恢复后 Agent 看到的上下文：之前的对话 + 已完成的工具结果 + "用户要求继续未完成的任务"

### FR-4：流式中断检查加强

#### FR-4.1：循环模式检测

- **FR-4.1.1**：检测最近 N 次工具调用中，相似度 > 80% 的调用是否超过 3 次
- **FR-4.1.2**：相似度计算基于工具名 + 输入参数的结构相似性（不要求完全相同）
- **FR-4.1.3**：触发时注入系统提示"你似乎在重复类似操作，请换个方法"

#### FR-4.2：Token 消耗异常检测

- **FR-4.2.1**：追踪单次 turn 的累计 token 消耗
- **FR-4.2.2**：当累计消耗超过 context window 的 50% 时触发警告
- **FR-4.2.3**：触发时注入系统提示"已消耗大量 token，请总结进展并决定是否继续"

#### FR-4.3：连续失败升级

- **FR-4.3.1**：保留现有 2 次失败提示换方法的行为
- **FR-4.3.2**：3 次失败提示停下来调查
- **FR-4.3.3**：5 次连续失败强制停止 turn（emit turn_failed）

#### FR-4.4：空转检测

- **FR-4.4.1**：检测连续 3 次工具调用结果都是空/无变化
- **FR-4.4.2**：触发时注入系统提示"最近几次操作没有产生有效结果，请重新评估方法"

#### FR-4.5：配置项

- **FR-4.5.1**：循环检测灵敏度（相似度阈值，默认 80%）
- **FR-4.5.2**：Token 消耗警告阈值（默认 context window 的 50%）
- **FR-4.5.3**：最大连续失败次数（默认 5）
- **FR-4.5.4**：所有配置项可在设置页调整

## 非功能需求

- **NFR-1**：所有新增模块不得影响现有 agent turn 的正常执行性能
- **NFR-2**：压缩和反思的 LLM 调用失败不得导致主流程崩溃
- **NFR-3**：Checkpoint 写入延迟不超过 50ms
- **NFR-4**：前端新增 UI 元素需支持中英文国际化
- **NFR-5**：所有新增模块需有单元测试覆盖核心逻辑
