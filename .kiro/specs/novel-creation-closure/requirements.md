# 小说创作流程闭环 — 需求文档

## 背景

NovelFork 的 Agent 工具链、运行时健壮性、对话体验已对齐 Claude Code 水平。但小说创作的核心流程仍有断裂：建书后经纬是空壳、用户无法手动管理经纬条目、部分写作工具 API 返回 404、模板市场内容不全。

本 spec 聚焦"让用户能完成从建书到写作的完整闭环"。

---

## Phase 1：建书流程接通（P0）

### 1.1 向导完成后调用 architect 生成经纬

**现状**: 新书向导 11 题完成后只写入 `*(待 AI 生成)*` 占位符，不调用 AI
**目标**: 向导完成后自动调用 architect agent 生成完整经纬内容

**实现方案**:
- 向导提交后（`/api/books/:id/guided-setup` 或类似接口），在后端触发 `architect.generateFoundation()`
- architect 基于向导答案生成 5 个文件：
  - `story_bible.md` — 世界观/角色/力量体系/核心设定
  - `volume_outline.md` — 第一卷大纲（5-10 章规划）
  - `book_rules.md` — AI 写作规则约束
  - `current_state.md` — 当前进度状态
  - `pending_hooks.md` — 初始伏笔种子
- 生成过程中前端显示进度（"正在生成经纬..."）
- 生成失败时保留空壳，提示用户可稍后重试
- 如果 AI 供应商未配置，跳过生成，保留占位符

### 1.2 向导答案传递给 architect

**现状**: 向导答案存在 book metadata 中但 architect 不读取
**目标**: architect 的 prompt 中包含向导 11 题的答案作为创作方向

**实现方案**:
- 从 book metadata 中提取向导答案（genre/premise/protagonist/goldenFinger/world/powerSystem/tone/writingPhilosophy/platform/chapterLength/aiTaste）
- 格式化为 architect prompt 的输入上下文
- architect 基于这些答案生成符合用户意图的经纬内容

## Phase 2：经纬条目可编辑（P0）

### 2.1 资源树右键菜单

**现状**: 资源树中经纬条目只能点击打开，不能新建/删除/重命名
**目标**: 右键菜单支持：新建条目、重命名、删除、移动到其他分区

**实现方案**:
- 资源树节点添加右键菜单（ContextMenu 组件）
- 菜单项：新建子条目、重命名、删除（确认）、复制
- 新建条目时弹出对话框：输入标题 + 选择分区（设定/大纲/状态/规则/伏笔）
- 调用后端 API 创建/删除/重命名文件
- 操作后刷新资源树

### 2.2 新建经纬条目入口

**现状**: 没有明确的"新建经纬条目"按钮
**目标**: 经纬资料分区旁边有 + 按钮，点击新建条目

**实现方案**:
- 资源树中每个分区节点（设定/大纲/状态/规则/伏笔）旁边显示 + 图标
- 点击后弹出新建对话框
- 新建的条目自动在画布中打开编辑

### 2.3 空态教学

**现状**: 空分区只显示空白
**目标**: 空分区显示功能说明 + 操作入口

**实现方案**:
- 当分区下无条目时，画布区显示引导卡片：
  - 说明该分区用途
  - 提供"新建条目"、"让 AI 生成"、"从模板导入"三个入口
- 各分区的说明文案：
  - 设定：世界观、力量体系、地图、规则、组织、物品
  - 大纲：卷大纲、章节规划、情节走向
  - 状态：当前写作进度、角色状态、时间线
  - 规则：AI 写作约束、文风要求、禁忌事项
  - 伏笔：已埋设线索、回收计划、过期提醒

## Phase 3：写作工具 API 修复（P1）

### 3.1 排查并修复 404 错误

**现状**: 写作工具面板中某个工具执行返回 404
**目标**: 所有写作工具 API 正常响应（即使返回空数据）

**实现方案**:
- 逐个测试 8 个写作工具 endpoint：
  - `/api/filter/scan`
  - `/api/books/:bookId/hooks/generate`
  - `/api/books/:bookId/chapters/:ch/rhythm`
  - `/api/books/:bookId/chapters/:ch/dialogue`
  - `/api/books/:bookId/health`
  - `/api/books/:bookId/conflicts/map`
  - `/api/books/:bookId/arcs`
  - `/api/books/:bookId/chapters/:ch/tone-check`
- 找到 404 的接口，检查路由注册和参数匹配
- 修复后确保所有接口至少返回空数据（不报错）

### 3.2 验证写作工具真实性

**现状**: 不确定哪些工具是真实计算 vs 返回假数据
**目标**: 确认每个工具的后端实现是否真实工作

**验证清单**:
- [ ] AI 味检测：12 规则本地扫描是否真实执行
- [ ] 章末钩子：是否调用 LLM 生成
- [ ] 段落节奏：是否真实分析句长分布
- [ ] 对话比例：是否真实计算对话占比
- [ ] 全书健康：是否真实聚合各项指标
- [ ] 矛盾地图：是否从经纬中提取矛盾
- [ ] 角色弧线：是否从章节中提取 arc beats
- [ ] 文风检测：是否对比目标文风
- [ ] 发布检查：敏感词库是否存在
- [ ] 导出：TXT/Word/ePub 是否真实生成文件

## Phase 4：模板市场补全（P2）

### 4.1 验证模板市场现状

**现状**: TemplateMarketPanel 存在，但不确定有多少可用模板
**目标**: 确认当前可用模板数量和质量

### 4.2 补全流派模板

**现状**: spec 要求 26 个流派全覆盖，当前只有 6 个
**目标**: 至少覆盖 Top 10 流派（玄幻/仙侠/都市/科幻/末日/穿越/重生/系统流/无限流/悬疑）

---

## Phase 5：画布渲染优化（P2）

### 5.1 经纬文件卡片化渲染

**现状**: 经纬文件在画布中以原始 Markdown 渲染，标题层级混乱
**目标**: 每个 section 渲染为独立卡片，未填充的显示占位 + "让 AI 生成"按钮

### 5.2 文件路径隐藏

**现状**: 画布顶部显示 `story/story_bible.md` 文件路径
**目标**: 对用户隐藏文件路径，只显示条目标题

---

## 实施顺序

```
Phase 1 — 建书流程接通（P0，首次体验核心）
  1.1 向导完成后调用 architect 生成经纬
  1.2 向导答案传递给 architect

Phase 2 — 经纬条目可编辑（P0，基本操作）
  2.1 资源树右键菜单
  2.2 新建经纬条目入口
  2.3 空态教学

Phase 3 — 写作工具 API 修复（P1）
  3.1 排查并修复 404
  3.2 验证写作工具真实性

Phase 4 — 模板市场补全（P2）
  4.1 验证现状
  4.2 补全流派模板

Phase 5 — 画布渲染优化（P2）
  5.1 经纬卡片化渲染
  5.2 文件路径隐藏
```

---

## 验证标准

- Phase 1：新建书籍后经纬文件有真实内容（不是占位符），内容与向导答案一致
- Phase 2：用户能右键新建/删除经纬条目，空分区有引导
- Phase 3：所有 8 个写作工具 API 返回 200，数据格式正确
- Phase 4：模板市场至少有 10 个可用流派模板
- Phase 5：经纬文件以卡片形式渲染，无文件路径暴露
