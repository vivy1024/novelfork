# 需求：统一资源版本系统 + 经纬上下文分层

## 背景

当前写作资源（章节/候选稿/草稿）分散在三套文件结构中（`chapters/index.json`、`generated-candidates/index.json`、`drafts/index.json`），状态管理不一致，无事务保护。经纬上下文注入策略粗暴（全量 global），导致每次 Agent 调用消耗过多 token。

## 目标

1. 将章节、候选稿、草稿统一为 `writing_resource` SQLite 表，支持完整状态机和版本链。
2. 实现候选稿/草稿操作闭环：采纳、拒绝、归档、转草稿、生成变体、版本历史。
3. 经纬上下文引入 core/relevant/reference 三层策略，按需注入，控制 token 消耗。

## 用户故事

### US-1：候选稿操作闭环
- 作为作者，我希望对 AI 生成的候选稿执行：采纳为正式章节（替换/合并/新章节）、退回草稿、拒绝、归档、删除。
- 作为作者，我希望从正式章节生成变体（新候选稿），以便尝试不同写法。
- 作为作者，我希望查看某章节的版本历史（谁派生了谁）。

### US-2：草稿操作闭环
- 作为作者，我希望编辑草稿并实时保存。
- 作为作者，我希望将草稿提交为候选稿进入审阅流程。
- 作为作者，我希望将草稿直接采纳为正式章节（跳过审阅）。

### US-3：经纬上下文分层
- 作为 Agent，我希望默认只获取 core + relevant 层经纬（~12000 tokens），不被全量数据淹没。
- 作为 Agent，我希望在需要完整世界观时主动拉取 full 模式。
- 作为作者，我希望标记某些经纬条目为"核心"，确保它们始终被 Agent 看到。

### US-4：数据迁移
- 作为现有用户，我希望升级后现有章节/候选稿/草稿自动迁移到新系统，不丢失数据。

## 非功能需求

- NFR-1：SQLite 单表查询 < 10ms（100 章 + 500 候选稿规模）。
- NFR-2：迁移过程幂等，重复执行不产生重复数据。
- NFR-3：经纬 auto 模式返回 token 总量 ≤ 15000。
- NFR-4：所有状态转换有事务保护，不会出现半完成状态。

## 验收标准

- AC-1：创建候选稿 → 采纳为章节 → 章节列表更新 → 资源树刷新。
- AC-2：从章节生成变体 → 新候选稿 → 拒绝 → 状态正确。
- AC-3：草稿编辑 → 提交候选稿 → 采纳 → 全链路通。
- AC-4：`jingwei.read_context` auto 模式返回 core+relevant，token ≤ 15000。
- AC-5：现有文件数据自动迁移，迁移后 API 返回与迁移前一致。
- AC-6：版本历史 API 返回正确的 parent 链。

## 不做

- 实时协同编辑
- Diff 可视化
- 版本树图形化
- AI 自动续写/润色按钮（只保留入口）
