# 02_灵材命名体系与批量转译规则

> **一句话版**：命名系统已经从“讨论原则”推进到“数据库实际落地”。现在每条矿物都可同时拥有 `现实学名 / 工程标准名 / 修仙俗名 / 古籍异名` 四层名字，其中只有少数世界观锚点保留固定俗名，其余绝大多数由规则引擎自动生成。
> **关联文档**：[[02_物理公式与参数手册]] · [[01_现实矿物灵材数据库首轮筛查]] · [[03_灵材样本层与工艺认证]]

---

## 1. 命名为什么必须规则化

你现在这条世界观路线有一个非常硬的前提：

> **现实世界不断会有新矿物、新材料、新工艺进来，所以灵材命名不能依赖手工硬编码。**

如果每个材料都人工起名，很快就会出现四个问题：

- 新材料一进库，命名风格立刻漂移
- 同一矿物在不同用途下会出现多套互相打架的名字
- 工程系统、黑市俗称、古籍名称无法互通
- 一旦接互联网数据，维护成本会直接爆炸

所以命名必须服从这条总原则：

```text
规则/翻译函数 > 手工命名表
```

## 2. 四层命名结构已经落地

### 2.1 第一层：现实学名

这是底层锚点，永远不丢。

例：

- `Quartz`
- `Sphalerite`
- `Hydroxylapatite`
- `Barioperovskite`

这一层负责：

- 与现实数据库对接
- 允许用户自己查资料、查论文、查工业原型
- 保证后续新材料接入时不需要重新定义世界观

### 2.2 第二层：工程标准名

当前实际落地格式：

```text
[主属性]-[功能角色]-[监管标签]-[现实学名]
```

其中三段含义分别是：

- `主属性`：当前规则引擎判出来的五行主耦合方向
- `功能角色`：`导体 / 容器 / 反应核`
- `监管标签`：`民用 / 特许 / 通用 / 实验 / 危险 / 观察`

例：

- `金-容器-民用-Quartz`
- `金-导体-民用-Sphalerite`
- `水-容器-民用-Hydroxylapatite`
- `火-导体-实验-Galena`
- `金-容器-特许-Barioperovskite`

这一层是工业、监管、实验室、材料所真正会用的名字。

### 2.3 第三层：修仙俗名

当前实际落地方式：

```text
[前缀] + [机制词根] + [后缀]
```

来源不是纯随机，而是由这些字段共同决定：

- `primary_affinity`
- `role_label`
- `danger_flag_json`
- `piezo_class / polar_class`
- `magnetic_score / water_score / wood_score`
- `category_json`

生成逻辑的直觉是：

- `金` 更容易落到 `金 / 鸣 / 霆 / 锋 / 震`
- `火` 更容易落到 `火 / 赤 / 炎 / 焰 / 炁`
- `水` 更容易落到 `水 / 涵 / 润 / 泉 / 津`
- `危险` 会改走 `幽 / 冥 / 阴 / 蚀`
- `压电/共振` 倾向 `鸣 / 雷 / 震 / 音`
- `极化/爆发` 倾向 `焰 / 炽 / 脉 / 烬`
- `容器` 常落 `石 / 璧 / 髓`
- `导体` 常落 `晶 / 片 / 锋`
- `反应核` 常落 `晶 / 核 / 髓`

### 2.4 第四层：古籍异名

古名层现在也已经落地，但它不再承担数据库主键功能。

它主要负责：

- 古代线、宗门线、秘本线的语感
- 同一材料在古今两个时代的文化差异
- 让“工程名”和“俗名”之外，还有第三种叙事入口

例如：

- `Quartz -> 太清石`
- `Sphalerite -> 鸣霄砂`
- `Hydroxylapatite -> 涵玉珠`
- `Galena -> 伏冥石`
- `Barioperovskite -> 赤髓晶`

## 3. 名称来源优先级

当前实际优先级如下：

```text
固定锚点名 > 固定公式名 > 规则生成名
```

也就是说：

1. **世界观锚点原型**保留固定俗名
2. **极少数公式锚点**允许固定映射
3. **其余 99% 材料**都用规则生成

这样做的好处是：

- `Quartz` 可以永远是 `灵石`
- `Sphalerite` 可以永远是 `雷灵砂`
- `Hydroxylapatite` 可以永远是 `水灵珠`
- 但新发现的现实材料不需要先等我们手工起名，直接就能入库

## 4. 已固定的世界观锚点名称

| 现实矿物 | 工程名 | 俗名 | 古名 | 来源 |
|----------|--------|------|------|------|
| `Quartz` | `金-容器-民用-Quartz` | `灵石` | `太清石` | `fixed` |
| `Sphalerite` | `金-导体-民用-Sphalerite` | `雷灵砂` | `鸣霄砂` | `fixed` |
| `Hydroxylapatite` | `水-容器-民用-Hydroxylapatite` | `水灵珠` | `涵玉珠` | `fixed` |
| `Galena` | `火-导体-实验-Galena` | `幽冥石` | `伏冥石` | `fixed` |
| `Barioperovskite` | `金-容器-特许-Barioperovskite` | `火灵晶` | `赤髓晶` | `fixed` |

这里最重要的不是表本身，而是它体现出来的一条设计规则：

> **工程名服从当前数值系统，俗名和古名可以保留世界观历史记忆。**

所以 `Barioperovskite` 现在会同时呈现：

- 工程上：`金-容器-特许-Barioperovskite`
- 民间/宗门里：`火灵晶`
- 古籍里：`赤髓晶`

这不是冲突，而是多语境共存。

## 6. 第二层样本命名也已经接上了

样本层名字不是重新造一套语言，而是在矿物俗名上继续加“工艺前缀”和“样本短标记”。

### 6.1 质量前缀

| 认证层级 | 前缀 |
|----------|------|
| `certified` | `真` |
| `precert` | `上品` |
| `qualified` | `精制` |
| `restricted` | `禁制` |
| `raw` | `原矿` |

### 6.2 样本短标记

| 样本形态 | 短标记 |
|----------|--------|
| `单晶样片` | `单晶` |
| `定向多晶` | `定向体` |
| `精炼粉体` | `精粉` |
| `定向晶胚` | `晶胚` |
| `生体陶瓷` | `生陶` |
| `精制珠胚` | `珠胚` |
| `定向块体` | `块体` |
| `致密烧结体` | `烧结体` |

### 6.3 实际样本名示例

| 基础矿物 | 样本工程名 | 样本俗名 | 认证 |
|----------|------------|----------|------|
| `Quartz` | `金-容器-民用-Quartz-单晶` | `真灵石·单晶` | `certified` |
| `Quartz` | `金-容器-民用-Quartz-定向体` | `上品灵石·定向体` | `precert` |
| `Sphalerite` | `金-导体-民用-Sphalerite-单晶` | `上品雷灵砂·单晶` | `precert` |
| `Hydroxylapatite` | `水-容器-民用-Hydroxylapatite-晶胚` | `上品水灵珠·晶胚` | `precert` |
| `Tenorite` | `火-导体-特许-Tenorite-单晶` | `真焰雷晶·单晶` | `certified` |
| `Majindeite` | `金-容器-特许-Majindeite-单晶` | `真鸣震髓·单晶` | `certified` |

这样一来，玩家看到名字本身就能读出四件事：

- 这是什么矿
- 当前更偏哪一行
- 它是导体、容器还是反应核
- 这是原矿、工业品、还是顶级单晶

## 7. 命名系统现在的边界和优势

### 7.1 什么地方仍然允许人工干预

当前仍然建议保留人工干预的只有三类：

- 世界观锚点原型
- 高频民用品
- 历史上有明确传说定位的危险材料

### 7.2 什么地方不该再硬编码

不该继续写死的，是下面这些东西：

- 某个现实材料永远只能叫一个唯一玄幻名
- 某个矿物一旦被叫成“火灵晶”，工程上就也必须是火属性
- 新材料必须先手工命名才能入库

换句话说：

> **固定名字只保留在文化层，扩展能力交给翻译函数。**

### 7.3 这套体系为什么适合长期扩容

因为以后无论你从互联网找到的是：

- 新矿物
- 新陶瓷
- 新型压电材料
- 新型生物矿化材料
- 甚至并非天然矿物、而是实验室合成样品

只要它能提供现实属性字段，就能被转成：

```text
现实学名 -> 工程名 -> 俗名 -> 古名 -> 样本名
```

这比手工硬起名字强太多了。

---

## 8. 一句话结论

> **命名系统已经从“讨论原则”落成到了“数据库可执行规则”：少数锚点保留固定传奇名，绝大多数材料自动生成工程名、俗名和古名，再在第二层继续扩展为样本名。这样后续不管接进来多少现实科学发现，都不需要重写世界观。**
