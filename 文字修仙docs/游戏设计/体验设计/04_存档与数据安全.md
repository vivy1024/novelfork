# 04_存档与数据安全

> **一句话版**：单机游戏不需要军工级加密，但需要让修改存档变得「不方便」——HMAC-SHA256 签名 + 自动存档 + 槽位管理。
> **关联文档**：[[01_设计理念与核心循环]] · [[13_死亡机制]]
> **代码对照**：`core/save_integrity.py` · `core/state_store.py` · `api/routes/session.py`

---

## §1 存档结构

SQLite 数据库，20+ 张表。每个存档一个 `.db` 文件。

### 核心表

| 表 | 内容 |
|-----|------|
| `save_meta` | 版本/tick/签名/角色名 |
| `player_core` | 境界/疲劳/受伤/熟练度/死亡状态 |
| `player_cultivation` | L_fit/Risk/污染/心性/丹田 |
| `player_resources` | 灵石/灵储/信用 |
| `player_talent` | 天赋列表 |
| `inventory_item` | 背包物品 |
| `npc_state` | 所有 NPC 状态 |
| `faction_state` | 势力好感度 |
| `world_state` | 世界参数 |
| `event_log` | 事件日志 |
| `experiment_log` | 实验记录 |
| `knowledge_entry` | 知识解锁 |
| `achievement` | 成就 |

## §2 签名机制（HMAC-SHA256）

```
签名范围 = player_core + player_cultivation + player_resources + save_meta.tick
密钥 = 硬编码（单机游戏，防君子不防小人）
签名存储位置 = save_meta.integrity_hash
```

### 写入时机
- 就寝后自动存档
- 突破后自动存档
- 手动存档

### 校验时机
- 读取存档时自动校验
- 校验失败 → 弹窗警告「存档数据已被修改」，但仍允许加载（不强制阻止）

## §3 槽位管理

- 最多 20 个存档槽位
- 支持删除/重命名
- Schema 版本化迁移（当前 v3），启动时自动执行

## §4 自动存档

| 触发时机 | 说明 |
|---------|------|
| 就寝 | 每天结束时自动保存 |
| 突破成功 | 重要节点保存 |
| 突破失败 | 走火后保存（防止回档刷突破） |

## §5 设计原则

- **不加密内容**：玩家可以用 SQLite 工具查看自己的存档数据（这和「公式开放」一致）
- **签名防篡改**：修改数据后签名不匹配 → 警告但不阻止
- **不做 DRM**：单机游戏没有反盗版需求
