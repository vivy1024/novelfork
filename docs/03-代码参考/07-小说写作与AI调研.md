## 一、产品最终定位

### 1.1 一句话

**NovelFork = 通用网文 AI 写作软件，核心是 4 表 Bible + 动态追踪 + 强制 AI 味过滤 + 按需 coding agent。不预制流派资产，提供模板市场与 agent 生成能力。**

### 1.2 必装核心（不可砍）

1. **4 个一等公民表**：角色 / 事件 / 设定 / 章节摘要
2. **三种可见性**（借 NovelCrafter Codex 范式）：tracked / global / nested
3. **时间线纪律**：`visible_after_chapter` 时间轴控制
4. **AI 味过滤**：默认开启、不可关闭、内置朱雀级检测 + 12 特征
5. **双写作哲学支持**：静态模式（Sudowrite-like）/ 动态追踪模式（NovelCrafter-like）

### 1.3 可选（模板市场）

1. **流派模板**（26 选）：每个包 = 默认设定条目 + 提示词预设 + 推荐节拍
2. **节拍模板**：英雄之旅 / 救猫咪 15 节拍 / 三幕 / 自定义
3. **文风 tone 预设**：悲苦 / 冷峻 / 古典 / 黑色幽默 / 沙雕
4. **消 AI 味 7 招预设**

### 1.4 按需 Coding Agent（NovelFork 独家基因）

- **默认关闭**，作者模式 UI 零暴露
- **工作台模式开关**
- **典型场景**：资产生成 / 数据抓取 / 朱雀检测 / 扫榜 / 一致性审计 / 提示词 AB

### 1.5 明确不做

- ❌ 强推流派锚定流程（改为可选引导）
- ❌ 预装大型数据库（药材 / 神话 / 黑话 ...）—— 改为 agent 按需生成
- ❌ 10 维度 Bible / 两级 RAG / 伏笔四态 这种游戏项目级复杂度
- ❌ 科学锚点 / Nature 引用 这种特定作者偏好（但支持**作者自定义字段**）
- ❌ 百年推演表这种游戏向复杂度（但支持**自定义专题文档**）

### 1.6 差异化

| 对标 | NovelFork 的差异化 |
|---|---|
| Notion / Obsidian | 有 AI 写作 + 动态 Bible + AI 味过滤 |
| Sudowrite / NovelCrafter | **中文优先** + **强制 AI 味过滤** + **coding agent 后台** + 中文网文流派模板 |
| 橙瓜 / 小黑屋 / 壹写作 | AI 原生、Bible 动态追踪、反 AI 监管合规 |
| 阅文作家助手 / 番茄 AI 工具 | **非平台绑定**、本地优先、作者拥有数据 |
| NarraFork（coder 向） | **作者友好 UI 优先**，coding agent 作为后台武器 |

## 二、平台反 AI 监管（强制核心依据）

### 2.1 三大平台红线（2025 最新）

| 平台 | 政策 | 生效 |
|---|---|---|
| **起点** | **只接受 100% 人工创作** | 2025 年 3 月起 |
| **晋江** | **仅允许 3 种 AI 辅助**：校对 / 角色名文章名生成 / 不涉及记叙文七要素的粗纲 | 2025 年 2 月《AI 辅助写作使用、判定试运行公告》 |
| **番茄** | 打击 AI 文 + 全勤奖改革 + 多模态联合检测 | 2024 末 → 2025 Q2 |
| **七猫 / 纵横** | **强制标注** AI 使用情况 | 早于番茄 |

### 2.2 国家级监管

- **《人工智能生成合成内容标识办法》** 2025-09-01 正式施行：平台必须提供标识功能、提醒用户主动声明
- **AIGC 管理办法草案**：预计将 AI 生成内容比例**统一上限设为 20%**
- **监管趋势**：平台协议 → 内容比例公示 → 检测阈值 → 处罚机制

### 2.3 主流检测工具

| 工具 | 研发方 | 特点 |
|---|---|---|
| **朱雀** | 腾讯混元朱雀实验室 | **中文主流**，图片+文本+视频全覆盖，阅文番茄均在用 |
| **谛听** | 复旦 | 学术级，识别 ChatGPT 成功率高 |
| **GPTZero / Copyleaks / Originality.AI / Pangram / Winston** | 海外 | 多语言含中文 |
| **技术原理** | — | RoBERTa 语义 + 手工统计特征双流融合 + 水印联合检测（2025 Q2 起图文联合） |

### 2.4 AI 味 12 特征（NovelFork 过滤器的核心清单）

1. 过度正式 / 官腔感
2. 固定句式模板（"首先/其次/最后"/"综上所述"）
3. 典型 AI 词汇（"值得注意的是"/"以下是为您"/"让我们一起来"）
4. 缺乏情感力量，只有客观陈述
5. 缺乏个人经验 / 口头禅 / 癖好
6. 句长方差小（人类写作句长有大幅波动）
7. 段落结构工整得像 PPT
8. 行话 / 技术术语密度高于实际需要
9. 形容词堆叠（"璀璨夺目、绚丽多彩"两连发）
10. 缺少具体细节、空话多（"他感到很开心"而不写具体动作）
11. 对话太书面语、缺口语化
12. **"伪人感"**（编辑圈术语）—— 总体像但细节不对

### 2.5 消 AI 味 7 招（写作预设）

1. **喂人类范文** → AI 学风格
2. **明确人格提示词**（"你是网文作者，轻松诙谐，脑洞清奇，台词幽默"）
3. **屏蔽鼓励人格**：对 DeepSeek 明确说"不需要鼓励，只需指出问题和修改方向"
4. **生成后人工改写 + 润色**
5. **朱雀检测 → 零感 AI 降重 → 二次朱雀** 循环
6. **每章结尾留钩子**（提示词显式要求）
7. **保存原稿 + 做好 AI 使用标注**（合规刚需）

## 三、业界 Bible 标杆：NovelCrafter Codex

NovelCrafter 的 **Codex** 是目前业界动态 Bible 的最佳实践，直接影响 NovelFork 的 Bible schema。

### 3.1 三种条目可见性

| 类型 | 规则 | 典型内容 |
|---|---|---|
| **tracked** | 场景文本提到条目名时才注入给 AI | 角色、地点、物品（绝大多数） |
| **global** | 始终对 AI 可见 | 文风、流派、全书主题、核心规则 |
| **nested** | 条目 A 引用条目 B 时自动带入 B | 组织层级、家族成员、阵营关系 |

这三档可见性解决了"条目太多 → 塞满上下文 → token 爆炸 + 注意力稀释"的经典难题。

### 3.2 时间维度 × 空间维度正交

- **时间**：`visible_after_chapter`（章节 N 之后才能注入，防剧透）
- **空间**：场景文本提到才注入（`tracked`）
- 二者正交，共同约束 AI 上下文

### 3.3 Progressions 与自动链接

- 写 `Sarah entered the café` → 自动链接 Sarah / café 条目
- 角色弧线按章节推进（progressions），追加不覆盖
- AI 上下文场景级控制（per-scene context）

### 3.4 Sudowrite vs NovelCrafter 哲学

| 工具 | 哲学 | 适合 |
|---|---|---|
| **Sudowrite** | brain dump + AI 自动补，Story Bible 静态 | 脑洞型 / 花园派作者 |
| **NovelCrafter** | Codex 严格管理 + 自动链接 + 大纲驱动 | 情节型 / 建筑师派作者 |

NovelFork 策略：**同时支持两种哲学**，作者可切换。默认静态，手动开启追踪。

## 五、真实作者痛点

### 5.1 日更压力

- **6000 字 = 及格线**，**日万 = 天花板**
- 超过 6000 字难度指数上升
- "大纲消耗快，每天欠读者 4000 字"
- 番茄首秀：**8 万字验证 / 10 万字首秀 / 3 周首秀期 / 每日 6000**

### 5.2 崩盘痛点

- 200 万字以上长篇**追订崩盘**
- 开头精彩、后续乏力 = 没搭框架或没按大纲写
- **"烂尾是能力问题，太监是态度问题"**（塔读圈名言）
- 99% 新人止步于**负反馈多 + 屡投不过 + 未来不确定**

### 5.3 作者实际要的 Bible 字段（社区自画像）

```
世界观 / 力量体系 / 地图 / 人物卡 / 金手指
/ 人物关系图谱 / 势力分布 / 故事背景
```

**8 个字段**。NovelFork 的 4 表核心已覆盖其中的大部分（角色=人物卡+金手指+关系图谱；设定=世界观+力量体系+地图+势力+背景；事件 / 章节摘要补充）。

### 5.4 扫榜 / 套路 / 流派选择

- **"套路 = 解决'写什么'的问题"**
- 工作流：看榜 → 筛流派 → 找方向 → 开文
- 飞卢特例：题材套路严格对标，脱离榜单易死

## 六、Coding Agent 作为作者助手

### 6.1 Claude Code 的能力范式（目标标准）

- **真 Agent**：自治（规划→执行→纠错）、跨界（MCP / 浏览器 / 终端）、有记忆（跨会话上下文）
- **200K 上下文 + 真实终端权限**（Git / Shell / 文件系统）
- **ultrathink 深度思考 + `/rewind` 检查点回滚**
- **双保险**：检查点 + Git 版本控制

### 6.2 NovelFork 的 coding agent 应用场景

按"**按需启用、用完即走、避免浪费 token**"原则：

| 场景 | 触发 | 输出 |
|---|---|---|
| **批量生成世界资产** | 作者："帮我做一个 200 条药材库" | SQLite / YAML / Markdown 表 |
| **外部数据抓取** | 作者："抓 TCMID 所有活血药" | 结构化数据 + 来源溯源 |
| **朱雀检测集成** | 写完章节自动扫 | AI 味评分 + 高亮段落 |
| **扫榜分析** | 作者："看起点 30 万字以下新书榜" | 榜单 + 流派分布 + 爽点对比 |
| **长篇一致性审计** | 定期 | 人设冲突 / 时间线矛盾 / 伏笔未回收 |
| **提示词批量测试** | 作者："我这 10 个提示词哪个更好" | 对照生成 + AI 味评分 |

### 6.3 双模界面

| 模式 | UI | 默认 | 成本 |
|---|---|---|---|
| **作者模式** | 编辑器 + Bible + AI 写作 | 默认 | 普通 token |
| **工作台模式** | 暴露 terminal / 文件树 / agent 日志 | 按需开启 | 高 token（仅需要时） |

**关键原则**：**作者模式 UI 零 coding 暴露**。工作台模式是**后台武器**，不是前台功能。

### 7.2 作者级技法精华（浓缩）

以下是对几位代表作者的技法提炼，供模板市场对应流派使用。

#### 7.2.1 熊狼狗谱系（没钱修什么仙 / 食物链顶端 / 寂静杀戮 / 魔神乐园）

- **核心命题**：现实社会的极端映射（修仙 = 借贷 / 内卷 / 阶层 / 药物依赖 / 信息垄断）
- **爽点涌现机制**：不设计爽点，从世界规则自然涌现（经济压力→算账的爽 / 信息垄断→发现隐藏路径的爽）
- **唯物视角**：灵气 ≈ 辐射；命丛 = 吸收辐射的器官（《寂静杀戮》）
- **力量增长模型**：加法（凡人）/ 乘法（半神）/ 幂次（魔神）—— 不是量的差距，是数学结构差距
- **高风险修炼设计**：六欲分魔章不可逆 / 越强越难吞 / 副作用影响人格
- **最终 boss 用经济解决**：百亿补天，债务 = 盾牌

#### 7.2.2 忘语《凡人修仙传》

- **凡人流范式**：主角无金手指、步步为营、老谋深算
- **苟住发育哲学**：伪装、藏拙、不暴露智识
- **时间跨度**：几十章才练气圆满，不玩快节奏

#### 7.2.3 玄鉴仙族（家族流范本）

- **胎息六轮 → 练气 → 筑基 → 紫府 → 金丹 → 元婴/道胎**
- **功法三分类**：正法（完整）/ 古法（缺一气）/ 秘法（改编古法，有缺陷）
- **天地灵气体系**：功法决定需要哪种灵气，形成"功法选择 → 灵气需求 → 资源获取"决策链
- **道基命名系统**：功法 + 灵气组合 → 独特道基名（湖月秋 / 浩瀚海 / 溪上翁）
- **家族经济**：灵田种植 / 灵石通货 / 功法分级 / 供奉体系

#### 7.2.4 远瞳《黎明之剑》（工业流 / 超凡公共化范本）

- **核心命题**：超凡力量能否工业化、公共化、教育化
- **魔网 = 电网**：集中发电、分布式供能 → 基础设施化
- **符文逻辑学 = 数学化魔法**：凡人参与超凡的杠杆
- **农奴解放 + 义务教育 + 魔导技术研究所**：制度三件套
- **"技术在普及中发展"** vs "法师塔垄断"

#### 7.2.5 猫腻（将夜 / 庆余年 / 朝闻道）

- **一句话哲学命题贯穿全书**："不服天命"
- **历史化 / 人文化 / 庙堂味**：文笔最"文学化"的网文作者
- **反爽文爽感**：慢热、主角情商高、少打脸

#### 7.2.6 爱潜水的乌贼《诡秘之主》

- **蒸汽朋克 + 克苏鲁 + 序列体系**
- **序列 22-0 + 22 途径**：极度结构化、适合做游戏化 RAG
- **仪式禁忌写法**："第一条第二条第三条"格式

#### 7.2.7 辰东（悲苦玄幻范本）

- **"悠悠苍天，何薄于我"** 式命运叙事
- **开局上帝视角大场面** + 主角慢热成长
- **悲剧感 + 热血感并存**

#### 7.2.8 蝴蝶蓝《全职高手》（群像教科书）

- **15 战队 + 300+ 角色**，每个配角至少一个辨识度标签
- **主角多身份**：联赛大神 / 网管 / 战术大师 / 队长
- **职业-现实交叉细节**：真实职业联赛骨架

### 7.3 古典叙事学根基

- **英雄之旅（17 阶段，约瑟夫·坎贝尔）**
- **救猫咪节拍表（15 节拍，Blake Snyder）**
- **三幕结构**
- **控制观念**（Robert McKee）：一句话哲学命题
- **扁平 vs 圆形人物**（E.M. Forster）
- **花园派 vs 建筑师派**（Stephen King 对 George R.R. Martin）

## 附录 A：调研方法论与边界

### A.1 四个边界（避免误导后续设计）

1. **读者逆向分析** ≠ 作者正向工作流
2. **游戏世界观文档** ≠ 小说 Bible（需求密度不同）
3. **特定作者偏好** ≠ 全体作者共性
4. **理论家框架** ≠ 实际写作者脑回路

### A.2 "共性 vs 个性"判据

| 共性（必装核心） | 个性（可选扩展） |
|---|---|
| 角色档案（防人设崩） | 角色金融资产 / 法骸配置 / 命丛 / 灵根...（特定流派） |
| 事件时间线（防逻辑崩） | 百年推演 / 多时间线战争（特定作品） |
| 设定条目（防设定违背） | 1018 条药材 / 200 条神话（项目级） |
| 章节摘要（AI 不忘前文） | 章节级物理公式 / 科学锚点（硬核作者） |
