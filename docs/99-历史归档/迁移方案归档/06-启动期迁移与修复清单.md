## 这份清单解决什么问题

Phase H 需要把“启动期迁移、索引恢复、状态修复”从零散实现整理成正式口径。

本清单只描述**当前代码已经存在或明确缺失**的行为，不把目标态当成已完成事实。

---

## 一句话结论

**当前 NovelFork 已有启动期最小恢复编排，但还没有“持久化巡检 + 后台持续修复”的总控任务。**

现在真正存在的是三类机制：

1. **启动时最小初始化**：只负责项目目录与静态资源入口。
2. **启动前 best-effort 恢复**：会先对已知书籍做 runtime state 补建，并重建内存搜索索引。
3. **按需触发的惰性修复**：在写作、读取 runtime state、同步章节产物时，才对单本书做迁移或修复。

## A. 服务启动阶段：当前会自动做什么

### A1. 项目根目录初始化

入口：`packages/studio/src/api/server.ts` → `startStudioServer()`

当前自动动作：

- 若 `<root>/novelfork.json` 不存在：
  - 创建项目根目录
  - 创建 `books/`
  - 写入最小 `novelfork.json`
- 这一步**不**会扫描现有书籍，也**不**会重建章节索引、搜索索引或运行时状态

### A2. Studio 静态资源选择

入口：`main.ts`

当前选择顺序：

1. 优先加载 `packages/studio/src/api/embedded-assets.generated.ts`
2. 若 embedded 模块不存在，则回退到 `packages/studio/dist`
3. 若两者都没有，仅启动 API，不承诺前端可访问

这属于**启动期资源装配**，不是数据迁移任务。

## B. 惰性迁移 / 修复：当前已经存在的正式任务

## B1. 结构化 runtime state 补建

核心实现：

- `packages/core/src/state/state-bootstrap.ts`
- `packages/core/src/state/manager.ts`
- `packages/core/src/state/runtime-state-store.ts`

触发入口：

- `StateManager.ensureRuntimeState()`
- `StateManager.getNextChapterNumber()`
- `loadRuntimeStateSnapshot()`
- 记忆检索 / 写作管线内部需要 runtime state 的路径

当前会做的事：

- 从 legacy markdown 真相文件补建：
  - `story/state/manifest.json`
  - `story/state/current_state.json`
  - `story/state/hooks.json`
  - `story/state/chapter_summaries.json`
- 以 `chapters/index.json` + 章节文件连续编号推导 `lastAppliedChapter`
- 自动去重 `chapter_summaries.json`
- 发现旧文件非法时，记录 `migrationWarnings` 并按 markdown 重建

当前边界：

- 这是**单书按需修复**，不是启动时全量巡检
- 章节进度仍以 durable chapter artifacts 为准，不信任 markdown 内可能幻觉出的章节号

## B2. legacy 书籍迁移提示

相关位置：

- `packages/cli/src/utils.ts` → `getLegacyMigrationHint()`
- `packages/cli/src/commands/status.ts`
- `packages/cli/src/commands/write.ts`
- `packages/cli/src/commands/doctor.ts`

当前行为：

- 若书籍缺少 `story/state/`，CLI 会提示这是 legacy format
- 真正迁移动作仍依赖后续写作/读取 runtime state 的惰性补建逻辑

这说明：

- **提示已正式存在**
- **批量启动期迁移编排尚不存在**

## B3. 章节重写前的索引回退与状态恢复

命令：`novelfork write rewrite [book-id] <chapter>`

实现位置：`packages/cli/src/commands/write.ts`

当前动作：

- 备份目标章及后续章节文件
- 截断 `chapters/index.json`
- 尝试从 `chapter - 1` 的 snapshot 恢复 story state
- 若后续重写失败，再把备份恢复回去

用途：

- 这是**重写路径上的状态回退任务**
- 不是服务启动时自动执行的修复

## B4. 最新正文回灌真相文件与索引

命令：`novelfork write sync [book-id] <chapter>`

实现位置：

- `packages/cli/src/commands/write.ts`
- `packages/core/src/pipeline/runner.ts` → `resyncChapterArtifacts()`

当前动作：

- 从已编辑的最新章节正文重新结算 state
- 更新 truth files
- 同步 narrative memory index
- 更新 snapshot
- 同步 current state fact history

当前边界：

- 只允许安全地处理**最新已落盘章节**
- 不是启动时自动扫全书

## B5. state-degraded 章节修复

命令：`novelfork write repair-state [book-id] <chapter>`

实现位置：

- `packages/cli/src/commands/write.ts`
- `packages/core/src/pipeline/runner.ts` → `repairChapterState()`

当前动作：

- 不重写正文，只重新结算章节状态
- 回写 truth files
- 同步 narrative memory index
- 重新打 snapshot
- 清理章节索引中的 `state-degraded` 标记

当前边界：

- 只允许修复**最新**且已标记 `state-degraded` 的章节
- 这是一条显式运维命令，不是服务启动时后台自动任务

## C. “索引恢复”当前到底指什么

### 已存在

- 章节索引维护：`chapters/index.json`
- narrative memory / SQLite 相关同步：写作、sync、repair 路径里按需执行
- structured runtime state：按需补建与校验
- 启动期最小恢复编排：`startStudioServer()` 会先跑 `runStartupOrchestrator()`，对已知书籍做 runtime state 补建并重建内存搜索索引

### 尚不存在

- 持久化的搜索索引恢复链路
- 启动时后台持续巡检 / 自动排队修复
- 启动时自动遍历 books 并执行 repair/sync 的完整运维编排

注意：`packages/studio/src/api/lib/search-index.ts` 当前是**内存索引单例**，因此这里只能说“启动后重建了内存搜索索引”，不能宣称“启动即可自动恢复持久化搜索索引”。

## D. 运维执行建议（按当前代码事实）

### 场景 1：legacy 书籍刚接入

建议顺序：

1. `novelfork status`
2. 对目标书执行一次正常写作或显式读取 runtime state 的路径
3. 检查 `story/state/` 是否已生成
4. 若需要，再继续写作

### 场景 2：人工编辑了最新章节正文，需要回灌真相文件

执行：

```bash
novelfork write sync [book-id] <latest-chapter>
```

### 场景 3：最新章节被标成 `state-degraded`

执行：

```bash
novelfork write repair-state [book-id] <latest-chapter>
```

### 场景 4：要重写某章并回退后续章节

执行：

```bash
novelfork write rewrite [book-id] <chapter>
```
