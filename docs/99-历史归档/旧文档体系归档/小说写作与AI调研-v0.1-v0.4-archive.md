## 一、经典叙事学：好故事的结构性答案

### 1.1 亚里士多德《诗学》（公元前 335）

悲剧六要素按重要性排序：**情节 > 性格 > 言语 > 思想 > 戏景 > 唱段**。

核心观点：

- **情节是灵魂**："情节乃悲剧的基础，有似悲剧的灵魂"
- **结与解**：结是矛盾冲突的重重设置，解是冲突的逐层解决
- **突转与发现**：人物命运由顺而逆或由逆而顺的转折点；从"未揭示的关系 / 事件端倪"到真相揭示的跳变
- **一时错误 (hamartia)**：角色堕落应该是因为一时错误，而非必然，才能感动观众

对 AI 写作的启示：**让 agent 始终追踪"当前场景是在设结还是在解结"**，是任何爽点 / 反转的起点。

### 1.2 罗伯特·麦基《故事》（1997）

好莱坞编剧圣经，被英国卫报称为"亚里士多德后最有影响力的讲故事理论家"。

五层结构自小到大：**节拍 → 场景 → 序列 → 幕 → 故事**。

- **节拍 (beat)**：场景中最小的动作单元；通常是行为的交替
- **序列**：3～5 个场景，每个冲击力递增，直到登顶
- **控制观念 (controlling idea)**：一句话表达故事的核心价值立场

McKee 写剧本三大忌：

1. 把剧本写成小说（文字当画面 vs 抒情句）
2. 用对话交代剧情
3. 角色依赖独白

对 AI 写作启示：**"节拍→场景→序列"这套粒度，是 AI 分层生成最自然的单位**。不是"生成一章 3000 字"，而是"生成 3 个序列，每个序列 3 个场景，每个场景 5 个节拍"。

### 1.3 救猫咪 15 节拍表（Blake Snyder，2005）

最普及的三幕剧本结构模型：

| 节拍 | 页码位置 |
|---|---|
| Opening Image | 1 |
| Theme Stated | 5 |
| Set-Up | 1-10 |
| Catalyst | 12 |
| Debate | 12-25 |
| Break into Two | 25 |
| B Story | 30 |
| Fun and Games | 30-55 |
| Midpoint | 55 |
| Bad Guys Close In | 55-75 |
| All Is Lost | 75 |
| Dark Night of the Soul | 75-85 |
| Break into Three | 85 |
| Finale | 85-110 |
| Final Image | 110 |

映射到网文章节：约每 7～8 页一个节拍；对应 **2000-3000 字章节跨度**。

对 AI 写作启示：**节拍表是 AI 生成长篇的最佳骨架约束**。让 agent 在写 Midpoint 时知道"必须有镜像时刻 / 虚假胜利或失败"，比让它"自由发挥"质量高一个量级。

### 1.4 坎贝尔《千面英雄》英雄之旅（1949）

17 阶段分三幕：

- **启程**（5 阶段）：历险的召唤 / 拒绝召唤 / 超自然助力 / 过第一道门槛 / 鲸鱼之腹
- **启蒙**（6 阶段）：考验之路 / 与女神相遇 / 诱惑 / 与父和解 / 神化 / 终极恩赐
- **回归**（6 阶段）：拒绝回归 / 超自然逃亡 / 外界营救 / 过回归门槛 / 两个世界的大师 / 自由自在

对 AI 写作启示：**中国网文的爽文套路大多是英雄之旅的变体**（穿越 = 召唤 / 金手指 = 超自然助力 / 打脸 = 与父和解 / 登顶 = 终极恩赐）。把 17 阶段做成模板，能机械地给出"下一步应该发生什么"的答案。

### 1.5 中国古典评点学（金圣叹 / 脂砚斋）

金圣叹评点《水浒》、《西厢》；脂砚斋评《红楼梦》。核心贡献：

- **技法 vs 内容并重**：不是只看故事，也看"这一笔怎么写出来的"
- **夹批 / 眉批 / 总批**：多层次点评体系，粒度从字到章
- **再创作**：评点本身是对原作的补充，不是旁观
- **伏笔与照应**：脂批大量标注"前文 XXX，此处方应"的长距关联

对 AI 工具的直接启示：**"AI 评点"应该成为产品一等能力**。让 AI 扮演"脂砚斋"角色，在作者写完一章后给出夹批式反馈（这一句太像套话 / 这里伏笔没埋好 / 此处情绪密度不够），而不是只"续写"。

## 二、中文网文方法论

### 2.1 黄金三章（有争议但普遍遵守）

合格开篇 5 要素：

1. 顺畅精彩的故事（不能一上来铺设定）
2. 聚焦主角（2000 字内必须出场）
3. 清晰世界观（但不是信息倾倒）
4. 有亮点的金手指（给主角动力和不公平优势）
5. 期待感和新颖性（一个"这本书和别的不一样"的钩子）

**争议**：大神不按黄金三章写也能火（辰东 / 猫腻）；但新人**必须**按，因为没有粉丝基础容错。

实操节奏（阅文作家助手推荐）：

- **定时锁**：到一定时间必须发生特定事情。套路 + 节奏的结合体
- 举例：每 3 章一个小爽点，每 10 章一个大爽点，每 30 章一次格局升级

### 2.2 爽文模型

核心循环：**憋屈 → 反击 → 打脸 → 收益**。

套路类型枚举：

- **傻瓜获胜**：被轻视的配角对抗强敌成功
- **碰撞奇遇**：偶然撞入改变命运的事件
- **重生**：带着未来记忆重新来过
- **穿越**：带现代知识到古代 / 异世界
- **系统流**：外挂型金手指（签到 / 签约 / 任务奖励）
- **反套路**：把已知套路做反转（真假千金反杀 / 追妻火葬场进阶版 / 死人文学虐渣）

### 2.3 金手指设计原则

> "金手指不是主角养的宠物，主角也有自己的事情要去完成。"

关键约束：

1. **金手指不作为主线**：用来突转、化险为夷，不当叙事主干
2. **必须有限制**：无限金手指会反噬代入感；限制越具体代入感越强
3. **不喧宾夺主**：读者记住的是主角，不是金手指本身
4. **叙事突转模式**：金手指的每次使用都应推动情节反转

案例：横扫天涯《天道图书馆》的图书馆金手指受"需要付出代价"限制，爱潜水的乌贼《诡秘之主》的序列体系是严格等级限制。

### 2.4 代入感

> "代入感 = 读者与主角之间的距离感。"

距离大 → 隔阂 → 弃书。

拉近距离的标准手段：

- 主角心理活动要让读者能 YY
- 主角面对的困境要读者能共情（穷 / 被看不起 / 暗恋）
- 主角性格不能过于"完美"（讨厌感会破坏代入）
- **用主角的感官写世界**（让读者透过主角的眼耳鼻舌去经验）

### 2.5 大神作者画像

| 作家 | 风格特点 | 可学的点 |
|---|---|---|
| 唐家三少 | 世界观构架上等；情节安排好；爽点高潮多；文笔顶级作者里倒数 | 结构 > 文笔；日更坚持（发烧 40 度还写 6000 字） |
| 辰东 | 《神墓》《遮天》《完美世界》《圣墟》；从不死不灭到堕落 | 中期商业化导致质量下滑——警惕机械量产 |
| 爱潜水的乌贼 | 《一世之尊》《诡秘之主》；融合基金会 + 克苏鲁 | 世界观原创性 + 长线伏笔 |
| 天蚕土豆 | 从《魔兽剑圣》到商业化 | 起于兴趣（打魔兽） |

## 三、AI 味识别与消除

### 3.1 AI 味的 12 个典型特征

1. **结构化三段式**：首先 / 其次 / 此外 / 最后
2. **程式化连接词**：综上所述 / 总而言之
3. **被动语态 + 虚构逻辑链**
4. **三段式论证**（论点-论据-结论）
5. **节奏均匀**：句长一致，没有长短交错
6. **情绪词堆叠**："他感到无比愤怒和悲伤，极度痛苦"（用形容词替代画面）
7. **状语多、空洞形容词**：冗余修饰
8. **排比重复**："似的，似的，似的" / "阿光…，阿光…，阿光…"
9. **累赘内心独白凑字数**
10. **不合时宜的抒情**
11. **缺乏感官细节**（show 替代 tell）
12. **正能量结尾**（与剧情脱节的升华）

### 3.2 AI 味的长篇退化现象

关键观察（腾讯新闻实测）：

> "0k 版语言风格较为平实，从第二个片段开始，结尾出现典型的 AI 味儿段落。随着轮次变多，AI 味儿占比越来越多，包括累赘的内心独白、重复的句子/段落。"

长篇 AI 写作的典型退化：

- 越写越爱凑字数
- 越写越爱用模板句式
- 越写越脱离主线（AI 自动续写后续情节）
- 两段故事衔接上出现问题

**关键工程认知**：AI 长篇退化是累积效应，不是一次性可解决的；需要**每 N 章触发一次风格校准**。

### 3.3 消 AI 味的 9 个有效手段

1. **第一人称经验植入**：用"我 / 我们"开头建立真实感
2. **时间锚点**："那天下午三点" > "某日"
3. **感官细节 (show don't tell)**：让读者看见 / 闻到 / 听到，而不是被告知
4. **融入矛盾点和个人观察**：真实文字必带主观倾向
5. **长短句交错**：1-3 字短句 + 15+ 字长句 互错
6. **分段生成 + 随时修正**：别一次性输出 5000 字
7. **角色 / 语气先定**：系统提示里明确设定 "你是一位有 20 年写作经验的小说作者，风格细腻"
8. **避免直接形容词堆叠 → 用动作 / 细节让读者体会**
9. **2k 直接仿写 > 0k 生成**：给风格样本比给描述有效几个量级

### 3.4 平台反 AI 监管（2025-04-28）

16 家头部网文平台发布《网络文学行业反洗稿自律公约》，起因：

- 2024 年番茄签约新增"AI 训练补充协议"
- 但 AI 洗稿 / 抄袭问题激增
- 监管转向：从"允许 AI 训练" → "遏制 AI 滥用"

**关键信号**：在中国网文市场，"消 AI 味"不再是可选项，是**法务必需**。产品必须内置"AI 痕迹检测 + 降 AI 率"能力。

## 五、社交媒体与实操共识

### 5.1 日更方法论

| 日更量 | 水平 | 月入 |
|---|---|---|
| 3000 字 | 新人基本线 | 几百～千元 |
| 5000 字 | 合格 | 数千 |
| 7000 字 | 追读分水岭 | 万元级 |
| 10000 字 | 头部预备 | 数万 |
| 20000 字 | 商业化量产 | 10+ 万 |

关键数据（匿名头部作者自述）：

- 最高码字时速 ≈ 3000 字
- 最高日更 ≈ 2 万字
- 最高月更 ≈ 40 万字
- "日更 7000 和 2 万差了**无数个次元**"（因为追读机制）

### 5.2 卡文对策

- 每周准备**多个选题 / 方案**，卡这个换那个
- 不要被动等灵感
- 平时多练笔
- 预先准备**完善大纲**是极限日更的前提

### 5.3 开篇 12 种（唐歌）

- 单刀直入（好处 / 论点）
- 诗歌开头（需文学功底）
- 设悬念
- 直抛矛盾冲突
- 对话开场
- 场景描写
- 第一人称自白
- 倒叙开头（从结局切回）
- 反问 / 设问
- 名言引用
- 数据引人（"全网 80% 的人都…"）
- 人物登场特写

### 5.4 平台投稿流程

| 字数 | 动作 |
|---|---|
| 5000 字 | 给编辑看 |
| 5000-30000 字 | 过稿发书 |
| 30000 字 | 签约 |
| 30000-120000 字 | 上推荐 |
| 80000-120000 字 | 上架收费 |

### 5.5 龙的天空 / 知乎 / 小红书 / 抖音的角色

- **龙空**：网文圈最专业评论社区；原创评论版参考价值最高；质量参差
- **知乎**：短篇爽文，16-25 岁高知女性受众；反套路 + 情绪共鸣 + 价值观点
- **小红书**：情绪价值 / 实用价值 / 新知；5000 粉可变现；emoji + 口语化
- **抖音**：算法推荐 90%（而非搜索）；数据驱动思维

### 6.2 番茄流量池模型（对 AI 工具最重要）

番茄的"支出-营收模型"是**所有 AI 写作工具都必须适配的数据契约**，因为它是当前最大的新人入口：

| 阶段 | 字数 | 流量池 | 晋级门槛 |
|---|---|---|---|
| 注入 | 0-5 万字 | 新人池 | 首秀吸量率 |
| 观察 | 5-10 万字 | 千次池 | **章节平均完读率 > 35%**；书架比 < 5:1 |
| 放量 | 10-20 万字 | 万次池 | **7 日留存 > 25%**；追更率 > 30% |
| 稳定 | 20 万字+ | 稳定池 | **30 日留存 > 15%**；拉新率 > 5% |

关键数据权重（按影响力排序）：

1. **次留 / 三留 / 七留**（最重要，早期权重）
2. **章节完读率**（10 万 / 20 万 / 30 万 / 100 万字节点决定晋级）
3. **追更率**
4. **吸量率**（推荐转化）
5. 评分 / 评论 / 催更（影响小）

### 6.3 题材进化（2024-2025）

**男频**：传统"升级"结构 → "理念"结构；"不爽"成爆款特征（读者不再要无脑爽，要逻辑和价值观）

- 反套路追妻火葬场进阶版

- 反宫斗 / 反内卷

**跨品类趋势**：跨界融合（如乌贼"基金会+克苏鲁"）；标题加强噱头性；内容加强快感机制

### 6.4 IP 变现：微短剧才是真正的金矿

2024 年数据（艺恩 / 中国社科院）：

- **微短剧用户 6.62 亿**（网民 59.7%）

## 七、对 NovelFork 的决策映射

> 前六节是素材，这一节是产出。

### 7.1 产品定位重新锚定

NovelFork = **为中文网文作者打造的 AI 协同写作工作台**，有三条硬约束：

1. **作者向而非 coder 向**：工具存在是为了赋能作者，不是为了展示 agent 能力
2. **消 AI 味是一等能力**：平台反 AI 监管已是现实，不是未来
3. **数据驱动贴合番茄模型**：章节完读率 / 留存率 / 追更率要成为产品内建语言

### 7.2 核心能力地图（按优先级）

#### P0 底座（无它不能动）

| 能力 | 说明 | 依据 |
|---|---|---|
| **SQLite + drizzle 存储** | session / message / history / buffered-queue；章节正文仍文件 | 当前 JSON 已反噬（session-store 孤儿诊断）；NarraFork v0.1.17 证据 |
| **Story Bible / 世界观资料库** | 人物 / 地点 / 设定 / 大纲 / 金手指规则的集中管理 | 所有成熟竞品标配（Sudowrite / NovelCrafter / Campfire） |

#### P1 作者价值核心（上线即可卖钱）

| 能力 | 说明 | 依据 |
|---|---|---|
| **写作预设库** | 续写 / 改写 / 审校 / 扩写 / 大纲 / 人设辩论 / 开头仿写 / 金手指设定 共 8～12 个内置预设 | 竞品普遍做法；替代 NarraFork "套路页" 的作者向形态 |
| **节奏 / 节拍表支持** | 每个章节 / 序列 / 场景可标记节拍；AI 按节拍生成 | 麦基《故事》五层结构 + 救猫咪 15 节拍表 + 英雄之旅 17 阶段 |
| **消 AI 味后处理** | 生成后自动检测 AI 味特征（12 种）+ 一键降 AI | 3.1/3.2/3.4 刚需；竞品笔灵主卖点 |
| **章节叙事线（重定义）** | 每章绑定独立会话 + chapterId + 预设 + 上下文；章节正文文件级隔离即够 | 不做 worktree 权限降级；NovelFork 场景无并发编辑同段 |
| **脂砚斋式评点工具** | AI 作为评点者给夹批（这句太 AI / 这里伏笔缺 / 情绪密度不够） | 1.5 节；差异化亮点 |

#### P1.5 作者向"多开"（重解 NarraFork 的并行叙事线）

把 NarraFork 的"并行章节叙事线"按作者心智**重新映射**：

| 原 NarraFork 概念 | NovelFork 新心智 | 实际价值 |
|---|---|---|
| 章节叙事线 × 多 worktree 并行 | **同章节多变体生成**：同一章在"热血 / 冷酷 / 搞笑 / 悲情"多风格下各生成一版，作者择优 | 作者真需要 |
| 合并冲突检查 | **变体对比视图**：并排展示不同变体，高亮差异 | 非常实用 |
| 不同条件 | **多视角 / 多情节走向 / 多结局** 快速探索 | 刚需功能 |

已有的 `WorktreeManager` 不删，但**重新定位**为"实验分支"——作者想试一条激进路线（比如大改中段）时用，不是常规流程。

#### P2 透明化（让作者信任 AI）

| 能力 | 说明 | 依据 |
|---|---|---|
| **上下文 % 指示器** | ChatWindow 顶部显示"本次对话能容 N 千字" | NarraFork 已做；第五节共识 |
| **工具调用折叠块** | 默认折叠，点击展开。作者按需查看 AI 调了什么工具 | 区分 coder / writer 心智 |
| **模型 / 推理强度即时调** | 头部三挡（快 / 均衡 / 深思） | 作者易理解 |
| **成本透明** | 每次请求显示 tokens + 预估费用 | 请求历史页聚合 |

#### P2 数据驱动（让作者看见市场）

| 能力 | 说明 | 依据 |
|---|---|---|
| **章节健康度卡片** | 每章显示：字数 / 估算完读率 / 节奏评分 / AI 味分数 / 情绪曲线 | 番茄流量池模型 6.2 |
| **阅读节奏可视化** | 按章画出情绪曲线 / 爽点分布 / 卡点位置 | McKee 节拍理论的可视化 |
| **开篇特检** | 针对黄金三章做 5 维自检（主角 / 金手指 / 钩子 / 期待感 / 世界观） | 2.1 节 |

#### P3 平台化（让作者长留）

| 能力 | 说明 | 依据 |
|---|---|---|
| **请求历史页** | TTFT / tokens / 成本 / 凭证 筛选；走 SQLite | NarraFork v0.1.14；第三节 |
| **供应商管理强化** | 推荐（DeepSeek v4 / Gemini 3.1 Pro）置顶；其他折叠 | 2026 年现实 |
| **自动更新** | Tauri plugin-updater + stable/beta 双通道 + changelog modal | NarraFork 标配 |

#### P4 延展（商业化扩展）

| 能力 | 说明 | 依据 |
|---|---|---|
| **短剧分镜生成** | 小说 → 分镜剧本（针对微短剧市场） | 6.4 节 504 亿市场 |
| **Scrivener / Word 导出** | 排版交接 | 竞品标配 |
| **人物关系图 / 时间线** | 可视化 Story Bible | 蛋黄对标 |

#### 不做 / 延后（明确排除）

- ❌ 章节节点图编辑器（线性阅读心智）
- ❌ 全开工具 + 最少内置提示词（作者向反设计）
- ❌ Admin 面板 10 Tab 套路页（作者劝退）
- ❌ 终端 / Browser / NarraForkAdmin 作一等 UI 入口（agent 内部能力即可）
- ❌ NAP / Kiro 反代（风控成本 > 收益；用 DeepSeek v4 / Gemini 3.1 Pro 直连）
- ❌ 容器 / Docker Compose（单机写作工具不需要）
- ❌ git worktree 权限降级（章节正文已文件级隔离）

# 第二轮加深：起点高质量作品作者级解剖

> v0.2 增补，2026-04-25。
> 起因：v0.1 的"中文网文方法论"停留在"黄金三章 / 套路枚举"表层，缺少作品和作者级的技法剖析。本节按用户指引，深入解剖近十几年起点真正公认的高质量作品，从"为什么它们好"反推工具应该怎么做。

## 八、起点高质量作品矩阵（按流派与时代）

### 8.1 作品分层法

不是按字数或人气排，按**对网文生态的贡献类型**分：

| 层级 | 定义 | 代表作 |
|---|---|---|
| **开山鼻祖** | 开创一整条流派 | 《飘邈之旅》（修真流）/ 《凡人修仙传》（凡人流）/ 《斗破苍穹》（废柴流+套路化）/ 《无限恐怖》（无限流） |
| **集大成者** | 在既有流派里做到天花板 | 《仙逆》《求魔》/ 《神墓》《遮天》/ 《将夜》《庆余年》/ 《诡秘之主》 |
| **反套路创新** | 有意识解构主流 | 《道诡异仙》（反爽虐主）/ 《赘婿》（慢工雕琢）/ 《我本无意成仙》（古典仙气） |
| **现役新势力** | 近 3 年 00 后作者的新标杆 | 《玄鉴仙族》/ 《没钱修什么仙》/ 《我的模拟长生路》/ 《大明王朝1627》/ 《苟在初圣魔门当人材》/ 《我本无意成仙》 |
| **工业党 / 异世科技** | 西幻 + 理工 | 《黎明之剑》/ 《放开那个女巫》 |

### 8.2 开山鼻祖层的技法贡献

#### 《飘邈之旅》（萧潜，2000s 早期）

- **首创修真体系**：把道家修行名词 + 境界划分 + 凡人成仙路径打包成一套可重复使用的世界观模板
- **补位价值**：武侠 vs 西幻之间的空白地带，专属中式凡人升仙路
- **对后续作品的基础贡献**：炼气→筑基→金丹→元婴→化神→炼虚→渡劫→仙道 这套境界表，**到今天所有仙侠文仍在用**

**对 AI 工具的启示**：**境界表应作为预制模板内置**，作者选一本仙侠就自动加载；改一条境界名，其他 AI 生成的文字不会出现"设定漂移"。

#### 《凡人修仙传》（忘语，2008）

- **凡人流开山**：主角资质平庸，无大金手指（仅绿瓶）；描写"**人与人、仙与仙之间的利害冲突**"；"**某种内卷式职场现**"
- **节奏**：慢热、扎实，非爽文
- **人物塑造**：韩立"意志坚定、行事缜密、不好面子、不逞英雄、见机行事、进退有据"—— 4500 万字总会员点击量下，这个人设撑住了

**对 AI 的启示**：**"人设六字诀"是人物卡的生成模板**。让 AI 为每个新角色生成六个形容词短语，而不是"性格开朗的 XX"。

#### 《斗破苍穹》（天蚕土豆，2009）

- **套路化写作的标准制定者**：邪月评价"土豆定义了东方玄幻下商业化、针对主流群众的小说的标准"
- **黄金三章模板**：第一章剧烈冲突 / 第二章拿到金手指 / 第三章稍微打一下脸
- **"三十年河东三十年河西，莫欺少年穷"**：废柴逆袭的情感内核
- **缺点**：文笔一般 → 但**他会讲故事**

**对 AI 的启示**：
- 故事讲得好 ≠ 文笔好
- 套路不是贬义词，是可复用的结构模板
- 产品应当内置"套路模板库"而非只做"消 AI 味"

#### 《无限恐怖》（zhttty，2007）

- 无限流开山
- 主主题：**"真实的人，在剧烈变化的环境中不断面对选择"**（作者原话）
- 实质不是升级超人小说

**对 AI 的启示**：每个流派有作者自己的"控制观念"（controlling idea / 麦基术语）。预设里应让作者**显式声明控制观念**再开写。

### 8.3 集大成者层的技法贡献

#### 耳根《仙逆》《求魔》《我欲封天》

- **悲苦文风 + 虐主倾向**：区别于爽文的反方向深挖
- **核心价值**："顺为凡，逆为仙" —— 把"**不服命**"做成叙事基调
- **深远孤独感**：王林成长旅程孤独而强大
- **化凡章节的神来之笔**：原本平庸，连载三个月后"化凡"章节的写法突变，凭这一章奠定仙侠标杆地位

**对 AI 的启示**：
- 长篇可以"**中段爆发**"而非开篇即强
- **情绪基调（tone）应作为会话级变量**：一旦定为"悲苦+孤独"，后续生成不能混入轻快

#### 辰东《神墓》（辰东成名作 / 太古战争流 + 玄幻悬疑流开创者）

- **宏大叙事结构**：非平铺直叙，伴随情节一步步展露世界观
- **倒叙 / 插叙 交错**：紧张到"像在看悬疑"
- **古老神秘气氛**：开篇"神死了…魔灭了…我还活着…为什么？"—— 用极简句式制造巨大悬念
- **文笔评价**：不甚好、浮夸、"两位至尊大战至宇宙边荒"这种不接地气的句子
- **但是**：大世界观 + 大群像 + 吹牛式升级，读者买账

**对 AI 的启示**：**"极简开篇"是一个可生成的预设**。要求 AI 用 ≤5 句话 + ≤2 个概念词制造最大悬念；这类任务反而 AI 容易做好。

#### 猫腻《庆余年》《将夜》《择天记》《间客》

- **每部一个哲学命题**："活着还是不活 / 人应当怎样活着 / 公平和正义 / 自由和爱情 / 命运与选择"
- **"情怀"落在"生命"支点上**：书院"随心所欲"=="顺心意"
- **网文史论地位**：《间客》2013 西湖奖银奖 + 入选"中国网络文学 20 年 20 部作品"榜首
- **人物描写**：尤其女性"各有各的特点"

**对 AI 的启示**：**产品应当让作者显式写"本书的哲学命题"字段**，作为每章生成的价值观锚点。没有命题的长篇会散。

#### 爱潜水的乌贼《诡秘之主》

- **文笔 vs 题材匹配**（马伯庸评）："文笔优劣跟辞藻没关系，得跟题材相配。诡秘走 19 世纪工业英伦范儿，文笔自然要偏冷峻质朴"
- **22 条神之途径 / 9 大源质**：严密的系统设计 + 序列体系
- **稀缺性**：作者文笔好 + 阅历够 + 脑洞大 + 放下身段写爽文
- **新作《宿命之环》反面教材**：书名和大纲都是编辑选 / 开书前一个月才开始写大纲 → 伏笔完全崩盘，被粉丝骂惨
- **关键教训**：**长篇伏笔的核心是前期推演**，不是即兴

**对 AI 的启示**：
- **文风要跟题材匹配**：不是越优美越好，是越贴题越好
- **伏笔表是刚需**：产品必须提供"伏笔登记 + 回收追踪"表
- **大纲推演深度 > 下笔速度**

## 九、作者级 Novel Bible 实战结构（基于《没钱修什么仙》本地 Bible）

用户本地 `@d:\DESKTOP\novelfork\没钱修什么仙\` 文件夹是一套**真实作者磨出来的、为 AI 消费而设计的 Novel Bible 工程级范本**。其结构远比所有竞品 Codex 都要成熟。

### 9.1 顶层设计：按 arc 分卷

| 层级 | 目录 | 职责 |
|---|---|---|
| 入口 | `README.md` | 索引 / 使用方法 / 故事阶段 / 全局速查 |
| 主人格层 | `core_memory/` | 给 AI 主人格的"小而硬的稳定书设记忆包"，6 文件 |
| 原始档案 | `arc01-arc05/` | 可回溯、可展开、可查细节的分卷档案 |

**为什么分这两层**：
- 主人格查稳定书设 → 走 `core_memory`（token 便宜、覆盖高频）
- AI 查细节 / 回溯 → 走 `arcXX` 档案（token 贵但深）

这是**两级 RAG 结构**，产品必须原生支持。

### 9.2 每卷 10 种专题维度

每个 arc 文件夹内的 10 个专题：

| 维度 | 职责 | 竞品有否 |
|---|---|---|
| **事件.md** | 总时间线骨架，串起其余所有 | 部分（时间轴） |
| **世界观.md** | 张羽"当时能看到的世界"，**按时间线变化** | 仅静态 |
| **角色.md** | 按时间线记录姿态、转折、功能变化、成梗方向 | 仅静态人物 |
| **人物.md** | 静态索引，配合 `角色.md` | 有 |
| **羽书系统.md** | 按时间线记录主角系统 / 金手指的每次变化 | 无 |
| **势力机构.md** | 按时间线索引大学 / 院系 / 派系 / 证件 / 商业势力 | 无 |
| **梗与设定.md** | 制度笑点、成梗场景、关键设定的展开 | **全无** |
| **名场面与成梗场景.md** | 最强传播场面、反差桥段、高频回响 | **全无** |
| **现实映射与社会隐喻.md** | 对教育 / 借贷 / 资本 / 现实热点的映射 | **全无** |
| **伏笔与回收.md** | 关键线索埋下、推进、阶段回收与卷末状态 | 部分 |

**6 个竞品完全没有的维度**：
- 羽书系统（= 主角金手指 / 系统 的时间线演化）
- 势力机构（= 组织 / 派系 / 制度 的演化）
- 梗与设定（= 名场面 / 笑点 / 俚语 的时间线）
- 名场面与成梗场景（= 爆款桥段库）
- 现实映射与社会隐喻（= 文学性维度）
- 伏笔与回收（以"埋下 → 推进 → 回收 → 卷末状态"四态追踪）

### 9.3 纪律：时间线约束

README 第 20 行明确：

> "默认遵守时间线纪律：所有专题文档都尽量和 `事件.md` 共用同一条章节推进线，只写当时可知，不把后文认知提前回填。"

这个纪律保证了 **AI 在回答 arc02 相关问题时不会泄露 arc05 的底**。

**对 NovelFork 的直接启示**：**章节 / 时间戳 / 知晓半径**应该是数据库字段。每条 Bible 条目必须标注"本条目在第 N 章之后才可见"。

### 9.4 core_memory 设计

6 个核心记忆文件：

```
00_人格与时间线规则.md    — 主人格回答书设时必须遵守的规则
01_阶段总索引.md          — 5 阶段总入口、人物锚点、事件锚点、归段规则
10_炼气高中期.md          — arc01 骨架（只挑骨头）
20_筑基大学期.md          — arc02 骨架
30_金丹大学期.md          — arc03 骨架
40_元婴宗门期.md          — arc04 骨架
50_化神及以后.md          — arc05 骨架
```

**两条设计原则**：

1. **"小而硬"**：main persona 吃的是精选骨架，不是原始档案
2. **"不固化现实映射与社会隐喻"**：这类内容是分析而非书设，不进主人格

### 9.5 产品化映射

如果 NovelFork 要把这套结构产品化：

| 需求 | 数据库表设计 |
|---|---|
| arc 分卷 | `arcs(book_id, arc_no, title, start_chapter, end_chapter)` |
| 10 维度专题 | `bible_entries(arc_id, dimension, title, content_md, visible_after_chapter)` |
| core_memory 主人格硬记忆 | `core_memories(book_id, scope, title, content_md, token_budget)` |
| 时间线纪律 | 所有条目都有 `visible_after_chapter` 字段；AI 查询时自动过滤 |
| 事件时间线骨架 | `timeline_events(arc_id, chapter, title, related_entries)` |
| 伏笔四态 | `foreshadowings(book_id, id, state: {planted/pushing/resolved/stale}, plant_chapter, push_chapters[], resolve_chapter)` |
| 成梗场景库 | `memes(book_id, arc_id, scene_title, quote, spread_score)` |
| 现实映射 | `allegories(arc_id, theme: education/debt/capital/..., mapped_scene_refs[])` |

基于第八、九节的新素材，回头修订第七节的产品决策：

### 10.1 新增 P1 能力：**AI-native Novel Bible**（替代原第七节的 "story-bible"）

原来写的 "Story Bible / 世界观资料库" 太薄了。按《没钱修什么仙》实战结构，应该是：

- **两级 RAG**：core_memory（硬记忆）+ arc 档案（原始档案）
- **10 维度专题**：事件 / 世界观 / 角色 / 人物 / 系统 / 势力 / 梗 / 名场面 / 现实映射 / 伏笔
- **时间线纪律**：`visible_after_chapter` 字段 + 自动过滤
- **按 arc 分卷**：每卷独立维护
- **伏笔四态追踪**：planted / pushing / resolved / stale

### 10.2 新增预设库（补第七节 12 个之上）

追加 6 个"高级作者级"预设：

| # | 预设名 | 触发场景 | 依据 |
|---|---|---|---|
| 13 | **群像刻画** | 写到配角但缺乏区分度 | 玄鉴仙族群像；凡人修仙传人设六字诀 |
| 14 | **名场面设计** | 想让一章成为传播点 | 《没钱》成梗场景文件；赘婿爆点地图 |
| 15 | **现实映射融入** | 世界观需要深度 | 《没钱》现实映射；猫腻哲学命题 |
| 16 | **伏笔登记 / 回收检查** | 多卷长篇必需 | 《没钱》伏笔文件；诡秘之主反面教材 |
| 17 | **古风金句收束** | 每章或每卷结尾 | 《我本无意成仙》豆瓣评分核心 |
| 18 | **史料 / 数据考据接入** | 历史流或严肃题材 | 《大明王朝1627》知乎博主群 |

总计 18 个预设。

### 10.3 数据指标再扩

在第七节 7.4 的章节 metadata 基础上追加：

```
chapter:
  - sceneType            (爽点 / 卡点 / 日常 / 转折 / 名场面 / 伏笔埋设 / 伏笔回收)
  - pacingBeat            (15 节拍表中的位置：Catalyst / Midpoint / All Is Lost ...)
  - heroJourneyStage     (英雄之旅 17 阶段：召唤 / 拒绝 / 试炼 / ...)
  - characterArcBeats[]   (本章里哪些角色推进了弧线)
  - foreshadowingRefs[]   (本章触及的伏笔 ID 数组)
  - memeScore             (成梗潜力 0-100)
  - allegoryRefs[]        (本章映射的现实议题)
```

### 10.4 差异化定位修正

**修订前**（第七节结论）：
> NovelFork = "AI 时代的脂砚斋 + 麦基 + 番茄流量池分析师"

**修订后**（更准）：
> **NovelFork = AI-native Novel Bible × 作者级预设库 × 时间线纪律引擎**
>
> 核心差异点（没有任何竞品做到的）：
> 1. **时间线纪律**：AI 永远不会用"未来章节"的认知回填早期内容
> 2. **10 维度 Bible**：超越"人物 + 地点 + 设定"的竞品 Codex
> 3. **伏笔四态**：planted → pushing → resolved → stale 自动追踪
> 4. **成梗 / 现实映射**预设：文学性维度竞品从不涉及
> 5. **模拟器 / 多变体同章对比**：对接 2024-2025 年最火的模拟器流

## 十二、补充流派作者级解剖

### 12.1 女频两极：宅斗古言 × Priest 异质派

#### 传统宅斗古言系（晋江主力）

代表作谱系：
- **《庶女攻略》**（吱吱）：种田文开山鼻祖；起点女频书友推荐榜 6 年第一；风靡一代
- **《知否知否应是绿肥红瘦》**：借鉴《庶女攻略》架构；明兰人设"在家从父、出嫁从夫"的安稳路径
- **《九重紫》《庶得容易》《庶难从命》《高门庶女》《一瓯春》**：宅斗子品类矩阵

**共同技法**：
- 嫡母精明 / 生母懦弱 / 女主聪颖通透的**三角人设**
- **慢热细水长流**："绝对不是流水账" —— 但靠细节密度撑起
- **环境与富贵感描写**：服饰、家具、菜肴、礼节 —— 沉浸感的核心
- **红楼遗风**：部分作品（如《高门庶女》）有意向古典致敬
- **风险**：无爽感 / 无高潮 / 环境细节缺失 → 会被读者评为"太平静"

#### Priest 异质派（女性向突破）

- **晋江顶级大神**：2014 年起常居作者积分总榜榜首
- **多题材横跨**：玄幻 / 都市 / 奇幻 / 机甲 / 武侠
- **世界观突破**：邵燕君（北大）评价："世界观设定比一般女频作者要大，处理的女性问题更深，深到终于捅破了女性向的壁垒，抵达了普遍人性"
- **代表作**：《镇魂》（2012）/《默读》（豆瓣推理 Top100 第 25 名）/《杀破狼》（2015）/《大哥》/《天涯客》（山河令原著）/《有匪》/《残次品》
- **跨界**：《山河令》《镇魂》《有翡》都是影视改编；顶级 IP 价值

**对 AI 的启示**：
- 女频不等于"小世界"—— **世界观尺寸不应按性别向预设**
- **"感情关系 + 大世界观"可以兼得**，Priest 证明了
- 古言宅斗的**环境描写密度**是可专项训练的预设

### 12.2 武侠：大陆新武侠的式微与现代仙侠的崛起

#### 大陆新武侠（2001-2005 兴盛期 / 2006- 转型期）

- **代表作者**："南凤歌·北晴川·西未寒·东小椴"+ 沧月 / 步非烟 / 江南
- **凤歌《昆仑》**："金庸痕迹明显，但有独特东西；数学与武侠结合（生命天平）；古典意象如'西方巍巍，大哉昆仑'"
- **70 后作者群像**：几乎清一色 75 后，向金庸时代致敬
- **结局**：武侠式微，读者转向玄幻仙侠
- **江南的转向**：放弃武侠、研究市场、写《龙族》—— "为了复兴武侠肯定得想办法迎合观众老爷"

#### 萧鼎《诛仙》（2003）

- **仙侠流派开创性经典**
- **主旨**："天地不仁，以万物为刍狗"（贯穿始终的哲学母题）
- **感情描写**：陆雪琪 / 碧瑶的三角；"描写感情十分细腻，为小说增色不少"
- **源流**：作者自述最大影响是**还珠楼主《蜀山剑侠传》**（仙侠真正的源头）
- **幸运**：2002 年就拿了台湾买断 2.8 万（网文作家最穷时期的一笔巨款）

**对 AI 的启示**：
- **武侠 → 仙侠是话语权迁移**：现代读者古学基础薄，古典江湖写不进去
- **感情戏与世界观并重** 是经典仙侠必备
- **单一哲学母题贯穿** 是长篇骨架的核心（参考 v0.2 猫腻命题部分）

#### 梦入神机《佛本是道》（2006）+《龙蛇演义》+《阳神》

- **洪荒流开山**：一本书创造了整个**洪荒封神体系**
- **"神机一出，谁与争锋"**
- **集成能力**：糅合《山海经》《封神演义》《西游记》《蜀山剑侠传》+ 民间传说
- **设定编造能力**："十二祖巫"已成网文固有名词；"跟脚来历、法术修成，一套自洽逻辑"
- **《龙蛇演义》国术流开山**：明劲 → 暗劲 → 化劲 —— 当代都市武修文的元设定
- **《阳神》**：武道证道系的集大成
- **评价**：梦入神机是"近代神话奠基人级别候选"

**对 AI 的启示**：
- **神话体系 = 可预制数据库**：十二祖巫 / 混元大罗金仙 / 先天灵宝等条目应有模板
- **国术体系**（明劲/暗劲/化劲/抱丹/返虚）同样是可复用资产
- **"跟脚来历"模板**：每个神话人物都可按固定字段展开（出身 / 师承 / 法宝 / 斩尸 / 劫数）

### 12.3 盗墓灵异：鬼吹灯 / 盗墓笔记双雄

#### 天下霸唱《鬼吹灯》（2006）

- **盗墓小说鼻祖**（无争议）
- **标志性口诀**："人点烛，鬼吹灯，鸡鸣灯灭不摸金"
- **核心贡献**：奠定盗墓世界的基础设定；"粽子"（僵尸）等词都是作者自创，现已成行业黑话
- **集陵寝文化 + 探险于一体**
- **志怪谱系**：《山海经》→《搜神记》→《聊斋志异》→ 鬼吹灯 的民间志怪传统
- **奇闻异事功底**：比三叔强

#### 南派三叔《盗墓笔记》（2007-）

- **起源**：一开始是《鬼吹灯》同人
- **三个关键改变**：把盗墓做成悬疑；把"八爷 / 三叔"做成 IP；主角小哥张起麟的神秘感
- **悬疑 > 盗墓**：整个故事的框架设计得很大；"谜团一个接一个，大部分至今没浮出水面"
- **问题**：龟速更新；作者直言"如果哪天心情不好可能就不写了"

**对 AI 的启示**：
- **"行业黑话预制表"**：摸金 / 发丘 / 搬山 / 卸岭四派；粽子 / 大粽子 / 老熊岭 等词汇
- **悬疑驱动 vs 探险驱动** 是两种节奏模板
- **长篇伏笔悬疑的 token 预算管理**：不能像三叔那样"谜团堆到收不回来"

### 12.4 科幻硬核：从粗放到精致

#### 玄雨《小兵传奇》（2004 左右）

- **早期科幻网文代表**
- 288 章，唐龙从训练营到大唐帝国到统一宇宙
- **核心叙事模式**：**小兵成长 + 星际战争 + 宇宙一统**
- **名场面导向**：起点自己做过"票选网文神级名场面"活动，小兵传奇的战术桥段常上榜
- **爽文骨架 + 科幻皮**

#### 《重生之超级战舰》（反面教材参考）

豆瓣评论典型问题：
- "文学性太差，读到走出太阳系就读不下去"
- **全片只有一个角色的设定**
- **全篇内心独白**（科幻长篇的致命伤）
- "流水账文学的可行发展方向"

**对 AI 的启示（科幻流负面清单）**：
- **主角 + 配角平衡**：不能只让主角思考
- **内心独白阈值监控**：单章超过 30% 应触发警告
- **科幻硬核的"设定密度 vs 叙事节奏"**：设定过多会吞噬情节

### 12.5 网游 / 电竞：《全职高手》群像教科书

#### 蝴蝶蓝《全职高手》（2011-2014）

- **起点 2013 年度票数总冠军 + 金键盘奖**
- **首部"千盟书"**（累计 1690 个盟主）
- **英文版 The King's Avatar** 出海
- **群像塑造教科书**：
  - 15 支战队（霸图 / 百花 / 嘉世 / 蓝雨 / 雷霆 / 轮回 / 微草 / 兴欣 / 虚空 / 烟雨 / 义斩 / 呼啸 / 皇风 / 301 / 神奇）
  - **即使篇幅最少的 301 也有杨聪（刺客-风景杀）这样"非本命粉丝都能说出特点"的配角**
- **主角叶修的多身份**：职业联赛叶秋大神 / 网吧网管 / 战术大师 / 可靠大哥 / 满级嘲讽队长
- **现实感**：电竞是真实职业，不同于纯游戏文
- **职业-现实交叉**：Ch1245 十年组合韩文清 & 张新杰握手拥抱；Ch877 叶修喊人打本

**对 AI 的启示（群像一等公民化）**：
- **配角不应是"工具人"**，应有**至少一个辨识度标签**
- **人物表 schema**：每个角色需有 `identity_tags[]`、`signature_move`、`relationship_map`
- **大事件交叉引用**：AI 应能记住"第 N 章谁和谁发生了什么"（即 NovelBible 里的 event 条目关联 character）

### 12.6 搞笑沙雕 / 轻小说：已成主流

#### 代表作脉络

- **《异常生物见闻录》**（远瞳）：房东接待神委托的奇葩房客
- **《史上第一混乱》**（张小花）：沙雕房东 + 穿越奇葩
- **《三千美娇娘》**（紫钗恨）：打着种马旗号的**讽刺小说** —— 表皮 ≠ 内核
- **《不许没收我的人籍》**：游戏公司主美重生 + 制卡师 + 话疗

#### 现状信号

> "现在的网络小说，搞笑玩梗的沙雕文成为了主流。现在最流行的是轻小说，玩梗，讲段子，抖包袱，作者不再怎么重视情节和人物刻画了。市场决定产出。"

**警告信号**：
- 轻小说化有反噬 —— 只玩梗不立人物，作品寿命短
- **反套路初看有趣，套路重复没创新就捡不起来**
- 沙雕爽文剧情不能较真，细究合理性就崩

**对 AI 的启示**：
- **"梗密度"** 应成为章节可选 metadata
- **"反套路套路化"预警**：同一反套路重复 3+ 次触发告警
- **沙雕文的"逻辑兜底"**：AI 生成的段子需要最低限度的合理性过滤

### 12.7 都市历史：齐橙 / 骁骑校 / 宰执天下

#### 骁骑校《橙红年代》

- **网络第一本点击过亿的都市小说**
- **"八年前是畏罪逃亡的烤肠小贩，八年后是经历过血与火考验的铮铮硬汉"**
- **底层逆袭模板**：保安 → 硬汉（硬汉文在都市流的核心形态）

#### 《宰执天下》（cuslaa）

- **北宋穿越历史流顶配**
- **作者史学功底深厚**：宋代制度、政治、经济、风俗信手拈来
- **"宰者宰相，执者执政。上辅君王，下安黎庶"** —— 一句话立住全书格调
- **在中国网络文学评论家心中排名第一的网络历史小说**

#### 齐橙《中医许阳》等

- **龚江辉，北师大副教授 / 中国作协会员 / 阅文签约作家**
- **用网络文学手法写现实题材**
- 系统设定把主角送到不同历史节点，亲历"新中国成立以来中医参与历次重大疫情防治"
- **时间跨度 + 空间范围 + 精神内核**

**对 AI 的启示**：
- **历史流的"史料密度 > 文笔"** —— 数据和制度要立得住
- **现实题材需要"系统"作为时空跳板**：否则容易变说教
- **一句话立全书格调** 是可训练的开篇预设

1. 过度正式 / 官腔感
2. 固定句式模板（"首先/其次/最后"/"综上所述"）
3. 典型 AI 词汇（"值得注意的是"/"以下是为您"/"让我们一起来"）

8. 行话 / 技术术语密度高于实际需要
9. 形容词堆叠（"璀璨夺目、绚丽多彩"两连发）
10. 缺少具体细节，空话多（"他感到很开心"而不是写具体动作）

12. **"伪人感"**（编辑圈专业术语）—— 总体像但细节不对

1. **喂人类范文** → AI 学风格
2. **明确人格提示词**（"你是网文作者，轻松诙谐，脑洞清奇，台词幽默"）

## 十八、NovelCrafter Codex 范式（动态追踪的标杆）

NovelCrafter 的 **Codex** 是目前业界动态 Bible 的最佳实践，直接影响 NovelFork 的 Bible schema 设计。

### 18.1 三种条目可见性（核心设计）

| 类型 | 规则 | 典型内容 |
|---|---|---|
| **tracked** | NC 在场景文本里看到条目名时，才注入给 AI | 角色、地点、物品（绝大多数内容） |
| **global** | 始终对 AI 可见 | 文风、流派、全书主题、核心规则 |
| **nested** | 条目 A 引用条目 B 时，自动带入 B | 组织层级、家族成员、阵营关系 |

**这三档可见性解决了"条目太多 → 塞满上下文 → token 爆炸 + AI 注意力稀释"的经典难题**。v0.2 我设计的"visible_after_chapter"只是时间维度，而 NovelCrafter 加了**空间维度（场景提到才注入）**，两者正交。

### 18.2 Progressions 与自动链接

- 写 `Sarah entered the café` → 系统自动链接 Sarah 和 café 条目
- 角色弧线可按章节推进（progressions），每次进度更新只追加不覆盖
- AI 上下文的"场景级控制"（per-scene AI context）

### 18.3 Sudowrite vs NovelCrafter 哲学分野

| 工具 | 哲学 | 适合 |
|---|---|---|
| **Sudowrite** | brain dump + AI 自动补，Story Bible 静态 | 脑洞型 / 花园派作者 |
| **NovelCrafter** | Codex 严格管理，自动链接，大纲驱动 | 情节型 / 建筑师派作者 |

**对 NovelFork 的启示**：**同时支持两种哲学**，作者可切换"静态模式"（Sudowrite-like）和"动态追踪模式"（NovelCrafter-like）。默认静态，作者手动开启追踪。

## 二十、真实作者痛点（四线访谈）

### 20.2 崩盘痛点

- 200 万字以上长篇 **追订崩盘**（起点 L5 作家十万菜团专题）
- 开头精彩、后续乏力 = **没搭框架或没按大纲写**
- **"烂尾是能力问题，太监是态度问题"**（塔读圈名言）
- 99% 新人止步于"**负反馈多 + 屡投不过 + 未来不确定**"

**8 个字段**。不需要我 v0.3 设计的 10 维度 × 两级 RAG × 伏笔四态那种复杂度。

| **提示词批量测试** | 作者："我这 10 个提示词哪个更好" | 对照生成 + AI 味评分 |

### 21.3 双模界面

| 模式 | UI | 默认 | 成本 |
|---|---|---|---|
| **作者模式** | 传统编辑器 + Bible + AI 写作 | 默认 | 普通 token 消耗 |
| **工作台模式** | 暴露 terminal / 文件树 / agent 日志 | 按需开启 | 高 token（但只在需要时） |

**关键设计原则**：**作者模式 UI 零 coding 暴露**。"工作台模式"是**后台武器**，不是"前台功能"。

## 二十二、v0.4 最终定位（取代 v0.3）

### 22.1 一句话（v0.4）

> **NovelFork = 通用网文 AI 写作软件，核心是 4 表 Bible + 动态追踪 + 强制 AI 味过滤 + 按需 coding agent。不预制流派资产，提供模板市场与 agent 生成能力。**

### 22.2 必装核心（不可砍）

1. **4 个一等公民表**：角色 / 事件 / 设定 / 章节摘要
2. **三种可见性**（借 NovelCrafter Codex）：tracked / global / nested
3. **时间线纪律**：visible_after_chapter（时间轴）
4. **AI 味过滤**：默认开启、不可关闭、内置朱雀级检测 + 12 特征
5. **双写作哲学支持**：静态（Sudowrite-like）/ 动态（NovelCrafter-like）

### 22.3 可选（模板市场）

1. **流派模板**（26 选，v0.3 继续保留，但改为**可选导入**而非强推）：每个包 = 默认设定条目 + 提示词预设 + 推荐节拍
2. **节拍模板**（英雄之旅 / 15 节拍 / 三幕 / 自定义）
3. **文风 tone 预设**（悲苦 / 冷峻 / 古典 / 黑色幽默 / 沙雕）
4. **消 AI 味 7 招预设**

### 22.4 按需 coding agent（NovelFork 独家基因）

- **默认关闭**，作者模式 UI 零暴露
- **工作台模式开关**
- **典型场景**：资产生成 / 数据抓取 / 朱雀检测 / 扫榜 / 一致性审计 / 提示词 AB

### 22.5 不做（明确排除）

- ❌ 强推流派锚定流程（改为可选引导）
- ❌ 预装数据库（1018 条本草 / 200 条神话 / 50 条黑话 ...）—— 改为 agent 按需生成
- ❌ 10 维度 Bible / 两级 RAG / 伏笔四态 这种"文字修仙 docs 级别"复杂度
- ❌ 科学锚点 / Nature 引用字段 这种"特定作者偏好"（但支持**作者自定义字段**）
- ❌ 伟大的百年推演表这种游戏向复杂度（但支持**自定义专题文档**）

### 22.6 名字冲突 / 项目关系（必须处理）

| 关系 | 状况 | 建议 |
|---|---|---|
| `novelfork` vs `RhythmicWave/NovelForge` | 大小写不同但读音一致 | **确认是否需要改名 / 差异化定位** |
| `vivy1024/inkos` vs `Narcooo/inkos` | 用户 workspace CorpusName 对应 | **澄清**是自家镜像还是同名巧合 |
