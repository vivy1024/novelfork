# NovelFork 技术栈选型

**更新日期**: 2026-04-20  
**状态**: 当前有效口径

---

## 目标技术路线

NovelFork 的目标不是继续做一个 `PWA + Node + pnpm + tsc` 的源码运行项目，
而是做成 **NarraFork 风格的本地单体应用**。

因此当前技术栈选择遵循以下优先级：

1. **Bun 优先** — 运行时与最终打包都围绕 Bun 设计
2. **单入口应用优先** — 前后端围绕一个本地服务入口组织
3. **单可执行文件优先** — 以 `bun compile` 产物为主要分发形态
4. **核心能力保留** — 已有的 React / Hono / TypeScript / agent 管线按可复用原则保留

---

## 目标平台模型

### 目标交付形态

```text
novelfork.exe / novelfork
  ├─ 内嵌前端静态资源
  ├─ 本地 Hono server
  ├─ 本地 SQLite 数据
  ├─ AI provider 配置
  └─ 浏览器打开 localhost 界面
```

这条路线更接近 NarraFork 当前已验证的产品形态，而不是传统的源码运行 Web 项目。

### 目标启动方式

- 用户启动单个可执行文件
- 程序拉起本地服务
- 程序自动打开浏览器页面
- 前端由内嵌静态资源直接提供

不再把以下方式当成最终目标：
- 先装 Node/pnpm
- 再 `pnpm install`
- 再 `pnpm build`
- 再 `novelfork studio`

这类流程最多只是迁移期的开发模式。

## 保留的技术

### 1. TypeScript
保留原因：
- 当前核心代码已是 TypeScript
- agents / pipeline / state / parser 全部建立在 TS 类型系统之上
- 重写收益低，迁移成本极高

结论：**保留**。

### 2. React
保留原因：
- 当前 studio 界面已基于 React 19 实现
- 多窗口、拖拽、编辑器、设置面板等已有资产都在 React 上
- NarraFork 本身也是 React 路线

结论：**保留**。

### 3. Hono
保留原因：
- 轻量
- 适合本地 HTTP server
- 同时兼容 Node/Bun
- 已有 API 路由层可以继续演化

结论：**保留**。

### 4. SQLite
保留原因：
- 本地单机工具最合适的数据库
- 当前世界状态、记忆、索引都适合落 SQLite
- 与 NarraFork 的本地状态模型不冲突

结论：**保留**。

## 需要降级为“过渡方案”的技术

### 1. pnpm workspace
当前作用：
- 管理 `packages/core`、`packages/cli`、`packages/studio`

问题：
- 更适合库/工具链项目
- 不适合最终产品交付模型
- 容易强化“多包、分层运行、源码启动”的惯性

结论：
- **短期保留用于迁移**
- **长期不应成为产品形态中心**

### 2. tsc -> dist 构建驱动
当前作用：
- 为 core/cli/studio server 生成 `dist/`

问题：
- 让运行模式依赖构建产物而不是源代码入口
- 让 CLI / studio / server 分裂得更明显
- 与 Bun 原生运行、最终 `bun compile` 路线不一致

结论：
- **短期保留做兼容**
- **长期应退出主链路**

### 3. packages/cli
当前作用：
- 充当命令行入口
- 再去启动 studio / pipeline

问题：
- 最终产品目标不是“命令行工具 + 可选 web 面板”
- 而是“一体化本地应用”

结论：
- **CLI 命令能力可保留**
- **但 `cli -> studio spawn` 不是长期结构**

### 4. PWA
当前状态：
- 前端已接入 `vite-plugin-pwa`
- 有 Service Worker 和 manifest

问题：
- PWA 可以是一个增强能力
- 但不能再被当成平台目标本身

结论：
- **PWA 可留作前端增强**
- **不再作为产品路线的主叙事**

## 需要重点评估的替换项

### 1. 数据库层：Node 内置 SQLite vs better-sqlite3 + drizzle
当前仓库主要走 Node 原生 SQLite 封装。

需要评估的问题：
- Bun 兼容性是否理想
- schema 演化是否足够稳定
- 未来管理复杂查询时，是否需要 ORM 与 migration 体系

倾向：
- **短期先不为了“像 NarraFork”而盲换**
- **中期认真评估 `better-sqlite3 + drizzle` 是否值得接入**

原则：
- 先回正平台外壳
- 再决定是否替换 DB 基础设施

### 2. 终端/PTY 层
如果未来 NovelFork 也要具备更强的内嵌工具链，
可参考 NarraFork 的：
- PTY
- 内嵌终端
- MCP 工具注册

但这不是当前第一优先级。

## 当前推荐技术组合

### 迁移期
- TypeScript
- React 19
- Hono
- Vite（仅作为前端构建器）
- SQLite
- pnpm workspace（临时）

### 目标态
- Bun
- TypeScript
- React

- SQLite
- `bun compile`
- 单入口本地应用

## 不再使用的旧结论

以下旧结论从现在开始失效：

- “PWA 优先，未来再考虑 Bun”
- “Node.js + pnpm 是稳定主路线”
- “Tauri/SEA 失败说明单体本地应用路线不成立”
- “CLI / studio / desktop 三线长期共存”

---

## 决策原则

以后新增或修改技术时，优先问这三个问题：

1. 它是否让 NovelFork 更接近单入口本地应用？
2. 它是否有利于最终 `bun compile` 单文件分发？
3. 它是在保留核心能力，还是继续加重旧外壳负担？

如果三个问题里大多数答案是否定的，这项技术决策大概率就是偏航。
