# 平台纠偏说明

**更新日期**: 2026-04-20  
**目的**: 把“NovelFork 到底应该朝哪里走”一次性说清楚

---

## 结论先行

NovelFork 过去的主要问题不是功能没做出来，而是：

> **把 NarraFork 误当成了 UI/功能参考，而不是平台原型。**

结果是：
- 交互层越来越像 NarraFork
- 运行时、构建、分发却越来越像 InkOS/Web IDE

从现在开始，唯一有效方向是：

> **NovelFork 应回归 NarraFork 风格的本地单体应用路线。**

即：
- Bun 作为首选运行时
- 本地 HTTP 服务 + 内嵌前端资源
- 单入口应用
- 最终通过 `bun compile` 生成单可执行文件

---

## 为什么会偏航

根据 git 历史与旧 docs，可归纳为四个原因。

### 1. 来源被错误拆分
旧文档长期把项目来源写成：
- InkOS = 核心代码基础
- NarraFork = UI/UX 与功能参考
- Claude Code = 实现技术参考

这个表述的问题是：
它把 NarraFork 从“平台原型”降级成了“交互参考”。

于是后续实现自然变成：
- 保留 InkOS 的 monorepo / CLI / studio / tsc / pnpm 外壳
- 只把 NarraFork 的聊天、多窗口、Worktree、设置/套路/管理抄进来

### 2. Tauri / SEA 的失败被错误归因
仓库里其实走过两条更接近 NarraFork 的路：
- Tauri 本地桌面化
- Node SEA / 单文件可执行打包

后来这些方案遇到阻力后，项目做出的结论不是：
- “换更适合的一体化壳继续做”

而是：
- “说明本地单体路线不合适，转向 PWA 更务实”

这个结论是错的。真正证明失败的只是：
- **Tauri 这条线不适合当时的实现方式**
- **Node SEA 这条线不适合当时的打包方式**

并不代表单体本地应用方向错了。

### 3. InkOS 的工程惯性太强
当前仓库天然继承了：
- `packages/core`
- `packages/cli`
- `packages/studio`
- `packages/desktop`
- `pnpm workspace`
- `tsc -> dist`
- CLI 再去启动 studio

这套结构更像“框架/工具链项目”，而不是“成品应用”。

一旦沿着这条线继续做，交付物就会越来越像：
- 需要源码运行
- 需要安装依赖
- 需要区分 CLI / studio / desktop

而不是 NarraFork 那种“单体、即开即用”的产品。

### 4. 错误方向被 docs 固化
旧 docs 里大量出现：
- PWA 优先
- Node.js + pnpm + Vite + Hono
- Tauri 不推荐
- SEA 已移除
- v0.0.1 准备公开发布
- 文档状态：生产就绪

一旦这些说法写成正式文档，后续实现、测试、发布都会围着它转。

所以偏航不是“偶然写歪了代码”，而是：

> **docs 先定义错了目标，代码只是忠实跟着错目标走。**

---

## 当前正确目标

### 平台目标
NovelFork 应该是：
- 面向个人创作者的本地 AI 写作工作台
- 以 Bun 为基础运行时
- 以单入口服务承载前端与后端
- 以单可执行文件作为主要分发形态

### 不再使用的旧口径
以下口径不再成立：
- “PWA 是平台目标”
- “Node + pnpm + tsc 是稳定长期方案”
- “CLI / studio / desktop 长期并存是目标结构”
- “先做源码运行和发布准备，再考虑 Bun”

### 仍然可保留的东西
并不是一切都要推翻。

可复用资产主要有：
- `packages/core` 的写作管线
- agents、parser、state、审计逻辑
- `packages/studio` 里的部分 UI 与 API 组织
- 已实现的网文行业能力

真正要回正的是：
- 运行时
- 构建链路
- 启动方式
- 分发方式
- 对外文档口径

---

## 今后如何判断有没有继续偏航

一个方案如果符合下面大多数条件，就算在朝 NarraFork 回正：

### 这是正确方向
- 减少多入口、多包相互启动
- 强化单入口 server
- 减少 `dist/` 驱动的运行模式
- 强化 Bun 原生运行与打包
- 优先考虑最终 exe 分发体验
- 让前端资源服务于本地单体应用，而不是独立 Web 部署

### 这是继续偏航
- 把 PWA/浏览器安装当作最终平台答案
- 继续围绕 `pnpm -r build` 组织产品形态
- 继续依赖 CLI 去启动 studio
- 把源码运行体验当作最终用户体验
- 只补 UI，而不碰构建/运行时/分发层

---

## 当前执行原则

在平台回正完成前，所有实现决策默认遵循：

1. **核心写作能力优先保留**
2. **错误的外壳可以重写**
3. **文档优先统一口径**
4. **平台目标以 NarraFork 为准，不以旧 PWA 文档为准**
