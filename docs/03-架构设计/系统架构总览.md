## 📋 架构概览

NovelFork 是专注于中文网文小说创作的 AI 辅助工作台，采用多 Agent 协作管线架构，实现规划、编排、写作、审计、修订全流程自动化。

### 项目来源

NovelFork 的架构设计综合了多个来源的最佳实践：

1. **核心代码基础**：Fork 自 [InkOS v1.1.1](https://github.com/Narcooo/inkos)
   - 多 Agent 协作写作管线
   - 长期记忆系统（7个真相文件）
   - 连续性审计机制（33维度）

2. **功能设计参考**：[NarraFork](https://github.com/Narcooo/narrafork)
   - UI/UX 交互设计
   - 功能优先级规划
   - 用户体验优化

3. **实现技术参考**：[Claude Code](https://github.com/anthropics/claude-code)
   - Git Worktree 多线并行实现
   - 多窗口卡片系统
   - 上下文管理机制
   - 虚拟滚动优化
   - 详见 [04-学习资料/](../04-学习资料/)

4. **开源项目调研**：调研了 15+ 个开源 AI 写作项目
   - 各类 AI 小说生成器
   - Agent 框架（LangChain、AutoGPT）
   - 商业产品（NovelAI、Sudowrite）

## 🎯 核心设计理念

### 1. 多 Agent 协作管线

**原理**: 8个专业 Agent 接力完成章节创作

```
用户请求 →
雷达扫描趋势 →
规划师制定意图 →
编排师选择上下文 →
建筑师规划结构 →
写手生成正文 →
观察者提取事实 →
审计员检查质量 →
修订者优化问题 →
输出成品章节
```

**优势**:
- ✅ 专业分工，各司其职
- ✅ 流水线并行，提高效率
- ✅ 质量保障，自动修订
- ✅ 零人工干预

### 2. 长期记忆系统

**架构**: 7个真相文件 + SQLite 时序数据库

```
结构化 JSON (Zod schema 校验)
         ↓
    真相文件权威来源
         ↓
  Markdown 人类可读投影
         ↓
   SQLite 时序检索
```

**数据规模**:
- 7个真相文件（JSON 格式，Zod 校验）
- SQLite 时序记忆数据库（Node 22+）
- 按相关性检索历史事实
- 避免全量注入导致上下文膨胀

### 3. 连续性审计

**架构**: 33维度检查 + 自动修订循环

```
草稿生成 →
33维度审计 →
发现问题？
  ├─ 是 → 自动修订 → 再审计
  └─ 否 → 通过发布
```

**审计维度**:
- 角色记忆（是否记起未见过的事）
- 物资连续性（是否拿出已丢失的物品）
- 伏笔回收（是否遗忘铺垫）
- 大纲偏离（是否偏离主线）
- 叙事节奏（是否节奏失控）
- 情感弧线（是否情感突变）
- AI 痕迹检测（是否有 LLM 味）
- ... 共33个维度

### 4. 去 AI 味

**策略**: 词汇疲劳词表 + 禁用句式 + 文风指纹

```
写手 Agent Prompt 层
    ↓
词汇疲劳词表（高频词检测）
    ↓
禁用句式（句式单调检测）
    ↓
文风指纹注入（统计特征匹配）
    ↓
从源头减少 AI 生成痕迹
```

**机制**:
- ✅ 内置词汇疲劳词表
- ✅ 禁用句式规则
- ✅ 文风指纹注入
- ✅ 审计维度检测
- ✅ 专门反检测改写

```
   ↓

   ↓
```

## 🎨 设计模式

### 1. 管道模式（Pipeline）

```typescript
class WritingPipeline {
  private stages: Stage[] = [
    new RadarStage(),
    new PlannerStage(),
    new ComposerStage(),
    new ArchitectStage(),
    new WriterStage(),
    new ObserverStage(),
    new AuditorStage(),
    new ReviserStage()
  ];

  async execute(input: Input): Promise<Output> {
    let context = input;
    for (const stage of this.stages) {
      context = await stage.process(context);
    }
    return context;
  }
}
```

### 3. 观察者模式（事件系统）

```typescript
class EventEmitter {
  private listeners: Map<string, Function[]> = new Map();

  on(event: string, callback: Function) {
    if (!this.listeners.has(event)) {
      this.listeners.set(event, []);
    }
    this.listeners.get(event)!.push(callback);
  }

  emit(event: string, data: any) {
    const callbacks = this.listeners.get(event) || [];
    callbacks.forEach(cb => cb(data));
  }
}

// 使用
pipeline.on('stage:complete', (stage) => {
  console.log(`${stage.name} completed`);
});
```

### 4. 单例模式（配置管理）

```typescript
class ConfigManager {
  private static instance: ConfigManager;
  private config: Config;

  private constructor() {
    this.config = this.loadConfig();
  }

  static getInstance(): ConfigManager {
    if (!ConfigManager.instance) {
      ConfigManager.instance = new ConfigManager();
    }
    return ConfigManager.instance;
  }

  get(key: string): any {
    return this.config[key];
  }
}
```
