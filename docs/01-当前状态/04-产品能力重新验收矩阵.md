## 1. 文档目的

本矩阵用于执行 `claude-codex-novel-agent-v1` 的 Phase 0 Task 1：重新核对 NovelFork 原始产品能力承诺与当前用户可见路径之间的差距。

此前许多 archived specs 已标记完成，但其中不少完成项属于 API、类型、组件、route、测试、合同、接线、退役或口径守卫。它们是有价值的底层资产，但不能自动等同于当前 Studio、CLI 或 headless 已经完成端到端功能。

本矩阵只回答一个问题：

> 用户今天能否在当前 NovelFork 主路径中真实完成这项能力？

---

## 2. 状态口径

| 状态 | 含义 | 是否可宣传为当前完整功能 |
|---|---|---|
| `current` | 已有 Studio 或 CLI/headless 端到端证据，且结果持久化、可恢复、失败透明 | 可以，但必须附证据 |
| `partial` | 有真实底座或部分路径，但未覆盖完整产品闭环 | 不可以 |
| `not-wired` | core/API/component 存在，但没有接入当前主路径或用户操作不自然 | 不可以 |
| `planned` | 已明确规划但当前未实现 | 不可以 |
| `unsupported` | 当前明确不支持，UI/API 应透明说明 | 不可以 |
| `non-goal` | 明确不做或不在本版本目标内 | 不可以 |

### 证据规则

- API 测试只能证明 route 存在。
- 组件测试只能证明 UI 可渲染。
- 单元测试只能证明内部函数行为。
- 文档、能力矩阵、spec checkbox 只能证明历史记录。
- **只有 Studio 浏览器路径或 CLI/headless 路径跑通，才能作为功能完成证据。**

## 5. 当前优先修复顺序

1. **Runtime 同源**：先把 Studio、CLI、headless 的 session、events、commands、tools、permissions、transcript 统一。
2. **设置/套路接实**：让设置和套路页真的影响 runtime，而不是只显示状态。
3. **`/novel:write-next` 闭环**：以叙述者为主入口，跑通 context → PGI → Guided Plan → approve → candidate → canvas。
4. **CLI/headless 同源验收**：同一写作任务必须能通过 `novelfork exec --output-format stream-json` 跑到同样边界。
5. **写作模式/工具回主流程**：选段写作和健康/叙事线工具回到当前画布与资源面。
6. **重定发布标准**：等端到端能力真实后，再决定恢复 v0.1.0 还是转向 v0.2.0。

---

## 6. 本矩阵的维护规则

- 每完成一个 `claude-codex-novel-agent-v1` 任务，必须更新相关行状态和证据。
- 能力从 `partial/not-wired/planned` 升为 `current` 前，必须补充 Studio 或 CLI/headless 端到端证据。
- 如果证据只有 API、组件、unit test，则最多只能标 `partial`。
- 如果能力只出现在 archive spec checkbox 中，当前路径不可用，则必须标 `not-wired`。
- 如果能力与 Claude/Codex 对标有关，必须说明真实源码/官方行为/可运行 CLI 证据来源。
