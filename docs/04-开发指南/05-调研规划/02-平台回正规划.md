# NovelFork 平台回正规划

**更新日期**: 2026-04-20  
**目标**: 回归 NarraFork 路线，而不是继续强化旧的 PWA/Node/tsc 外壳

---

## 总体阶段

| 阶段 | 目标 | 状态 |
|------|------|------|
| Phase 0 | 统一 docs 与平台口径 | 当前进行中 |
| Phase 1 | 运行时回正（Bun 主入口） | 待开始 |
| Phase 2 | 构建与分发回正（单入口 + `bun compile`） | 待开始 |
| Phase 3 | 数据层与基础设施评估 | 待开始 |
| Phase 4 | UI 与写作能力融合收口 | 待开始 |

---

## Phase 0：口径统一

### 目标
先停止继续按错路线开发。

### 产出
- docs 统一为 NarraFork 路线口径
- 明确当前哪些内容是历史资料
- 删除误导性的发布/假完成文档

### 完成标准
- `docs/README.md` 成为唯一总入口
- 架构与规划区只保留当前有效口径
- 旧的发布准备文档不再存在于主 docs 导航

---

## Phase 1：运行时回正

### 目标
从“Node + CLI + studio 多入口”收敛到 “Bun 主入口”。

### 关注点
- 找出当前真正的应用主入口应放在哪里
- 逐步减少 `cli -> studio spawn` 依赖
- 明确哪些 server 逻辑可以直接以 Bun 运行

### 完成标准
- 有一个清晰的主应用入口
- 在开发模式下可直接由 Bun 启动主应用
- 旧 CLI 启动链路降级为过渡方案

---

## Phase 2：构建与分发回正

### 目标
把交付方式从“源码运行项目”转成“单体本地应用”。

### 关注点
- 前端静态资源如何内嵌
- server 与 UI 如何在单入口下协作
- `bun compile` 的构建链路
- 启动时自动打开浏览器

### 完成标准
- 有稳定的单入口打包流程
- 能生成单个可执行文件
- 用户不再需要先理解 workspace / build / link / studio 命令

---

## Phase 3：数据层与基础设施评估

### 目标
评估当前基础设施是否继续沿用，还是向 NarraFork 靠拢。

### 重点问题
- 现有 SQLite 封装是否继续保留
- 是否引入 `better-sqlite3 + drizzle`
- MCP / 工具注册 / 终端能力是否需要增强

### 原则
- 先看是否真的服务于平台回正
- 不为了“像 NarraFork”而盲目替换一切

---

## Phase 4：UI 与写作能力融合收口

### 目标
在新的平台壳上，重新整合现有写作能力。

### 关注点
- 当前 studio 中哪些 UI 直接迁移
- 哪些交互应该保留
- 哪些旧的“Web IDE 习惯”应当删掉

### 完成标准
- 用户面对的是统一产品，而不是若干包拼装起来的工程
- 核心写作能力仍完整可用
- 产品形态与 NarraFork 路线一致

---

## 当前不做的事

在平台回正完成前，默认不优先推进：
- 假发布准备
- 继续围绕 PWA 做包装叙事
- 继续堆与平台回正无关的 UI 小功能
- 为旧外壳补更多测试来强化旧路线
