# NovelFork 平台迁移 Backlog

**更新日期**: 2026-04-20

---

## 使用说明

本 backlog 只记录真正服务于“回归 NarraFork 路线”的任务。

以下类型的任务默认不列为当前主 backlog：
- 继续包装旧的发布准备
- 继续强化旧 PWA 叙事
- 与平台回正无关的零碎 UI 美化

---

## P0：方向锁定

- [ ] 完成 docs 纠偏
  - 重写架构与规划主文档
  - 删除误导性发布文档
  - 明确 archive 仅作历史参考

- [ ] 画出当前真实启动链路
  - 从 `packages/cli` 到 `packages/studio` 到 `packages/core`
  - 明确哪些地方是主入口残留

- [ ] 定义未来主入口位置
  - 决定主应用入口应落在哪个目录/模块
  - 明确过渡期兼容边界

---

## P1：运行时回正

- [ ] 让主应用支持 Bun 直接启动
- [ ] 减少 `cli -> studio spawn` 的中心地位
- [ ] 梳理 studio server 与 core 的直接依赖关系
- [ ] 识别哪些 `tsc -> dist` 步骤只是历史包袱

---

## P2：构建与分发回正

- [ ] 设计前端静态资源内嵌方案
- [ ] 设计单入口 server 的静态资源服务方式
- [ ] 验证 `bun compile` 的可行构建路径
- [ ] 设计启动即打开浏览器的流程
- [ ] 明确开发态与分发态的差异

---

## P3：数据层与基础设施评估

- [ ] 审查当前 SQLite 封装是否适合 Bun 主路线
- [ ] 回看历史 `better-sqlite3 + drizzle` 调研，仅保留为备忘；当前实现与文档统一按 `bun:sqlite + drizzle` 执行
- [ ] 审查配置管理与日志体系是否适合单体应用
- [ ] 审查 MCP / 工具执行层是否需要重新收口

---

## P4：产品收口

- [ ] 选择 studio 中可直接迁移的 UI 区块
- [ ] 选择需要砍掉的旧 Web IDE 交互
- [ ] 重新定义“NovelFork 最终用户启动与使用流程”
- [ ] 重新定义安装说明与快速开始

---

## 暂不优先

这些事项不是永远不做，而是当前不该排在平台回正前面：

- 主题切换
- 国际化
- 假发布与 release note 润色
- 围绕旧 CLI/旧 PWA 补大量测试
- 继续往旧 monorepo 外壳上贴功能

---

## 判断标准

一个新任务如果满足以下条件之一，应优先进入 backlog：
- 能减少旧外壳复杂度
- 能让产品更接近单入口本地应用
- 能为 `bun compile` 铺路
- 能保留核心写作能力同时减少历史包袱
