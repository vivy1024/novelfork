## 范围

Phase A 提供 Novel Bible 的最小闭环：SQLite schema、Repository、可见性规则、上下文构建、Studio REST API、最小 UI、写作管线占位注入。

Phase B 在此基础上补齐认知层核心对象：Conflict、WorldModel、Premise、CharacterArc，并把它们接入上下文注入、Studio REST API 与 Bible UI。

Phase C 补齐引导式创作闭环：Questionnaire 问卷模板与事务提交、Dynamic 模式 ratify 问卷、CoreShift 变更协议、PGI 生成前追问。

非目标：批量导入、关系图谱、ai-taste-filter 深度集成、CoreShift 自动重写已写章节、Continuity/Outline 的完整规则实现。

## 数据模型

基础表由 `packages/core/src/storage/migrations/0002_bible_v1.sql` 创建：

- `book`：记录 `bible_mode` 与 `current_chapter`
- `bible_character`：角色，支持别名、角色类型、可见性规则、软删除
- `bible_event`：事件，支持相关角色、伏笔状态、可见性规则、软删除
- `bible_setting`：设定，支持分类、nested refs、可见性规则、软删除
- `bible_chapter_summary`：章节摘要，`metadata_json` 预留 `filterReport` 等后续字段

经纬结构化表由 `0006_story_jingwei.sql` 创建，`0012_jingwei_overhaul.sql` 重构，`0013_jingwei_cooccurrence.sql` 补充共现关系，`0014_bible_to_jingwei_migration.sql` 完成 bible 数据迁移：

- `story_jingwei_section`：经纬栏目
- `story_jingwei_entry`：经纬条目

Phase B 表由 `packages/core/src/storage/migrations/0003_bible_phaseB.sql` 创建：

- `bible_conflict`：矛盾一等公民，含 scope、priority、evolution_path、resolution_state、resolution_chapter 与软删除
- `bible_world_model`：每书 1 行，经济 / 社会 / 地理 / 力量体系 / 文化 / 纪年 6 个 JSON 维度
- `bible_premise`：每书 1 行，logline、theme、tone、targetReaders、uniqueHook、genreTags
- `bible_character_arc`：角色弧线，支持同一角色多条并行弧线、转折点与可见性规则

Phase C 表由 `packages/core/src/storage/migrations/0004_bible_phaseC.sql` 创建：

- `questionnaire_template`：内置问卷模板，按 `id + version` 幂等 seed，包含 tier、targetObject、questionsJson
- `questionnaire_response`：每书问卷回答，保留原始 answersJson，支持 draft / submitted / skipped
- `core_shift`：核心设定变更记录，含 from/to snapshot、影响章节、状态与 appliedAt

TypeScript 类型集中在 `packages/core/src/bible/types.ts`，Repository 在 `packages/core/src/bible/repositories/`。

## 可见性规则

`VisibilityRule` 有三类：

- `global`：章节窗口内始终注入
- `tracked`：动态模式下，sceneText 命中名称或别名后注入
- `nested`：父条目被注入后，通过 `nestedRefs` 或 `parentIds` 展开

相关模块：

- `visibility-filter.ts`：章节时间线过滤
- `alias-matcher.ts`：名称/别名命中
- `nested-resolver.ts`：nested BFS + 环保护
- `token-budget.ts`：中文字符 × 0.6 估算 tokens，按认知层顺序与来源优先级裁剪
- `compose-context.ts`：格式化 `【类型】名称：内容`
- `format-descriptor.ts`：WorldModel JSON 维度格式化为中文自然文本
- `stalled-detector.ts`：检测 `escalating` 且超过 10 章未推进的 stalled conflict
- `build-bible-context.ts`：总入口

## 上下文构建 API

### 新默认入口：核心包 + 目录

Agent 不再默认全量读取经纬。新默认协议是：

1. `jingwei.read_brief` / `buildJingweiBrief()`：返回小型核心包、分类目录、推荐下一步读取分类。
2. `jingwei.read_category` / `readJingweiCategory()`：按分类、分页、预算、详情等级读取条目。
3. `jingwei.search` / `searchJingwei()`：按标题、别名、标签、关键词、摘要、正文搜索条目。

核心模块位于 `packages/novel-plugin/src/engine/jingwei/read-model/`：

- `category-map.ts`：标准分类与 section/tag fallback 映射。
- `entry-summary.ts`：`summaryMd` / `contentMd` 摘要 fallback 与可读条目格式。
- `build-jingwei-index.ts`：分类目录统计。
- `build-jingwei-brief.ts`：核心包构建，默认 4000 tokens 预算。
- `read-jingwei-category.ts`：分类分页读取。
- `search-jingwei.ts`：关键词搜索。
- `token-budget.ts`：预算裁剪和分页。

`story_jingwei_entry.summary_md` 由 `0019_jingwei_summary_md.sql` 添加。没有摘要的旧条目会从 `contentMd` 自动截断生成摘要。

### 兼容入口：`buildJingweiContext`

`buildJingweiContext` 仍保留，用于旧调用和上下文预览；新 agent/tool 流程应优先使用核心包 + 目录协议。数据源为 `story_jingwei_entry` 表：

```ts
import { buildJingweiContext } from "@vivy1024/novelfork-novel-plugin/engine";

const result = await buildJingweiContext({
  bookId,
  currentChapter,
  sceneText,       // 最近用户消息，用于 tracked 条目匹配
  mode: "auto",    // "auto" | "core" | "relevant" | "full"
  tokenBudget: 15000,
});
// result.items: JingweiContextItem[] — 已排序、已裁剪的条目
// result.totalTokens: number
// result.droppedEntryIds: string[] — 因预算被裁剪的条目
```

行为：
- `global` 条目始终注入（受时间窗口约束）
- `tracked` 条目仅当 sceneText 中出现别名/关键词时注入
- `nested` 条目仅当其父条目被注入时递归解析（最多 3 层）
- 按 core > relevant > reference 优先级排序
- 超出 tokenBudget 时从低优先级开始裁剪

### 调用路径

| 调用方 | 入口函数 | 说明 |
|--------|----------|------|
| 对话工具默认读取 | `jingwei.read_brief` → `buildJingweiBrief()` | 先返回核心包 + 分类目录 |
| 对话工具按需读取 | `jingwei.read_category` / `jingwei.search` | 按分类、分页、搜索补读细节 |
| 旧对话兼容入口 | `jingwei.read_context` | 兼容旧字段；`full` 不再代表无界全量读取 |
| 对话模式（叙述者会话） | `buildAgentContext()` → `buildJingweiContext()` | 自动注入到 session context，后续应迁移到预算门禁 |
| Pipeline 模式（PipelineRunner） | `withBibleContext()` → `buildJingweiContext()` | 注入到 Pipeline 上下文 |
| `pipeline.generate_chapter` 工具 | Composer → 内部调用 | 通过 Composer 组装完整上下文包 |

### 对话模式完整注入链

`buildAgentContext()` 在 `studio/src/api/lib/agent-context.ts` 中编排完整注入：

```
buildAgentContext({ bookId, sceneText, modelContextWindow })
├── buildJingweiContext() — 经纬条目（global/tracked/nested）
├── buildChapterBriefing() — 活跃角色/伏笔/硬约束
├── buildRecursiveSummaryContext() — 卷摘要 + 近 5 章
├── 大窗口模式（≥500K）：前一章全文 + full 模式
└── 预设/节拍注入（enabledPresetIds + beatTemplateId）
```

### Legacy 代码

`build-bible-context.ts` 和 `pipeline-bridge.ts` 仍存在但不再被调用。保留是为了向后兼容测试和可能的回退需求。

## Studio REST API

路由在 `packages/novel-plugin/src/routes/`（小说领域代码遵循插件化边界）。

- `GET/POST /api/books/:bookId/jingwei/sections`
- `GET/PUT/DELETE /api/books/:bookId/jingwei/sections/:sectionId`
- `GET/POST /api/books/:bookId/jingwei/entries`
- `GET/PUT/DELETE /api/books/:bookId/jingwei/entries/:entryId`
- `POST /api/books/:bookId/jingwei/templates/apply`
- `POST /api/books/:bookId/jingwei/preview-context`

Agent 工具入口：

- `jingwei.read_brief`：核心包 + 分类目录。
- `jingwei.read_category`：分类分页读取，支持 `detailLevel=summary|normal|full`。
- `jingwei.search`：关键词/别名/标签搜索。
- `jingwei.read_context`：兼容入口，保留 `items/totalTokens/droppedEntryIds`，但 `full` 只返回核心包和目录建议。

Studio 侧使用动态 import core，避免 server tests 中旧 mock 未覆盖新导出时崩溃。

## Phase C 模块说明

- 问卷 seed：`packages/core/src/bible/questionnaires/seed/` 内置 10 套模板（Tier 1 通用/玄幻/都市，Tier 2 多目标，Tier 3 WorldModel 全维）。启动时 `seedQuestionnaireTemplates()` 幂等写入。
- 问卷提交：`submitQuestionnaireResponse()` 在 SQLite transaction 内先按 mapping 写入目标 Bible 对象，再写入 response；任一 mapping 失败会整体回滚。
- Ratify 问卷：`createRatifyQuestionnaireForChapter()` 扫描章节文本中的 `[[人物:...]]` / `[[设定:...]]` / `[[矛盾:...]]` 标记，过滤已建档对象后生成临时问卷。
- CoreShift：`proposeCoreShift()` 做影响分析并创建 proposed；`acceptCoreShift()` 应用 `toSnapshot` 并给 affected 章节追加复核标记；`rejectCoreShift()` 只记录拒绝状态，不改当前 Bible。
- PGI：`generatePGIQuestions()` 当前实现两条规则：在场 escalating 矛盾、回收窗口 ±3 章内的 buried 伏笔；人设漂移与大纲偏离保留 stub 边界。

## Phase B 注意事项

- Conflict 的 active 查询由 `getActiveConflictsAtChapter(bookId, chapter)` 负责，过滤 `resolved` / `deferred`，并按 `evolution_path.first.chapter` 到 `resolution_chapter` 覆盖章节范围
- stalled 检测只标记，不自动修改：`resolutionState = escalating` 且 `currentChapter - lastEvolutionChapter > 10`
- WorldModel 维度 JSON 允许局部为空；空维度不会注入上下文
- Premise / WorldModel 是每书 1:1，使用 upsert 保持唯一行
- CharacterArc 支持同一角色多条并行，注入时跟随已注入角色
- filterReport 写入 `bible_chapter_summary.metadataJson.filterReport`

## Phase C 注意事项

- 问卷模板命名约定：`tier{N}-{genre-or-common}-{target}`，如 `tier1-common-premise`；版本当前为 `1.0.0`。
- CoreShift 事务边界：propose 只记录变更，不覆盖目标对象；accept 才应用 `toSnapshot` 并标记 affected 章节；reject 不恢复/覆盖当前对象，只保留拒绝记录。
- PGI 规则扩展点在 `packages/core/src/bible/pgi/pgi-engine.ts`；新增规则必须返回稳定 question id，并在结果中追加 `heuristicsTriggered`。
