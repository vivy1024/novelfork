## 1. 当前存储底座

storage-migration 已建立第一版本地 SQLite 底座：

- 数据库位置：`resolveRuntimeStorageDir()/novelfork.db`
- 环境变量覆盖：`NOVELFORK_SESSION_STORE_DIR`
- SQLite 驱动：`bun:sqlite`，通过 `packages/core/src/storage/db.ts` 暴露同步 `StorageSqliteDatabase` facade
- ORM：`drizzle-orm/bun-sqlite`
- 运行时边界：NovelFork 产品运行时只认 Bun，不再把 `better-sqlite3`、`node:sqlite` 或 Node runtime 作为当前落地方向
- migration 目录：`packages/core/src/storage/migrations/`
- schema 入口：`packages/core/src/storage/schema.ts`
- 启动流程：打开数据库 → 启用 WAL / `synchronous=NORMAL` → 运行 migration → 执行 JSON 一次性导入

当前基础表：

| 表 | 归属 | 用途 |
|---|---|---|
| `session` | storage-migration | 叙述者会话元数据 |
| `session_message` | storage-migration | 会话消息，按 `(session_id, seq)` 保证顺序 |
| `session_message_cursor` | storage-migration | 会话游标与可恢复边界 |
| `kv_store` | storage-migration | 迁移标记与轻量键值状态 |
| `drizzle_migrations` | storage-migration | 已应用 migration 记录 |

## 2. Schema 扩展步骤

新 spec 需要新增表时，按以下顺序执行：

1. 在 `packages/core/src/storage/` 新增 feature schema 子文件，例如：
   - `schema-bible.ts`
   - `schema-filter.ts`
   - `schema-questionnaire.ts`
2. 在子文件中使用 drizzle schema API 定义表与索引。
3. 在 `packages/core/src/storage/schema.ts` 统一 re-export：

```ts
export * from "./schema-session.js";
export * from "./schema-bible.js";
export * from "./schema-filter.js";
```

4. 为该 spec 新增 repository，放在 `packages/core/src/storage/repositories/` 或对应 feature 子目录。
5. 在 spec 的 `design.md` 中引用具体表名、依赖关系和 migration 编号。
6. 生成并检查 migration SQL。

> 现阶段 `schema.ts` 仍可直接承载 storage base 表；后续新增 Bible / Filter 表时，优先拆为子文件并由 `schema.ts` 聚合导出。

## 3. Migration 编号规划

默认编号规划如下：

| 编号 | 文件名 | 来源 spec | 内容 | 依赖 |
|---|---|---|---|---|
| `0001` | `0001_storage_base.sql` | storage-migration | session / session_message / cursor / kv_store / drizzle_migrations | — |
| `0002` | `0002_bible_v1.sql` | novel-bible-v1 Phase A | book / bible_character / bible_event / bible_setting / bible_chapter_summary | 0001 |
| `0003` | `0003_bible_phaseB.sql` | novel-bible-v1 Phase B | bible_conflict / bible_world_model / bible_premise / bible_character_arc | 0002 |
| `0004` | `0004_bible_phaseC.sql` | novel-bible-v1 Phase C | questionnaire_template / questionnaire_response / core_shift | 0003 |
| `0005` | `0005_filter_v1.sql` | ai-taste-filter-v1 | filter_report | 0001 |

协调规则：

- 编号以实际落库顺序为准，上表是默认推荐顺序。
- 新增 migration 前必须先同步最新分支并检查 `packages/core/src/storage/migrations/`，避免编号冲突。
- 每个 migration 文件只允许包含该 spec / phase 自己的表、索引和约束。
- 禁止把多个 spec 的表合并到一个 migration 中。
- 若 `ai-taste-filter-v1` 先于 Bible Phase B 完成，可调整为 `0003_filter_v1.sql`，Bible Phase B 顺延。

## 4. Multi-Phase migration 约定

当一个 spec 分多 Phase 交付时：

1. 每个 Phase 独立生成一个 migration 文件。
2. Phase N+1 可以引用 Phase N 的表，例如 `bible_character_arc.character_id → bible_character.id`。
3. 禁止 Phase N 反向依赖 Phase N+1 的表。
4. Phase A 如需给后续 Phase 注入上下文，必须预留 stub，例如 `buildBibleContext()` 先返回空数组或空片段。
5. 每个 Phase 的 repository 和测试只覆盖该 Phase 已落库的表，不假设未来表存在。

## 5. Migration 生成命令

在仓库根目录运行：

```bash
pnpm --filter @vivy1024/novelfork-core drizzle-kit generate:sqlite
```

如果需要显式指定配置：

```bash
pnpm --filter @vivy1024/novelfork-core drizzle-kit generate:sqlite --config=drizzle.config.ts
```

生成后必须检查：

- 文件名编号是否符合规划。
- SQL 是否只包含当前 spec / phase 的表。
- 外键依赖是否只指向已存在或同一 migration 内创建的表。
- 是否误删、重命名或重建已有表。

## 6. Repository 与事务边界

- 写入多表或多行时必须使用 SQLite 事务。
- Repository 只依赖 `StorageSqliteDatabase` facade 暴露的 `.prepare/.transaction/.exec/.pragma/.open/.close`，不得把 Bun 驱动细节扩散到业务层。
- 会话消息 `seq` 由 repository 在事务内生成，依赖 `(session_id, seq)` 唯一约束。
- append 同一 session 冲突时最多重试一次；二次冲突直接抛错。
- `session.message_count`、`session.updated_at` 与 `session_message_cursor` 必须在同一事务内更新。
- Repository 层统一抛 `StorageError`，携带 `op`、`sessionId` 与 `cause`。

## 7. 测试模式

存储层测试优先使用临时目录 + SQLite fixture：

```ts
const storage = createStorageDatabase({ databasePath: join(tempDir, "novelfork.db") });
runStorageMigrations(storage);
```

必须覆盖的类型：

| 类型 | 示例 |
|---|---|
| schema / migration | migration 幂等、失败不标记 applied |
| repository | create / update / list / delete / cursor |
| 协议兼容 | snapshot / history / resetRequired / replaceState |
| 并发 | N=50 append 同 session，seq 无空洞 |
| 原子性 | 事务中途失败后无半写 |
| JSON 导入 | 全新 / 部分损坏 / 已迁移 done 标记 |

运行常用验证：

```bash
pnpm --filter @vivy1024/novelfork-core exec vitest run src/__tests__/storage-db.test.ts src/__tests__/storage-repositories.test.ts src/__tests__/storage-json-import.test.ts
pnpm --filter @vivy1024/novelfork-studio test
pnpm --filter @vivy1024/novelfork-studio typecheck
```

## 8. 禁止事项

- 禁止业务代码直接执行裸 DDL。
- 禁止绕过 `schema.ts` / schema 子文件定义表。
- 禁止跨 spec 合并 migration。
- 禁止重复使用 migration 编号。
- 禁止修改已发布 migration 的语义；需要变更时新增后续 migration。
- 禁止在 migration 中引入 Bible / Filter / Template 等未归属当前 spec 的表。
- 禁止在测试中只 mock repository 而不保留至少一组真实 SQLite fixture 回归。

---

## 9. 当前关键约束速记

- 数据库：本地单进程 SQLite，不做分布式访问。
- WAL：数据库打开时启用 `PRAGMA journal_mode=WAL` 与 `synchronous=NORMAL`。
- 消息顺序：同一 session 的 `seq` 严格递增，不允许空洞。
- 导入：老 JSON 导入完成后写 `migration:json-to-sqlite:done=true`，并把源 JSON 重命名为 `.migrated-<timestamp>.bak`。
- 扩展：新 spec 新增表必须先更新 schema，再生成 migration，再写 repository 与测试。
