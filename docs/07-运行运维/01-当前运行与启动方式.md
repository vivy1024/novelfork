## 本文定位

本文记录 NovelFork 当前真实运行方式。它是运行事实源，不是最终商业分发承诺。installer、signing、auto-update、first-launch UX 等仍不属于当前已完成交付边界。

## 当前主入口

当前 NovelFork 的正统运行路径已经切到 Bun 主入口：

```text
novelfork studio
  -> packages/cli/src/commands/studio.ts
  -> 优先定位仓库根 main.ts
  -> bun run main.ts --root=<project>
  -> 启动本地 Studio 服务
  -> 最佳努力打开浏览器 http://localhost:4567
```

直接运行也成立：

```bash
bun run main.ts --root=. --port=4567
```

## 启动后发生什么

当前 server 启动流程大致如下：

```text
读取项目配置
  -> 初始化 SQLite / runtime store
  -> 执行 storage migrations
  -> 导入或迁移历史 JSON 状态
  -> 创建 Hono app
  -> 挂载 Studio API routes
  -> 准备 static provider
  -> 运行 startup recovery
  -> 注册 WebSocket routes
  -> 监听端口
```

这意味着启动过程不只是“打开网页”，还会准备存储、诊断和恢复状态。

## 当前主要端口

| 项 | 当前口径 |
|---|---|
| 默认端口 | `4567` |
| CLI / Studio 默认访问 | `http://localhost:4567` |
| 环境变量 | `NOVELFORK_STUDIO_PORT` 可控制 Studio port |
| 旧端口拆分 | 旧脚本曾区分前端 `4567` / 后端 `4569`，不再作为当前最终方案 |

## 开发运行

```bash
bun run main.ts --root=. --port=4567
```

适合：

- 开发 Studio API。
- 调试前端工作台。
- 验证文档中描述的 current 能力。
- 快速看 startup diagnostics 和日志。

## 编译运行

```bash
pnpm bun:compile
./dist/novelfork.exe --root=. --port=4567
```

`pnpm bun:compile` 当前包含：

```text
bun:build-client
  -> 构建 Studio 前端
bun:embed-assets
  -> 生成 embedded assets
bun build ./main.ts --compile
  -> 生成单文件产物
```

## Static provider 模式

启动时 static provider 可能有三类状态：

| 模式 | 含义 |
|---|---|
| embedded | 使用内嵌前端静态资源 |
| filesystem | 使用文件系统中的前端静态资源 |
| missing | 未提供前端静态资源，可能进入 API-only 或缺前端状态 |

如果浏览器页面无法加载，但 API 有响应，应优先检查 static provider 是否可用。

## Startup recovery 当前边界

当前可以对外统一表述为已闭环的范围：

- `pnpm bun:compile`。
- embedded / filesystem 双路径。
- startup recovery 最小链路。
- runtime-state bootstrap。
- index rebuild。
- 结构化 `recoveryReport`。

仍不能写成已完成的范围：

- installer。
- signing。
- auto-update。
- first-launch UX。
- 完整 repair / migration 终态。
- 长时运行资源泄漏诊断终态。

## 日志和诊断

启动日志会包含：

- project config loaded。
- storage / SQLite ready。
- static provider 状态。
- startup recovery report。
- server listen URL。
- WebSocket route registered。

排障时优先查这些信息，而不是直接假设前端或模型配置有问题。
