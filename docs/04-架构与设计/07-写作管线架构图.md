# 写作管线架构图

**版本**: v1.0
**创建日期**: 2026-06-14
**目的**: 根治"35 个 agent = 黑盒"的认知误区，画清主写作链路、对抗式审查、结构化状态机的真实运行结构。

---

## 一句话总览

**`packages/novel-plugin/src/engine/agents/` 有 35 个文件，但运行时主写作链路只编排 3 个重型 LLM agent（Writer / ContinuityAuditor / ReviserAgent）。** 其余是：①给这 3 个当零件的 prompt 构造器/输出解析器；②行内写作旁路（续写/扩写/对话/变体）；③纯函数工具（词频/AI腔/敏感词检测）；④已删前端入口的死代码。

「35 个文件 ≠ 35 个运行的 agent」。

---

## 主写作链路（pipeline.write v2）

入口：session 工具 `pipeline.write` → `handlers/pipeline-write-service.ts` 的 `executePipelineWrite()`。

```
用户「写下一章」
  │
  ├─ 0. 外置规划：scene.spec 工具先生成结构化 SceneSpec（不在本管线内）
  │
  ▼ executePipelineWrite(sceneSpec)
  1. StateManager.loadBookConfig
  2. 加载控制文档（P0-2）：author_intent.md（全书长视野）+ current_focus.md（近章焦点）
  3. 构造 ContextPackage（结构化对象，非字符串！P0-0 修复点）
       selectedContext = [author_intent, current_focus, scene.spec, jingwei, 前章尾]
     + RuleStack（hard = sceneSpec.constraints）
  4. WriterAgent.writeChapter()          ← LLM 调用①（内部三阶段）
       阶段1 creative：buildWriterSystemPrompt + buildGovernedUserPrompt
              （注入 author_intent / 词频提示 P4-1）→ 生成正文
       阶段2 settle：
         2a observer：提取本章事实
         2b settler：产出 RuntimeStateDelta（hookOps/resourceOps/knowledgeOps/timelineOp/...）
       写后纯函数校验：validatePostWrite + detectCrossChapterRepetition + analyzeAITells
  5. 长度治理（P0-1）：漂出 hard 区间 → LengthNormalizerAgent 归一化一次 → 仍漂出记 warning
  6. handleChapterAuditV2 预检（零成本硬约束：canon/POV）
  7. 审查（P1）：
       - 单 agent 模式（默认）：ContinuityAuditor.auditChapter（37 维一次跑）  ← LLM 调用②
       - 对抗模式（adversarialAudit=true）：3 视角独立审查 + 纯函数合成   ← LLM 调用②a/b/c
  8. 多轮自愈（P1-3）：evaluateGate → S1/S2 触发 ReviserAgent.reviseChapter   ← LLM 调用③（每轮）
       最多 maxReviseRounds 轮，每轮 re-audit；达上限仍有 S1/S2 → needsHumanReview
  9. 存候选稿（writing_resource，status=candidate）
```

**真实重型 agent = 3 个**：Writer（含三阶段）、ContinuityAuditor、ReviserAgent。
Planner/Composer 已外置到 scene.spec；旧管线 `pipeline.generate_chapter`（v1，已废弃隐藏）才用它们。

## 对抗式审查（P1-1，恢复多 agent 原本意图）

历史教训：多 agent 本是为「对抗式生成/审查」设计，曾被错误实现成「会话绑定不同工具」，复杂化后删除。现在恢复**正确语义**：

```
同一份正文
  ├─ 视角 A 连续性审查官（17 维：设定/时间线/数值/伏笔/信息边界/世界规则）  ← 独立 LLM 调用
  ├─ 视角 B 叙事质量审查官（12 维：节奏/弧线/爽点/台词/配角/视角）          ← 独立 LLM 调用
  └─ 视角 C 文本质量审查官（8 维：文风/AI腔/段落/套话/词汇/敏感词）        ← 独立 LLM 调用
       （三视角互不知对方结论 → 暴露单 agent 自审盲区）
  ▼
  synthesizeAdversarialAudits（纯函数合成器）
    去重 / 取更高严重度 / 视角间分歧标"需复核" / 按严重度排序
```

文件：`engine/agents/adversarial-audit.ts`（编排 + 合成器）、`severity-gate.ts`（S1-S4 门禁）。
开关：`pipeline.write` 的 `adversarialAudit`（默认 false 走单 agent 控成本）。

## 结构化状态机（P2/P3，"机制强制" 替代 "LLM 自觉"）

运行时状态是**文件快照**（`story/state/*.json`），无 SQLite 迁移，向后兼容靠 zod default。

| 状态切片 | 文件 | settler 产出 | reducer 校验 | 用途 |
|---------|------|-------------|-------------|------|
| hooks | hooks.json | hookOps | hook-health 断线/跨卷(P3-2) | 伏笔生命周期 |
| 资源账本 | resource_ledger.json | resourceOps | newBalance=old+delta，不符告警(P2-1) | 数值连续性 |
| 知识边界 | knowledge.json | knowledgeOps | findKnowledgeViolations(P2-2) | 防信息越界 |
| 时间线 | timeline.json | timelineOp | findTimelineConflicts ordinal倒流(P3-1) | 时序一致性 |
| 当前状态 | current_state.json | currentStatePatch | 章节单调 | 主角状态卡 |
| 章节摘要 | chapter_summaries.json | chapterSummary | 去重 | 节奏/前情 |

核心：`core/state/state-reducer.ts` 的 `applyRuntimeStateDelta`（统一应用所有 ops + 校验，
告警走回调非全局状态）。settler 产出 → `settler-delta-parser` 解析 → reducer 应用 → store 持久化。

## 旁路：行内写作（非整章）

入口：`routes/writing-modes.ts` HTTP 路由（前端面板调用），不走 pipeline.write：
- inline-writer（续写/扩写/桥接）、dialogue-generator（对话）、variant-generator（变体）、outline-brancher（大纲分支）
- 都是 prompt 构造器，无独立 agent 编排。

## 新人最短理解路径（读这 5 个文件）

1. `handlers/pipeline-write-service.ts` — 主链路编排全貌
2. `engine/agents/writer.ts`（writeChapter + settle）— Writer 三阶段
3. `engine/agents/adversarial-audit.ts` — 对抗审查结构
4. `core/state/state-reducer.ts` — 结构化状态机
5. `api/lib/session-tool-registry.ts`（NOVEL_CORE_TOOLS / DEPRECATED_V1_TOOLS）— 活/死工具总开关
