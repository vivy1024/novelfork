# 经纬系统架构

**版本**: v1.0.0
**状态**: ✅ 当前有效

经纬系统（Jingwei）是 NovelFork 的世界观与设定管理核心，负责将角色、事件、设定、矛盾、世界模型等结构化数据按可见性规则提供给 AI 写作上下文。系统包含数据模型层、核心包 + 分类目录读取协议、可见性过滤、关联层（共现 + 扩散 + 梦境）和 token 预算策略，确保 AI 先获得小型故事基本盘，再按分类逐步读取细节，避免全量注入导致 token 膨胀。

数据持久化使用 SQLite `story_jingwei_entry` / `story_jingwei_section` 表（不再使用 md 文件）。Agent 通过 `jingwei.upsert_entry` 工具写入条目，支持 `inferCategory` 智能分类（自动推断条目应归属的分区）。

## 上下文读取管线

### 核心包 + 分类目录协议

新默认 Agent 读取协议不再要求一次性全量注入经纬：

```
Agent task begins
│
├─[1] jingwei.read_brief(bookId, chapterNumber, sceneText, chapterIntent)
│     → 返回 coreBrief（默认 4000 tokens 内）
│     → 返回 index.categories（分类目录、条目数、估算 tokens、推荐使用场景）
│     → 返回 recommendedReads（下一步建议读取分类）
│
├─[2] jingwei.read_category(category, page, limit, tokenBudget, detailLevel)
│     → 按分类分页读取
│     → detailLevel=summary 时优先使用 summary_md / 自动摘要
│     → detailLevel=full 仍受 tokenBudget 和分页限制
│
└─[3] jingwei.search(query, categories, tokenBudget)
      → 按标题、别名、标签、关键词、摘要、正文搜索
      → 用于缺信息时精准补读
```

`jingwei.read_context` 保留为兼容层：`auto/core/relevant` 映射到核心包读取；`full` 不再返回无界全量正文，而是返回核心包、目录和分页读取建议。

### buildJingweiContext 完整过滤逻辑（兼容入口）

```
buildJingweiContext(bookId, currentChapter, sceneText, tokenBudget)
│
├─[1] 加载所有 participatesInAi=true 的条目
│     → 分区级 participatesInAi + 条目级 participatesInAi 双重检查
│
├─[2] 时间线过滤
│     filterEntriesVisibleAtChapter(allEntries, currentChapter)
│     → 排除 visibleAfterChapter > currentChapter
│     → 排除 visibleUntilChapter < currentChapter
│
├─[3] 可见性分类注入
│     ├── global 条目 → 直接注入（始终可见）
│     ├── tracked 条目 → matchTrackedByAliases(candidates, sceneText)
│     │   → 遍历条目的 title + aliases
│     │   → sceneText 中出现匹配 → 标记为 tracked source，注入
│     └── nested 条目 → resolveNestedRefs(injectedItems, allItems, maxDepth=3)
│         → 遍历已注入条目的 relatedEntryIds
│         → 查找引用的条目 → 递归解析（最多 3 层）
│         → 标记为 nested source (priority: 20)
│
├─[4] Token 预算裁剪
│     applyTokenBudget(merged, tokenBudget)
│     → sortByContextPriority() — 按 phaseOrder + sourceRank + priority 排序
│     → 超出预算时从低优先级开始裁剪
│     → 记录 droppedIds
│
└─[5] 返回 { items, totalTokens, droppedIds, mode }
```

### 对话模式注入（v1.0.7）

对话模式（叙述者会话）通过 `buildAgentContext()` 自动注入经纬上下文，替代了旧的粗暴 SQL 查询：

```
session-chat-service → buildAgentContext({ bookId, sceneText, modelContextWindow })
│
├── buildJingweiContext({ bookId, currentChapter, sceneText, mode, tokenBudget })
│   → global/tracked/nested + 时间轴 + 优先级 + token 预算控制
│   → sceneText = 最近用户消息（使 tracked 条目动态匹配）
│
├── buildChapterBriefing(bookId, currentChapter)
│   → 活跃角色 / 未解决事项 / 活跃伏笔 / 硬约束
│
├── buildRecursiveSummaryContext(bookId, currentChapter)
│   → 卷摘要 + 近 5 章摘要
│
├── 大窗口模式（modelContextWindow ≥ 500K）：
│   → tokenBudget 提升到 50,000
│   → mode 切换到 "full"（注入所有层级条目）
│   → 注入前一章全文（最多 20,000 字符）
│
└── 预设/节拍注入
    → enabledPresetIds → buildPresetInjections()
    → beatTemplateId → 模板名 + 节拍序列
```

与 Pipeline 模式的区别：
- Pipeline 模式：`PipelineRunner.withBibleContext()` → `buildJingweiContext()`（同一入口）
- 对话模式：`buildAgentContext()` → `buildJingweiContext()`（同一入口）
- 两条路径使用相同的 `buildJingweiContext` 函数，数据源统一为 `story_jingwei_entry` 表

## 可见性规则

### 三种可见性类型

| 类型 | 含义 | 注入条件 |
|------|------|----------|
| `global` | 全局可见 | 始终注入（受时间窗口约束） |
| `tracked` | 追踪可见 | 仅当 sceneText 中出现该条目的别名时注入 |
| `nested` | 嵌套可见 | 仅当其父条目被注入时，递归解析注入 |

### 时间纪律

每个条目可设置：
- `visibleAfterChapter`: 该章之后才可见（用于延迟揭示）
- `visibleUntilChapter`: 该章之后不再可见（用于过期信息）

```typescript
function isVisibleAtChapter(entry, currentChapter): boolean {
  if (visibleAfterChapter !== undefined && currentChapter < visibleAfterChapter) return false;
  if (visibleUntilChapter !== undefined && currentChapter > visibleUntilChapter) return false;
  return true;
}
```

### 别名匹配（tracked 模式）

```
matchTrackedByAliases(trackedCandidates, sceneText)
│
├── 遍历每个 tracked 条目
├── 解析 aliasesJson → string[]
├── 检查 sceneText 是否包含任一别名
└── 命中 → 标记为 tracked source，注入上下文
```

### 嵌套解析

```
resolveNestedRefs(injectedItems, allItems, { maxDepth: 3 })
│
├── 遍历已注入条目的 nestedRefsJson
├── 查找引用的条目 ID
├── 递归解析（最多 3 层）
└── 标记为 nested source (priority: 20)
```

## 关联层

### 共现矩阵 (`cooccurrence.ts`)

追踪哪些实体在同一章节中共同出现：

```
extractChapterEntities(bookId, chapterText)
  → 用经纬条目的 title + aliases 作为字典
  → 在章节文本中查找匹配
  → 返回出现的实体名列表

updateCooccurrence(bookId, chapterNumber, entities)
  → 生成所有实体对 (canonical order)
  → INSERT OR UPDATE jingwei_cooccurrence
  → count + 1, lastChapter = current
```

数据结构：
```sql
jingwei_cooccurrence (
  book_id TEXT,
  tag_a TEXT,      -- 字典序较小的实体
  tag_b TEXT,      -- 字典序较大的实体
  count INTEGER,   -- 共现次数
  last_chapter INTEGER
)
```

### 扩散路由 (`spike-routing.ts`)

基于共现图的激活扩散，发现弱关联但有用的条目：

```
propagateSpikes(bookId, seeds, { maxHops=3, decayRate=0.5, topK=10, minActivation=0.1 })
│
├── 初始化: seeds 各 activation = 1.0
├── BFS 遍历共现图:
│   for each frontier node:
│     neighbors = SELECT from jingwei_cooccurrence (limit 20)
│     for each neighbor:
│       propagated = current.activation × decayRate × (count / totalCount)
│       if propagated > minActivation → 加入 frontier
├── 排除 seeds
└── 返回 top-K { entity, activation, hops, path }
```

用途：当用户写到某个场景时，通过 seed 实体扩散找到"可能相关但未直接提及"的设定。

### 梦境系统 (`dream-system.ts`)

每卷（30 章）完成后运行的后台分析任务：

```
buildDreamPrompt(volumeSummary, cooccurrenceHighlights, unresolvedChains)
  → 构建 LLM 分析提示
  → 输入: 卷摘要 + 高频共现对 + 未解决因果链
  → 输出: DreamDiscovery[]

DreamDiscovery:
  - type: hidden_connection | forgotten_thread | potential_callback | pattern
  - entities: string[]
  - description: string
  - confidence: 0-1
  - suggestedAction?: string
```

发现类型：
- **hidden_connection**: 隐藏的实体关联
- **forgotten_thread**: 被遗忘的线索
- **potential_callback**: 潜在的回调机会
- **pattern**: 叙事模式

## Token 预算策略

### 优先级排序 (`token-budget.ts`)

```
phaseOrder (注入优先级):
  premise        → 100  (故事基线，最高优先)
  world-model    → 90   (世界模型)
  character      → 80   (角色)
  character-arc  → 75   (角色弧线)
  event/setting  → 60   (事件/设定)
  conflict       → 50   (矛盾)
  nested         → 40   (嵌套引用)
  chapter-summary → 10  (章节摘要，最低优先)

sourceRank (来源权重):
  global  → 3
  nested  → 2
  tracked → 1
```

### 裁剪算法

```
applyTokenBudget(items, tokenBudget = 30000)
│
├── sortByContextPriority() — 按 phaseOrder → sourceRank → priority → updatedAt 排序
├── 计算 totalTokens
├── while totalTokens > tokenBudget:
│   ├── sortByDropPriority() — 反向排序（低优先级先裁剪）
│   ├── 从最低优先级开始移除
│   └── 记录 droppedIds
└── 返回 { items, totalTokens, droppedIds }
```

### Token 估算

```typescript
function estimateTokens(text: string): number {
  return Math.ceil(text.length * 0.6); // 中文约 0.6 token/字符
}
```

默认预算：
- `composeJingweiContext` 默认 8000 tokens
- `applyTokenBudget` 默认 30000 tokens
- 章节摘要滑动窗口：最近 15 章
- 卷级摘要：每 30 章压缩一次

## 竞品对比

| 维度 | NovelFork 经纬 | NovelCrafter Codex | Sudowrite | VCP Wave Memory |
|------|---------------|-------------------|-----------|-----------------|

| **关联发现** | 共现矩阵 + 扩散路由 + 梦境系统 | 无 | 无 | 向量相似度 |
| **Token 管理** | 优先级排序 + 预算裁剪 + 滑动窗口 | 手动控制 | 无限制（依赖大窗口） | 自动截断 |
| **递归摘要** | 卷级摘要（30 章压缩）+ 最近 5 章 | 无 | 无 | 层级摘要 |
| **核心移位** | 变更追踪 + 影响分析 + 审批流 | 无 | 无 | 无 |
| **问卷引导** | 分层问卷（tier 1-3）+ AI 建议 | 无 | 简单引导 | 无 |
| **PGI** | 生成前追问（基于经纬缺口分析） | 无 | 无 | 无 |

### NovelFork 经纬的独特优势

1. **动态可见性**：不是简单的全量注入，而是根据当前场景文本动态决定注入哪些条目
2. **嵌套引用解析**：设定 A 引用设定 B，当 A 被注入时自动拉入 B（最多 3 层）
3. **关联层**：通过共现统计和扩散算法发现隐含关联，而非仅依赖显式引用
4. **梦境系统**：定期后台分析，发现被遗忘的线索和潜在回调机会
5. **核心移位追踪**：当世界观发生重大变更时，自动分析影响范围并提示作者确认

## 自动链接引擎 (`auto-linker.ts`)

章节保存时自动扫描文本，建立条目与章节的关联关系。

### 实现逻辑

```typescript
// auto-linker.ts
export async function autoLinkChapter(bookId: string, chapterNumber: number, chapterText: string) {
  // 1. 加载所有经纬条目的 title + aliases
  const entries = await loadAllEntries(bookId);
  const dictionary = buildMatchDictionary(entries); // Map<pattern, entryId>

  // 2. 扫描章节文本，查找匹配
  const matchedEntryIds = new Set<string>();
  for (const [pattern, entryId] of dictionary) {
    if (chapterText.includes(pattern)) {
      matchedEntryIds.add(entryId);
    }
  }

  // 3. 更新匹配条目的 relatedChapterNumbers
  for (const entryId of matchedEntryIds) {
    await appendRelatedChapter(bookId, entryId, chapterNumber);
  }
}
```

### 触发时机

- 章节保存（`PUT /api/books/:bookId/chapters/:number`）后自动触发
- 候选稿接受（accept → merge/replace）后对目标章节触发
- 异步执行，不阻塞保存操作

### 匹配字典

```
buildMatchDictionary(entries):
  for each entry:
    add entry.title → entry.id
    for each alias in entry.aliases:
      add alias → entry.id
  return Map<string, string>
```

## ToolConfigBar → Session toolPolicy 接通

ToolConfigBar 组件允许用户在会话中控制 Agent 可使用的工具集。

### 接通机制

```
ToolConfigBar UI
  → 用户切换工具开关（如禁用 candidate.create_chapter）
  → 更新 session.toolPolicy.deny[]
  → 下次 Agent 调用时，runtime 检查 toolPolicy
  → deny 列表中的工具被拒绝执行，返回 tool_denied 错误
```

### 数据流

```typescript
interface ToolPolicy {
  deny: string[];  // 被禁用的工具名列表
}

// ToolConfigBar 更新 session
function onToolToggle(toolName: string, enabled: boolean) {
  if (!enabled) {
    session.toolPolicy.deny.push(toolName);
  } else {
    session.toolPolicy.deny = session.toolPolicy.deny.filter(t => t !== toolName);
  }
  updateSession(session);
}
```

### 用途

- 临时禁止 Agent 创建候选稿（只做分析不写作）
- 禁止特定工具避免意外操作
- 会话级别生效，不影响其他会话

## 图谱节点可见性图标

经纬条目在资源树和编辑器中通过图标标识可见性类型：

| 可见性类型 | 图标 | 含义 |
|-----------|------|------|
| `global` | 🌐 | 全局可见，始终注入 AI 上下文 |
| `tracked` | 👁 | 追踪可见，场景文本匹配时注入 |
| `nested` | 🔗 | 嵌套可见，被关联条目引用时级联注入 |

### 显示位置

- 资源树中条目节点的前缀图标
- JingweiEntryEditor 标题栏的可见性标识
- 上下文预览面板中每个注入条目的来源标记
