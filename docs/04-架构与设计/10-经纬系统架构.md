# 经纬系统架构

经纬系统（Jingwei）是 NovelFork 的世界观与设定管理核心，负责将角色、事件、设定、矛盾、世界模型等结构化数据按可见性规则注入 AI 写作上下文。系统包含数据模型层、上下文注入管线、可见性过滤、关联层（共现 + 扩散 + 梦境）和 token 预算策略，确保 AI 在写作时获得精确且不超限的世界观信息。

## 上下文注入管线

### 完整流程

```
buildJingweiLegacyContext(bookId, currentChapter, sceneText, tokenBudget)
│
├─[1] 加载所有候选条目
│     loadAllCandidateEntries()
│     ├── characters → characterToItem()
│     ├── events → eventToItem()
│     ├── settings → settingToItem()
│     └── chapterSummaries → chapterSummaryToItem()
│         (滑动窗口: 最近 15 章)
│
├─[2] 时间线过滤
│     filterEntriesVisibleAtChapter(allEntries, currentChapter)
│     → 排除 visibleAfterChapter > current
│     → 排除 visibleUntilChapter < current
│
├─[3] 注入锚点条目
│     ├── injectPremise() → 故事基线 (priority: 100)
│     ├── injectWorldModel() → 世界模型各维度 (priority: 90)
│     └── injectConflicts() → 活跃矛盾 (priority: 70)
│
├─[4] 模式分支
│     ├── mode === "static"
│     │   → 只取 global 条目 + 角色弧线
│     │   → composeJingweiContext()
│     │
│     └── mode === "dynamic"
│         ├── 无 sceneText → 同 static
│         └── 有 sceneText
│             ├── globals = filter(type === "global")
│             ├── tracked = matchTrackedByAliases(candidates, sceneText)
│             ├── nested = resolveNestedRefs([globals, tracked], all, maxDepth=3)
│             ├── arcs = injectCharacterArcs(characterIds)
│             └── merged = dedupeByBestSource([...all])
│
├─[5] 格式化 + 预算裁剪
│     composeJingweiContext(merged, { mode, tokenBudget })
│     ├── sortByContextPriority() — 按 phaseOrder + source + priority 排序
│     ├── formatJingweiContextItem() — 添加类型标签前缀
│     └── applyTokenBudget() — 从低优先级开始裁剪
│
└─[6] 返回结果
      { items, totalTokens, droppedIds, mode }
```

### 章节 Briefing

```
buildChapterBriefing(bookId, chapterNumber)
│
├── 活跃角色 (lifecycle='active', limit 10)
│   → 名称 + 境界 + 目标
│
├── 未解决事项 (urgency='high'|'overdue', limit 5)
│   → 触发事件 + 已多少章未推进
│
├── 活跃伏笔 (category='foreshadowing', lifecycle='active', limit 5)
│   → 标题 + 状态
│
└── 硬约束 (category in worldview/special/power-system, global visibility)
    → 标题 + 内容前 80 字
```

```
buildRecursiveSummaryContext(bookId, currentChapter)
│
├── 卷级摘要 (每 30 章一卷)
│   → 当前卷 + 上一卷的压缩摘要
│
└── 最近 5 章摘要
    → 按章节号正序排列
```

## 可见性规则

### 三种可见性类型

| 类型 | 含义 | 注入条件 |
|------|------|----------|
| `global` | 全局可见 | 始终注入（受时间窗口约束） |
| `tracked` | 追踪可见 | 仅当 sceneText 中出现该条目的别名时注入 |
| `nested` | 嵌套可见 | 仅当其父条目被注入时，递归解析注入 |

### 时间纪律

每个条目可设置：
- `visibleAfterChapter`: 该章之后才可见（用于延迟揭示）
- `visibleUntilChapter`: 该章之后不再可见（用于过期信息）

```typescript
function isVisibleAtChapter(entry, currentChapter): boolean {
  if (visibleAfterChapter !== undefined && currentChapter < visibleAfterChapter) return false;
  if (visibleUntilChapter !== undefined && currentChapter > visibleUntilChapter) return false;
  return true;
}
```

### 别名匹配（tracked 模式）

```
matchTrackedByAliases(trackedCandidates, sceneText)
│
├── 遍历每个 tracked 条目
├── 解析 aliasesJson → string[]
├── 检查 sceneText 是否包含任一别名
└── 命中 → 标记为 tracked source，注入上下文
```

### 嵌套解析

```
resolveNestedRefs(injectedItems, allItems, { maxDepth: 3 })
│
├── 遍历已注入条目的 nestedRefsJson
├── 查找引用的条目 ID
├── 递归解析（最多 3 层）
└── 标记为 nested source (priority: 20)
```

## 关联层

### 共现矩阵 (`cooccurrence.ts`)

追踪哪些实体在同一章节中共同出现：

```
extractChapterEntities(bookId, chapterText)
  → 用经纬条目的 title + aliases 作为字典
  → 在章节文本中查找匹配
  → 返回出现的实体名列表

updateCooccurrence(bookId, chapterNumber, entities)
  → 生成所有实体对 (canonical order)
  → INSERT OR UPDATE jingwei_cooccurrence
  → count + 1, lastChapter = current
```

数据结构：
```sql
jingwei_cooccurrence (
  book_id TEXT,
  tag_a TEXT,      -- 字典序较小的实体
  tag_b TEXT,      -- 字典序较大的实体
  count INTEGER,   -- 共现次数
  last_chapter INTEGER
)
```

### 扩散路由 (`spike-routing.ts`)

基于共现图的激活扩散，发现弱关联但有用的条目：

```
propagateSpikes(bookId, seeds, { maxHops=3, decayRate=0.5, topK=10, minActivation=0.1 })
│
├── 初始化: seeds 各 activation = 1.0
├── BFS 遍历共现图:
│   for each frontier node:
│     neighbors = SELECT from jingwei_cooccurrence (limit 20)
│     for each neighbor:
│       propagated = current.activation × decayRate × (count / totalCount)
│       if propagated > minActivation → 加入 frontier
├── 排除 seeds
└── 返回 top-K { entity, activation, hops, path }
```

用途：当用户写到某个场景时，通过 seed 实体扩散找到"可能相关但未直接提及"的设定。

### 梦境系统 (`dream-system.ts`)

每卷（30 章）完成后运行的后台分析任务：

```
buildDreamPrompt(volumeSummary, cooccurrenceHighlights, unresolvedChains)
  → 构建 LLM 分析提示
  → 输入: 卷摘要 + 高频共现对 + 未解决因果链
  → 输出: DreamDiscovery[]

DreamDiscovery:
  - type: hidden_connection | forgotten_thread | potential_callback | pattern
  - entities: string[]
  - description: string
  - confidence: 0-1
  - suggestedAction?: string
```

发现类型：
- **hidden_connection**: 隐藏的实体关联
- **forgotten_thread**: 被遗忘的线索
- **potential_callback**: 潜在的回调机会
- **pattern**: 叙事模式

## Token 预算策略

### 优先级排序 (`token-budget.ts`)

```
phaseOrder (注入优先级):
  premise        → 100  (故事基线，最高优先)
  world-model    → 90   (世界模型)
  character      → 80   (角色)
  character-arc  → 75   (角色弧线)
  event/setting  → 60   (事件/设定)
  conflict       → 50   (矛盾)
  nested         → 40   (嵌套引用)
  chapter-summary → 10  (章节摘要，最低优先)

sourceRank (来源权重):
  global  → 3
  nested  → 2
  tracked → 1
```

### 裁剪算法

```
applyTokenBudget(items, tokenBudget = 30000)
│
├── sortByContextPriority() — 按 phaseOrder → sourceRank → priority → updatedAt 排序
├── 计算 totalTokens
├── while totalTokens > tokenBudget:
│   ├── sortByDropPriority() — 反向排序（低优先级先裁剪）
│   ├── 从最低优先级开始移除
│   └── 记录 droppedIds
└── 返回 { items, totalTokens, droppedIds }
```

### Token 估算

```typescript
function estimateTokens(text: string): number {
  return Math.ceil(text.length * 0.6); // 中文约 0.6 token/字符
}
```

默认预算：
- `composeJingweiContext` 默认 8000 tokens
- `applyTokenBudget` 默认 30000 tokens
- 章节摘要滑动窗口：最近 15 章
- 卷级摘要：每 30 章压缩一次

## 竞品对比

| 维度 | NovelFork 经纬 | NovelCrafter Codex | Sudowrite | VCP Wave Memory |
|------|---------------|-------------------|-----------|-----------------|
| **数据模型** | 结构化表（角色/事件/设定/矛盾/世界模型） | 自由文本条目 + 标签 | 自由文本 "Story Bible" | 结构化记忆块 |
| **可见性** | 三级规则 (global/tracked/nested) + 时间窗口 | 手动开关 | 无（全量注入） | 自动衰减 |
| **上下文注入** | 动态模式：别名匹配 + 嵌套解析 + 预算裁剪 | 关键词匹配 | 全量注入 | 相关性检索 |
| **关联发现** | 共现矩阵 + 扩散路由 + 梦境系统 | 无 | 无 | 向量相似度 |
| **Token 管理** | 优先级排序 + 预算裁剪 + 滑动窗口 | 手动控制 | 无限制（依赖大窗口） | 自动截断 |
| **递归摘要** | 卷级摘要（30 章压缩）+ 最近 5 章 | 无 | 无 | 层级摘要 |
| **核心移位** | 变更追踪 + 影响分析 + 审批流 | 无 | 无 | 无 |
| **问卷引导** | 分层问卷（tier 1-3）+ AI 建议 | 无 | 简单引导 | 无 |
| **PGI** | 生成前追问（基于经纬缺口分析） | 无 | 无 | 无 |

### NovelFork 经纬的独特优势

1. **动态可见性**：不是简单的全量注入，而是根据当前场景文本动态决定注入哪些条目
2. **嵌套引用解析**：设定 A 引用设定 B，当 A 被注入时自动拉入 B（最多 3 层）
3. **关联层**：通过共现统计和扩散算法发现隐含关联，而非仅依赖显式引用
4. **梦境系统**：定期后台分析，发现被遗忘的线索和潜在回调机会
5. **核心移位追踪**：当世界观发生重大变更时，自动分析影响范围并提示作者确认

## PGI 引擎 (`pgi-engine.ts`)

Pre-Generation Inquiry — 写作前追问机制：

```
PGI 流程:
1. 分析当前经纬数据的缺口
2. 生成针对性追问（如：角色动机不明确、场景地理未设定）
3. 用户回答后更新经纬条目
4. 确保 AI 写作时有足够的世界观支撑
```

## 问卷系统 (`questionnaires/`)

分层引导式世界观构建：

| 层级 | 目标对象 | 问题类型 | 用途 |
|------|---------|---------|------|
| Tier 1 | premise | 基础问题 | 故事基线（logline/基调/读者） |
| Tier 2 | character, conflict | 中级问题 | 角色设定 + 核心矛盾 |
| Tier 3 | world-model, setting | 深度问题 | 世界观细节 |

支持：
- `ai-suggest`: AI 辅助回答建议
- `apply-mapping`: 答案自动映射到经纬字段
- `ratify-questionnaire`: 问卷批准流程
- `template-validator`: 模板校验

## 核心移位 (`core-shift/`)

当世界观发生重大变更时的追踪与影响分析：

```
CoreShiftService:
  - 创建移位提案 (proposed)
  - 影响分析 (impactAnalysis)
    → 哪些章节受影响
    → 哪些角色弧线需要调整
    → 哪些矛盾需要重新评估
  - 作者审批 (accepted/rejected)
  - 应用变更 (applied)
```
