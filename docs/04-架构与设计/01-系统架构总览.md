## Monorepo 结构

```
novelfork/
├── packages/core/            # 通用基础设施（storage/llm/state/hooks/mcp/runtime）
├── packages/studio/          # Web 工作台（Hono API + React SPA）
├── packages/cli/             # CLI 工具（novelfork 命令）
├── packages/novel-plugin/    # 小说领域插件（engine/routes/handlers/pages）
├── packages/fitness-plugin/  # 健身领域插件骨架（证明可扩展性）
├── docs/                     # 项目文档
├── .kiro/                    # Kiro steering 配置
└── package.json              # 根 workspace 配置
```

五个包通过 pnpm workspace 管理。`core` 是通用基础设施层，`studio` 和 `cli` 依赖 `core`，`novel-plugin` 和 `fitness-plugin` 作为领域插件依赖 `core` 并被 `studio` 加载。

## v1.1.0 架构升级

### 约束驱动写作系统

v1.1.0 引入 Harness Engineering 方法论，将写作系统从"prompt 驱动"升级为"约束驱动"：

- **硬约束体系**：H1(Token 天花板) / H2(Canon 不可变) / H3(候选区边界) / H4(Scene Spec 完整性) / H5(用户确认门) / H6(经纬工具写入) / H7(POV 检测)
- **工具精简**：从 48 个工具精简到 10 个核心工具，模型认知负担降低 60%
- **管线精简**：从 5 次 LLM 调用降到 2 次（Writer + AuditRevise）
- **经纬三层分离**：Canon（不可变）/ Dynamic（每章更新）/ Reference（按需查阅）
- **Token 预算前置**：ContextBudgetManager 在 prompt 构造前强制预算，工具输出按类型截断
- **Provider 健康管理**：ProviderHealthManager 实现熔断、错误分类、fallback 建议

## packages/core 职责

通用基础设施层，不依赖任何 HTTP 框架或 UI 框架，纯 TypeScript 逻辑。提供存储、LLM、状态管理、hooks、MCP、运行时等通用能力。

| 模块 | 职责 |
|---|---|
| `storage/` | 存储层：SQLite schema、15 个 migration、repositories（kv/session/message/request_log） |
| `llm/` | LLM 客户端：createLLMClient、chatCompletion、chatWithTools、Prompt Cache |
| `state/` | 状态管理：StateManager、MemoryDB、projections |
| `hooks/` | 生命周期钩子系统 |
| `mcp/` | MCP 协议客户端 |
| `runtime/` | 进程适配器、沙箱执行（none/basic/strict） |
| `models/` | 数据模型：BookConfig、ChapterMeta、ProjectConfig、RuntimeState 等 |
| `registry/` | 工具注册表 + 命令注册表 + ToolSearch |
| `plugins/` | 插件系统 |
| `notify/` | 通知分发（飞书/企微/Telegram/Webhook） |
| `relay/` | 本地中继 |
| `utils/` | 工具函数集 |

## 部署模型

```
┌────────────────────────────────────────┐
│  novelfork-v1.0.0-windows-x64.exe      │
│                                        │
│  Bun compile 单文件产物                 │
│  内嵌：Hono API + Vite 构建的 SPA      │
│  运行时：本地 SQLite + 本地文件系统      │
│  数据库：~/.novelfork/novelfork.db      │
│  端口：localhost:3000（默认）            │
└────────────────────────────────────────┘
```

- 本地单机运行，无需外部数据库或容器
- exe 启动后同时提供 API 服务和静态前端
- LLM 调用通过配置的 API endpoint 外发（支持 Sub2API 网关）
- 数据存储：SQLite 主数据库（~/.novelfork/novelfork.db，15 个 migration）+ 文件系统（真相文件、章节 markdown）
- Agent 运行时对标 Claude Code：CLAUDE.md 读取、LLM 压缩摘要、Subagent、后台任务、Prompt Cache、ToolSearch、Skills、MCP、沙箱、Browser、Terminal

## 设计决策

1. **core 与 novel-plugin 分离**：core 是通用基础设施（存储/LLM/状态/hooks/MCP/runtime），novel-plugin 包含所有小说领域代码（引擎/路由/服务/页面）。这样 core 可被其他领域插件复用（如 fitness-plugin），而小说功能完全内聚在 novel-plugin 中。

2. **backend-contract 层**：前端不直接拼 URL 或处理 HTTP 细节，而是通过 ContractClient + 领域客户端统一管理。每个请求携带 capability 元数据，前端可据此决定 UI 状态（启用/禁用/只读/预览）。

3. **SQLite 而非远程数据库**：本地单机场景，SQLite 足够且零运维。通过 Drizzle ORM 管理 schema 和 migration。

4. **Bun compile**：将整个应用编译为单个 exe，用户下载即用，无需安装 Node.js 或 Bun 运行时。
