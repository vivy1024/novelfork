## Monorepo 结构

```
novelfork/
├── packages/core/       # 核心写作引擎
├── packages/studio/     # Web 工作台（Hono API + React SPA）
├── packages/cli/        # CLI 工具（novelfork 命令）
├── docs/                # 项目文档
├── .kiro/               # Kiro steering 配置
└── package.json         # 根 workspace 配置
```

三个包通过 pnpm workspace 管理，`core` 是纯逻辑层，`studio` 和 `cli` 都依赖 `core`。

## 技术栈

| 层 | 技术 |
|---|---|
| 运行时 | Bun |
| 语言 | TypeScript（strict） |
| 前端框架 | React 19 |
| 后端框架 | Hono |
| 数据库 | SQLite（bun:sqlite） |
| 构建 | Vite（前端）+ Bun compile（exe） |
| 包管理 | pnpm workspace |

## packages/core 职责

核心引擎，不依赖任何 HTTP 框架或 UI 框架，纯 TypeScript 逻辑。

| 模块 | 职责 |
|---|---|
| `pipeline/runner.ts` — PipelineRunner | 写作管线主引擎：writeNextChapter、writeDraft、reviseDraft、resyncChapterArtifacts、generateStyleGuide、importChapters、importCanon、runRadar |
| `agents/continuity.ts` — ContinuityAuditor | 连续性审计：读取章节+真相文件+LLM 分析，检测剧情矛盾 |
| `agents/ai-tells.ts` — analyzeAITells | AI 味检测：纯规则分析（段落长度均匀度、填充词密度、公式化转折、列表结构） |
| `agents/style-analyzer.ts` — analyzeStyle | 文风分析 |
| `utils/chapter-splitter.ts` — splitChapters | 章节拆分 |
| `llm/provider.ts` | LLM 客户端：createLLMClient、chatCompletion、chatWithTools |
| `storage/` | 存储层：SQLite schema、migrations、repositories（kv/session/message） |
| `state/manager.ts` — StateManager | 状态管理：loadBookConfig、loadChapterIndex、bookDir 等 |
| `models/` | 数据模型：BookConfig、ChapterMeta、ProjectConfig、RuntimeState 等 |
| `bible/` | 设定百科：角色/事件/设定/世界模型 repositories + 上下文构建 |
| `jingwei/` | 精卫（真相文件管理）：条目/分区 repositories + 模板 |
| `presets/` | 预设系统：题材包、节拍模板、逻辑风险规则、去 AI 味规则 |
| `filter/` | 敏感词过滤：AC 自动机 + 朱雀客户端 + 规则引擎 |
| `compliance/` | 合规检查：AI 比例估算、格式检查、发布就绪度 |

## packages/studio 职责

Web 工作台，同时包含 Hono 后端和 React 前端。

| 层 | 路径 | 职责 |
|---|---|---|
| API 服务器 | `src/api/` | Hono 路由 + 服务层，对外暴露 REST API 和 WebSocket |
| API 路由 | `src/api/routes/` | 按领域拆分：session、pipeline、providers、books、writing-modes 等 |
| API 服务 | `src/api/lib/` | 业务逻辑：cockpit-service、session-chat-service、provider-manager 等 |
| 前端入口 | `src/app-next/` | React SPA：shell、backend-contract、writing-workbench、agent-conversation |
| 前端组件 | `src/components/` | UI 组件：ToolCall、Routines、Bible、jingwei 等 |
| 共享类型 | `src/shared/` | 前后端共享的类型定义和契约 |

## packages/cli 职责

命令行工具，提供 `novelfork` 命令。

| 命令 | 功能 |
|---|---|
| `write` | 写下一章 |
| `draft` | 生成草稿 |
| `audit` | 连续性审计 |
| `detect` | AI 味检测 |
| `revise` | 修订章节 |
| `chat` | headless 对话 |
| `studio` | 启动 Web 工作台 |
| `init` | 初始化项目 |
| `import` | 导入章节 |
| `status` | 查看项目状态 |
| `radar` | 市场雷达 |

CLI 直接调用 `core` 的 PipelineRunner，不经过 HTTP 层。

## 部署模型

```
┌────────────────────────────────────────┐
│  novelfork-v0.7.1-windows-x64.exe      │
│                                        │
│  Bun compile 单文件产物                 │
│  内嵌：Hono API + Vite 构建的 SPA      │
│  运行时：本地 SQLite + 本地文件系统      │
│  端口：localhost:3000（默认）            │
└────────────────────────────────────────┘
```

- 本地单机运行，无需外部数据库或容器
- exe 启动后同时提供 API 服务和静态前端
- LLM 调用通过配置的 API endpoint 外发（支持 Sub2API 网关）
- 数据全部存储在本地：SQLite 数据库 + 文件系统（真相文件、章节 markdown）

## 设计决策

1. **core 与 studio 分离**：core 是纯逻辑，可被 CLI 直接调用，也可被 studio API 层调用。这样 CLI 和 Web 工作台共享同一套写作引擎。

2. **backend-contract 层**：前端不直接拼 URL 或处理 HTTP 细节，而是通过 ContractClient + 领域客户端统一管理。每个请求携带 capability 元数据，前端可据此决定 UI 状态（启用/禁用/只读/预览）。

3. **SQLite 而非远程数据库**：本地单机场景，SQLite 足够且零运维。通过 Drizzle ORM 管理 schema 和 migration。

4. **Bun compile**：将整个应用编译为单个 exe，用户下载即用，无需安装 Node.js 或 Bun 运行时。
