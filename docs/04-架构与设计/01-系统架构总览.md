# NovelFork 系统架构总览

**版本**: v1.0.0
**创建日期**: 2026-04-28
**更新日期**: 2026-04-28
**状态**: ✅ 当前有效
**文档类型**: current

---

## 先看结论

NovelFork 当前不是一个“坏项目”，而是一个：

- **核心写作能力已经很多**
- **平台外壳方向走错了**

因此架构判断必须分成两层：

1. **当前可复用资产**
2. **当前需要回正的平台残留**

---

## 一、当前可复用资产

### 1. 核心写作引擎：`packages/core`
这是当前仓库最值得保留的部分。

已实现的主要能力包括：
- 多 Agent 写作管线
- 章节规划 / 编排 / 写作 / 审计 / 修订
- 运行时状态管理
- Lorebook / memory / hook / summary 等上下文机制
- AI 痕迹检测、敏感词检测、文风分析
- 章节导入、重写、恢复、长度治理

这部分代码代表的是 **NovelFork 的产品能力**，不是偏航点。

### 2. Web 工作台中的可复用 UI 与 API：`packages/studio`
虽然 studio 的平台定位被旧文档写歪了，但其中很多能力仍可复用：
- React 前端组件
- 多面板工作区
- 配置页与管理页的部分实现
- Hono API 组织方式
- 局部的工具执行与可视化能力

可复用不等于当前结构正确。可复用的意思是：
- 这些能力可以迁移到新的单入口应用中
- 不需要全盘重写

### 3. 题材与规则资产
`packages/core/genres/` 以及相关规则读取、书籍约束、审计逻辑，属于长期可复用资产。

### 4. 文档中保留的历史证据
`docs/archive/` 里虽然很多内容已经过时，但它们仍然有一个价值：
- 说明项目曾经怎么判断平台问题
- 说明为什么后来会偏航

---

## 二、当前平台残留

真正需要回正的，不是写作能力，而是下面这些外壳层。

### 1. 多入口结构
当前入口分散在：
- `packages/cli`
- `packages/studio`
- `packages/desktop`

这更像一个开发中的 monorepo，而不像一个完整产品。

### 2. `cli -> studio` 启动链路
当前存在明显的“CLI 去拉起 Web 工作台”的结构。

这会导致：
- 启动路径绕
- 用户体验更像开发工具
- 最终分发困难

### 3. `tsc -> dist` 驱动的运行模式
当前 build 成功，但这套运行方式更适合源码项目，不适合最终产品形态。

### 4. 旧的 PWA/发布叙事
旧文档把平台目标叙述成：
- PWA
- 可安装 Web 工作台
- v0.0.1 发布准备

这套叙事已经失效。

---

## 三、当前事实状态

基于当前代码与验证结果，可做如下判断。

### 构建状态
- `packages/core` 可编译
- `packages/cli` 可编译
- `packages/studio` client/server 可编译

### 测试状态
- core 大部分测试通过，但仍有失败
- cli/studio 有明显测试与基础设施问题
- 失败更多集中在：
  - 测试环境配置
  - mock 不完整
  - studio runtime / 路径解析

### 平台状态
当前仓库更像：
- 一套有较强写作能力的 monorepo 工程
- 而不是一个交付完成的本地单体应用

所以“当前状态”应该被描述为：

> **核心能力可用，平台外壳未回正。**

而不是“生产就绪”。

## 四、当前架构（已实现）

### 实际结构

```text
novelfork (monorepo)
  ├─ packages/cli         — CLI 入口（27 个子命令）
  ├─ packages/studio      — Web 工作台 + API 服务
  │   ├─ src/api/         — Hono REST + WebSocket
  │   ├─ src/app-next/    — React 19 前端（工作台/设置/套路/仪表盘）
  │   └─ src/stores/      — 全局状态 (appStore)
  ├─ packages/core        — 写作引擎
  │   ├── agents/         — 13 个 Agent 类
  │   ├── pipeline/       — Agent 循环 + 编排 + system prompt
  │   └── registry/       — 工具注册表（18 内置 + MCP）
  └─ docs/                — 文档中心
```

### 启动模型（已实现）

```text
源码开发 → bun run dev → http://localhost:4567
产品分发 → bun run compile → dist/novelfork.exe (115MB)
  → ./novelfork.exe → API :4567 (API-only 模式)
```

### 分发模型（已实现）

```
bun compile → 单可执行文件 → 拷贝到任意目录 → 运行
PWA 安装 → 浏览器「添加到主屏幕」→ 离线可用
公网访问 → cloudflared tunnel → 任何设备浏览器打开
```

## 五、从当前到目标的差距

### 差距 1：入口仍然分裂
需要把当前多入口逐步收敛为一个主入口。

### 差距 2：运行模式仍围绕源码与构建产物
需要从“先 build，再由不同包各自运行”转向“单入口应用驱动”。

### 差距 3：分发方案没有定型
需要把 `bun compile` 确立为主分发路线。

### 差距 4：文档与实现长期不一致
这次 docs 回正就是先解决这个问题，让后续改造有统一依据。

---

## 六、当前阶段的架构原则

在代码真正迁移前，所有架构判断遵循：

1. **保留核心写作引擎**
2. **允许重写错误的产品外壳**
3. **先统一单体应用目标，再做具体代码迁移**
4. **不再把旧的 PWA/发布流程当成目标态**

---

## 七、目录角色的重新理解

### 当前仍保留
- `packages/core`：产品核心资产
- `packages/studio`：可迁移 UI/API 资产
- `packages/cli`：过渡期命令能力
- `packages/desktop`：历史残留，暂不作为主路线

### 未来倾向
- `core` 能力下沉为主应用内部模块
- `studio` 成为前端界面层
- `cli` 可能并入主入口或只保留调试用途
- `desktop` 是否保留，取决于单 exe 路线是否完全覆盖需求

## 八、窗口 / 会话 / 快照三者关系（Package 4 / 6 收口口径）

NovelFork 的前端把"用户看到的内容"拆成三层，彼此解耦：

- **窗口 (`chatWindow` in `useWindowStore`)**：只是"打开/最小化/位置"的视图壳。`persist` 仅保留布局与 `sessionId` 关联，不承载任何消息内容。
- **会话 (`NarratorSessionRecord`)**：后端是消息链、模式、参数的唯一事实源。关闭窗口不销毁会话，关掉最后一个窗口后会话仍在 `SessionCenter` 卡片里可重新打开。
- **快照 (`NarratorSessionChatSnapshot` + `windowRuntimeStore`)**：后端广播的增量/回放快照驱动前端渲染，`recoveryState` 走 `idle / recovering / reconnecting / replaying / resetting` 五态，由 `windowRecoveryPresentation.ts` 提供统一的徽标/文案展示。

推论：**窗口是视图、会话是对象、快照是协议**。任何 UI 层新的"打开/关闭/恢复"逻辑都应该基于这三层心智，不要在窗口持久化里偷藏消息，也不要绕开快照直写会话记录。

## 九、Project / Book / Chapter / Narrator 对象模型最小闭环（Package 5 收口口径）

closure spec 的 Package 5 只负责**明确当前事实源**，不负责把长期愿景一次实现。当前事实源如下：

| 对象 | 现状 | 事实源 | 已 server-first | 最小闭环 |
|---|---|---|---|---|
| **Project** | 单例配置（`novelfork.json` 的 `name / language / llm`） | `storage/adapter.ts::ProjectConfig` + `api/routes/storage.ts GET/PUT /api/project` | ✅ | ✅ 已形成 |
| **Book** | 多实例，代表一本小说 | `BookSummary` / `BookDetail`（`shared/contracts.ts`） + `/api/books/*` | ✅ | ✅ 已形成 |
| **Chapter** | 每本 Book 下的多章节，含 audit / detection / length telemetry | `ChapterSummary` / `ChapterDetail` + `/api/books/:id/chapters/*` | ✅ | ✅ 已形成 |
| **Narrator 会话** | 跨 Book 复用的 chat 会话，Package 4 已 server-first 收敛 | `NarratorSessionRecord` + `SessionChatSnapshot` + `/api/sessions/*` + WS `/api/sessions/:id/chat` | ✅ | ✅ 已形成 |

**关系**：

```
Project (singleton)
  ├─ llm / language / name 全局配置
  └─ Books (many)
        ├─ book.json（元数据、language、platform、genre）
        ├─ chapters/（章节正文 + chapter_index.json 摘要）
        ├─ story/（truth files：story_bible、人物设定等）
        └─ .novelfork-project-init.json（若启用 project bootstrap，含 worktree 归属）

Narrator Session（独立于 Book）
  ├─ 可通过 projectId = bookId 关联到某本书
  ├─ sessionMode = chat | writer | editor
  └─ 通过 SessionCenter 管理，关闭窗口不销毁会话
```

**最小闭环的判定**：以下四条全部成立，即为"闭环已形成"：

1. 每个对象的**读写**都走统一 HTTP/WS 协议，`storage/adapter.ts` 层接口不被 UI 绕过
2. 每个对象的**前端状态**由服务端快照驱动，不依赖窗口 `persist` 里藏的副本
3. 每个对象的**异常恢复**（断网/重启/快照丢失）都有明确 UX 反馈（recovery 五态、startup diagnostics、session-store repair）
4. 每个对象的**ID / 命名空间**不冲突，跨对象引用通过明确字段（如 `session.projectId = bookId`）

当前四条**全部成立**。

**不属于"最小闭环"的长期愿景项**（在 closure 之外）：

- 多 Project 切换（目前单 Project、多 Book；未来可能引入"工作空间"概念让 Project 多实例化）
- Narrator 与 Book 的强绑定（目前 Narrator 可关也可不关某本 Book，未来可能要求必关）
- Chapter 与 Narrator 消息的双向追溯（"这条消息产出的是第 N 章草稿"回链）
- 跨 Book 的知识库/人物库共享（目前仅 story/ 目录局部）

## 十、Closure 完成后的长期愿景项（Package 5 归档）

closure spec 全部主线包完成后，以下事项**明确不在 closure 范围**，作为长期演进方向归档，后续任何时刻可拆独立 spec 推进：

**A. 顶层对象模型完整化**

1. 多 Project / 工作空间支持：当前是"单 Project（`novelfork.json`）+ 多 Book"，未来可扩展为"多 Project 可切换"，对应 Tauri/Web 端的项目选择器入口
2. Chapter 与 Narrator 消息的回链：当前 Narrator 产出的章节草稿与 `chapters/` 目录只靠 UI 约定关联，未来需要在消息元数据里记录"这条消息产出的是哪一章的哪个版本"
3. Narrator 与 Book 的强绑定选项：当前 Narrator 可以完全独立于 Book 存在，未来可能引入"必须挂靠某 Book"的创建模式
4. 跨 Book 的知识库 / 人物库共享：当前 truth files 仅在单 Book 的 `story/` 下，未来可能引入全局 lorebook

**B. 交付与运维成熟度**

1. 安装器 / 代码签名：当前仅 `bun compile` 产物，未做 installer wrapper、未做 Windows / macOS 代码签名
2. 自动更新：未接入 Tauri updater 或等价机制
3. 首启 UX 深度闭环：当前首启落到 SessionCenter 空态，未做 onboarding 向导 / 示例工程
4. 系统级实时诊断终态：当前 `startup-diagnostics` 覆盖 session-store / worktree / unclean-shutdown / provider；未覆盖长时运行期间的资源泄漏、磁盘占用增长、WS 连接池饱和等运行态诊断

**C. 运行时 repair / migration 终态**

1. Book 级 repair：当前 `startup-diagnostics` 在 session-store 层修 orphan；未覆盖"某本 Book 的 chapter_index.json 与 chapters/ 目录漂移"的自动修复
2. Schema migration：当前 `book.json` / `novelfork.json` 格式扩展靠"新字段 optional"兜底，未建立正式的 migration 机制
3. 跨版本 exe 数据迁移：未定义"exe v0.0.1 产生的 workspace，exe v0.0.2 如何识别并 migrate"的流程

归档这份清单的目的不是立即执行，而是让任何后续 PR / 会话能**立刻判断**："这个想法属于 closure 主线，还是属于长期愿景"。
