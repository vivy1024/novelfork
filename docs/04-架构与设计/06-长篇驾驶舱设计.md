> 当前口径（2026-05-03）：本文件保留为历史规划参考。`agent-native-workspace-v1` 已将工作台主入口调整为右侧固定叙述者会话、中间画布、左侧资源栏；长篇驾驶舱不再作为右侧常驻主入口实现。本文中“驾驶舱”模块可作为 `cockpit.*` session tools、工具结果卡片或中间画布组件复用，不能覆盖当前 session-first 架构事实。

## 1. 设计定位

长篇驾驶舱是 NovelFork Studio 面向长篇网文创作的中枢视图。它不替代章节编辑器、经纬编辑器、候选稿机制或 Agent 管线，而是在现有能力之上，把主线、人物反差、伏笔、设定规则、章节影响与 AI 运行状态聚合到一个可操作界面中。

核心目标是解决长篇 AI 写作最容易失败的几个问题：

- **梗点散落**: 爽点、反差、伏笔和设定分散在章节、经纬、Truth 文件里，作者难以快速判断下一章该推进什么。
- **长线失控**: AI 能写当前段落，但容易忘记十章前的承诺、人物秘密、势力利益链和世界规则。
- **状态不透明**: 用户不知道 AI 结果来自哪个模型，也不知道它依据了哪些上下文。
- **误写风险**: AI 产物不能绕过候选稿、草稿、审计与作者确认直接进入正式正文。

长篇驾驶舱应遵守当前项目事实：经纬、候选稿、草稿、章节编辑、Truth 文件查看、AI metadata、pending hooks 追踪已经具备基础；大纲 viewer/editor 仍处于规划中，不能在 UI 或文档中描述为已完成。

## 2. 设计原则

### 2.1 当前事实优先

所有界面状态必须区分真实可用、规划中、过程内存、prompt-preview 和 unsupported。没有接入真实 route 的能力不得显示为成功，也不得用本地假结果伪装。

### 2.2 非破坏式 AI 输出

AI 输出默认进入以下容器之一：

- **候选稿**: 可合并、替换、另存为草稿或放弃。
- **草稿**: 可编辑、保存，等待作者后续处理。
- **报告**: 审计、风险、AI 味、设定检查等只读结果。
- **预览**: prompt-preview 或章节作战卡，不能冒充正文。

正式章节只能由明确的用户操作更新。

### 2.3 一屏看清长篇状态

驾驶舱要让作者快速回答：

- 这本书当前主线在推进什么？
- 本章最该打的爽点是什么？
- 哪些伏笔快过期或需要回收？
- 哪些人物反差还没兑现？
- 哪些设定规则不能被下一章破坏？
- AI 这次写作依据了哪些上下文？

### 2.4 作者主导，Agent 辅助

AI 不应直接替作者决定长线方向。驾驶舱提供诊断、建议、候选方案和风险提示，最终由作者确认。

### 2.5 不剧透后续 Arc

驾驶舱默认只显示当前创作阶段允许暴露的信息。后续 arc 的隐藏设定、终局反转、未解锁真相应支持隐藏、折叠或按可见范围过滤。

## 3. 页面结构 UI/UX

### 3.1 总体布局

推荐采用“四区布局”：

```text
顶部操作栏
├─ 作品选择 / 模型状态 / 当前焦点 / 主要 AI 动作

左侧资源树
├─ 正式章节
├─ 候选稿
├─ 草稿
├─ 大纲
├─ 故事经纬
├─ Story / Truth 文件
├─ 素材
└─ 发布报告

中间主编辑区
├─ 章节编辑器
├─ 候选稿编辑器
├─ 草稿编辑器
├─ 经纬条目编辑器
├─ Markdown viewer
└─ 大纲 viewer/editor（规划中）

画布/工具结果中的长篇驾驶舱卡片
├─ 总览
├─ 主线与卷目标
├─ 人物反差
├─ 伏笔追踪
├─ 设定规则
├─ 章节影响
└─ AI 运行与审计

底部 AI 结果抽屉
├─ 候选稿生成结果
├─ 审计结果
├─ 修订结果
├─ 错误与 unsupported 信息
└─ 来源 metadata
```

### 3.2 顶部操作栏

顶部操作栏用于承载当前作品与高频操作，不承载复杂编辑。

建议字段：

- **作品选择**: 切换当前书籍。
- **模型门禁状态**: 显示 provider、model、可用/不可用、运行中。
- **当前焦点摘要**: 从 `current_focus.md` 或用户输入中读取一行摘要。
- **主要动作**:
  - 生成下一章候选稿。
  - 生成章节作战卡。
  - 审计当前章。
  - 去 AI 味。
  - 刷新驾驶舱。

UX 要求：

- 模型不可用时，AI 动作按钮 disabled，并显示原因。
- 运行中必须显示 loading，不允许静默等待。
- 成功后不直接切换正式章节，应进入候选稿或报告视图。

### 3.3 左侧资源树

左侧资源树继续沿用当前 Workspace 资源模型。长篇驾驶舱不新增一套独立导航，避免用户在两个资源体系中迷路。

建议分组：

- **正式章节**: 展示章节号、标题、状态、字数、审计状态。
- **候选稿**: 展示目标章节、AI 来源、生成时间。
- **草稿**: 展示草稿标题、更新时间、来源。
- **大纲**: 当前为规划中能力；无真实 editor 时必须透明显示未完成。
- **故事经纬**:
  - 人物。
  - 地点。
  - 势力。
  - 物品。
  - 伏笔。
  - 世界规则。
  - 章节摘要。
- **Story / Truth 文件**:
  - `story_bible.md`。
  - `volume_outline.md`。
  - `book_rules.md`。
  - `current_state.md`。
  - `particle_ledger.md`。
  - `pending_hooks.md`。
  - `chapter_summaries.md`。
  - `author_intent.md`。
  - `current_focus.md`。

UX 要求：

- 资源节点点击后，中间区显示真实 editor/viewer。
- 没有实现的节点使用 UnsupportedCapability，不显示假编辑器。
- 伏笔节点应能追踪到 `pending_hooks.md` 的原始来源。

### 3.4 中间主编辑区

中间区是用户真正工作的地方。长篇驾驶舱只提供辅助，不抢夺主编辑区。

主要视图：

- **章节编辑器**: 编辑正式章节，保存后刷新资源树与驾驶舱摘要。
- **候选稿编辑器**: 只读展示候选内容，并提供合并、替换、另存草稿、放弃。
- **草稿编辑器**: 编辑草稿，保存后保持为草稿，不自动进入正式章节。
- **经纬条目编辑器**: 新建、查看、编辑人物、设定、事件、伏笔等条目。
- **Markdown viewer**: 查看 Story / Truth 文件原文。
- **大纲 viewer/editor**: 任务 35 规划中，完成前只显示透明未实现状态。

UX 要求：

- 中间区的主操作必须有保存状态：loading、dirty、saving、saved、error。
- 候选稿合并/替换前必须二次确认。
- 替换正式章节需要明确显示影响范围。

### 3.5 长篇驾驶舱卡片 / 画布组件

长篇驾驶舱在当前 session-first 架构中不再默认常驻右侧，而应作为工具结果卡片或中间画布组件提供“当前创作决策”相关信息。

建议使用 tab + card 结构：

```text
长篇驾驶舱
├─ 总览
├─ 主线
├─ 人物
├─ 伏笔
├─ 设定
├─ 章节影响
└─ AI 运行
```

右侧宽度建议 360px 到 440px。窄屏时收起为抽屉，避免压缩中间编辑器。

## 4. 驾驶舱模块设计

### 4.1 总览 Tab

总览是默认页，回答“下一章该怎么写”。

展示内容：

- **当前焦点**: 来自 `current_focus.md` 或用户最近输入。
- **本卷目标**: 来自 `volume_outline.md` 或未来 outline 数据。
- **最近章节摘要**: 来自 `chapter_summaries.md` 或经纬 chapter summaries。
- **待处理风险**:
  - 审计失败章节。
  - state-degraded。
  - 快过期伏笔。
  - 未解决 blocking issue。
- **推荐下一步**:
  - 生成章节作战卡。
  - 生成候选稿。
  - 补充人物设定。
  - 回收伏笔。

交互：

- 点击风险卡片跳转到对应章节、经纬条目或 Truth 文件。
- 点击“生成章节作战卡”进入 AI 预览流程，而不是直接写正式正文。

### 4.2 主线 Tab

主线 Tab 用来判断当前故事是否还在推进正确方向。

展示内容：

- **当前卷目标**。
- **当前敌人/阻力**。
- **阶段胜负条件**。
- **主线推进进度**。
- **最近 5 章主线动作**。
- **偏航风险**: 连续多章没有推进主线、支线过长、冲突弱化。

数据来源：

- `volume_outline.md`。
- `current_focus.md`。
- `chapter_summaries.md`。
- 未来 outline editor 输出。

规划能力：

- 主线推进度初期可以由摘要启发式计算。
- 后续可引入 `storyline` 结构化记录。

### 4.3 人物反差 Tab

人物 Tab 重点不是人物百科，而是可用于写作的“反差与张力”。

每个人物卡片建议展示：

- **表面身份**。
- **真实欲望**。
- **隐藏秘密**。
- **人物反差**。
- **当前关系**。
- **最近出场章节**。
- **下一次可用爆点**。
- **OOC 风险**。

交互：

- 点击人物进入经纬人物编辑器。
- 点击“用于下一章”把该人物加入章节作战卡上下文。
- 对未完善人物显示“缺少欲望/恐惧/反差”等轻量提示。

数据来源：

- 经纬 `characters`。
- `story_bible.md`。
- `chapter_summaries.md`。
- ContinuityAuditor 的 OOC 相关 issues。

### 4.4 伏笔追踪 Tab

伏笔 Tab 是长篇驾驶舱的核心模块之一。

伏笔状态建议分为：

- **已埋**: 已在正文出现，等待推进。
- **待强化**: 需要再次提醒读者。
- **待回收**: 已到回收窗口。
- **过期风险**: 太久未处理。
- **已回收**: 已完成闭环。
- **废弃/冻结**: 作者明确不再使用。

每条伏笔展示：

- **hook id**。
- **文本**。
- **来源章节**。
- **关联章节**。
- **关联人物/设定**。
- **预计回收窗口**。
- **当前状态**。
- **来源文件**。

数据来源：

- `pending_hooks.md`。
- 经纬 `events`。
- 写作工具 hook apply 结果。
- 章节摘要。

交互：

- 点击伏笔跳转到来源章节或 `pending_hooks.md`。
- 点击“加入下一章”把伏笔加入章节作战卡。
- 点击“标记已回收”时必须经过真实 API 或经纬编辑，不允许仅 UI 本地改状态。

### 4.5 设定规则 Tab

设定 Tab 用来避免战力、资源、世界规则崩坏。

展示内容：

- **世界规则**。
- **战力体系**。
- **资源账本**。
- **势力利益链**。
- **禁忌与代价**。
- **最近变更**。
- **风险提示**。

数据来源：

- 经纬 `settings`。
- `book_rules.md`。
- `particle_ledger.md`。
- `current_state.md`。
- ContinuityAuditor 的设定冲突、战力崩坏、数值检查 issues。

交互：

- 点击规则进入经纬设置条目。
- 点击账本来源进入 `particle_ledger.md` viewer。
- AI 写作前将高风险规则加入 rule stack。

### 4.6 章节影响 Tab

章节影响 Tab 用来判断“当前章写完后改变了什么”。

展示内容：

- **新增事实**。
- **人物关系变化**。
- **新增伏笔**。
- **回收伏笔**。
- **资源/数值变化**。
- **主线推进**。
- **需要人工确认的变化**。

数据来源：

- Writer 的 RuntimeStateDelta。
- `current_state.md`。
- `particle_ledger.md`。
- `pending_hooks.md`。
- `chapter_summaries.md`。
- runtime trace。

UX 要求：

- 如果来源是 runtime 中间产物，必须标注来源与可信度。
- 如果状态校验失败，显示 state-degraded，不允许假装入库成功。

### 4.7 AI 运行 Tab

AI 运行 Tab 展示当前或最近一次 AI 行动。

展示内容：

- **运行类型**: write-next、draft、audit、revise、de-ai、continuity。
- **运行状态**: pending、running、success、failed、unsupported。
- **provider / model / run id**。
- **输入摘要**。
- **输出容器**: 候选稿、草稿、报告、预览。
- **错误原因**。

数据来源：

- AI action runtime state。
- SSE / Studio event。
- 候选稿、草稿、报告 metadata。
- model gate。

UX 要求：

- route 不存在或未接入真实能力时显示 unsupported。
- 模型不可用时显示 model unavailable。
- 上游失败时显示失败信息，不写入假候选稿。

## 5. 数据流设计

### 5.1 初始加载流

```text
WritingWorkbenchRoute mount
  -> load books
  -> select activeBookId
  -> load backend-contract resource tree
  -> build Workbench resource tree
  -> load cockpit snapshot
      -> book config
      -> chapter index
      -> candidates / drafts
      -> bible categories
      -> story files
      -> truth files
      -> model gate
  -> render editor + cockpit
```

第一阶段可以由前端聚合已有 API，不强制新增 cockpit API。这样能最快复用当前能力，并避免提前设计过重的数据层。

### 5.2 驾驶舱快照流

建议中期引入只读聚合接口：

```text
GET /api/books/:id/cockpit/snapshot
```

返回建议结构：

```json
{
  "bookId": "book-id",
  "summary": {
    "currentFocus": "当前创作焦点",
    "volumeGoal": "当前卷目标",
    "latestChapter": 12,
    "riskCount": 3
  },
  "mainline": [],
  "characters": [],
  "hooks": [],
  "rules": [],
  "chapterImpact": [],
  "aiRuntime": {
    "status": "idle",
    "provider": "provider",
    "model": "model"
  },
  "sources": []
}
```

约束：

- 该接口只读，不修改章节、候选稿、草稿或 Truth 文件。
- 无法解析的数据必须返回空数组、unknown 或 unsupported，不返回假内容。
- 每条聚合结果应尽量携带 source，方便 UI 跳转到原文。

### 5.3 生成章节作战卡流

章节作战卡是正式写作前的 AI 预览，不是正文。

```text
用户点击生成章节作战卡
  -> 检查 model gate
  -> 收集 cockpit snapshot
  -> 收集 current_focus / author_intent
  -> 调用 Planner 或专用 planning route
  -> 返回 prompt-preview / chapter-card
  -> 用户确认或编辑
  -> 可选：生成候选稿
```

章节作战卡建议字段：

- **章节目标**。
- **主冲突**。
- **本章爽点**。
- **人物推进**。
- **伏笔动作**。
- **设定约束**。
- **信息边界**。
- **结尾钩子**。
- **禁止事项**。

### 5.4 生成下一章候选稿流

```text
用户确认章节作战卡
  -> 调用 write-next 或 draft route
  -> PipelineRunner 准备输入
  -> Planner / Composer / Writer
  -> PostWriteValidator
  -> ContinuityAuditor
  -> Reviser if needed
  -> StateValidator
  -> 生成候选稿或正式待审产物
  -> 保存 AI metadata
  -> UI 显示候选稿
  -> 用户选择合并 / 替换 / 另存草稿 / 放弃
```

产品侧推荐：

- Studio 默认走候选稿路径。
- CLI 或高级模式可以继续使用完整写入管线，但 UI 需要明确风险与结果状态。
- 候选稿 metadata 必须显示 provider、model、run id 或 request id。

### 5.5 伏笔追踪流

```text
用户生成或应用 hook
  -> 写作工具 route 产出 hook
  -> apply 后写入 pending_hooks.md
  -> 伏笔分类解析 pending_hooks.md
  -> 驾驶舱 hooks tab 刷新
  -> 用户可跳转来源章节 / story file viewer
```

约束：

- `pending_hooks.md` 是可追踪来源之一，不应被 UI 私自覆盖。
- 标记回收、关联章节等操作必须通过真实 API 或经纬编辑器持久化。

### 5.6 章节保存与刷新流

```text
用户保存正式章节
  -> chapter API 保存正文
  -> 更新 chapter index / updatedAt
  -> refresh resource snapshot
  -> refresh cockpit snapshot
  -> 审计状态保持原样或标记需重新审计
```

建议：

- 如果用户手动改动已审计章节，驾驶舱显示“建议重新审计”。
- 不自动触发 AI 审计，避免用户每次保存都消耗模型。

### 5.7 错误与 unsupported 流

```text
用户触发 AI action
  -> adapter/model/route 检查
  -> unsupported / unavailable / failed / success
  -> UI 显示明确状态
  -> 不生成假候选稿
  -> 不更新正式章节
```

错误类型：

- **unsupported**: 当前 route 或 adapter 未实现。
- **model-unavailable**: 模型未配置或 runtime gate 不通过。
- **upstream-failed**: Provider 返回错误。
- **state-degraded**: Truth 文件或状态校验失败。
- **validation-failed**: 输入缺少 bookId、chapterId 或必要字段。

## 6. Agent 管线关系

长篇驾驶舱不应破坏现有 Agent 单一职责。

现有职责保持：

- **Planner**: 生成章节意图。
- **Composer**: 组装上下文、规则栈与 trace。
- **Writer**: 写正文与状态 delta。
- **PostWriteValidator**: 检查重复、节奏、AI tells 等。
- **ContinuityAuditor**: 审计连续性和题材规则。
- **Reviser**: 根据问题修订。
- **StateValidator**: 校验状态落盘。

驾驶舱新增的是“上下文可视化与决策辅助”，不是新的万能 Agent。

建议分层：

```text
UI Cockpit Panel
  -> Cockpit Snapshot Service
    -> Existing API / storage / truth files / bible routes
      -> PipelineRunner / agents remain unchanged
```

未来如需新增能力，优先新增服务而不是让 Agent 直接读写 UI 状态：

- **CockpitSnapshotService**: 聚合只读数据。
- **ChapterCardService**: 基于快照生成章节作战卡。
- **LongFormRiskAnalyzer**: 分析伏笔过期、主线偏航、人物长时间未出场等风险。

这些服务可以调用 Agent，但不应让 Agent 绕过 PipelineRunner 直接改 Truth 文件。

## 8. 推荐实施阶段

### Phase 1：只读驾驶舱

目标：不新增复杂写入能力，先把现有事实展示出来。

范围：

- cockpit 工具结果卡片和中间画布 shell。
- 总览、人物、伏笔、设定、AI 运行 tab。
- 从已有 API 聚合数据。
- 所有未实现能力显示 unsupported 或 planning。

验收：

- 有书时能显示真实数据。
- 无书时显示空状态。
- route 不支持时显示透明 unsupported。
- 点击来源能跳转资源树节点或打开 viewer。

### Phase 2：Cockpit Snapshot API

目标：把前端聚合逻辑下沉为只读后端接口。

范围：

- `GET /api/books/:id/cockpit/snapshot`。
- 标准化 source 引用。
- 风险卡片聚合。
- AI runtime metadata 聚合。

验收：

- API route 测试覆盖有数据、空数据、book 不存在、解析失败。
- UI 测试覆盖加载、刷新、错误显示。

### Phase 3：章节作战卡

目标：写正文前先让作者确认“这一章怎么打”。

范围：

- 生成章节作战卡。
- 支持用户编辑作战卡。
- 作战卡作为 write-next/draft 的 externalContext。
- 输出仍进入候选稿或草稿。

验收：

- 模型不可用时不能触发。
- 作战卡不会写入正式章节。
- 用户确认后才进入候选稿生成。

### Phase 4：长篇风险分析

目标：让系统主动提醒长篇风险。

范围：

- 伏笔过期风险。
- 人物长期未出场。
- 主线连续未推进。
- 设定规则冲突。
- 审计失败聚合。

验收：

- 风险项有 source。
- 风险项可跳转。
- unknown 不显示为 passed。

## 9. 测试策略

### 9.1 UI 测试

覆盖场景：

- 有作品、有经纬、有 pending hooks 时正常展示。
- 无作品时显示空状态。
- 点击人物、伏笔、设定卡片能进入对应编辑器或 viewer。
- 模型不可用时 AI 动作 disabled。
- unsupported route 显示透明提示。
- 候选稿操作仍需要确认。

### 9.2 API 测试

如果新增 cockpit snapshot route，覆盖：

- book 存在且有完整数据。
- book 存在但 story files 缺失。
- book 不存在。
- `pending_hooks.md` 格式不完整。
- bible route 数据为空。
- metadata 缺失时返回 unknown。

### 9.3 集成测试

覆盖流程：

```text
创建作品
  -> 新建经纬人物
  -> 应用 hook
  -> 写入 pending_hooks.md
  -> 打开驾驶舱
  -> 看到人物与伏笔
  -> 生成章节作战卡
  -> 生成候选稿
  -> 候选稿另存草稿
```

### 9.4 禁止回归

测试必须防止：

- AI action 回到“即将推出”但仍显示成功。
- prompt-preview 被当成正式正文。
- unsupported 被当成已接入能力。
- 候选稿绕过确认直接覆盖正式章节。
- process-memory 被描述成跨重启保存。

## 10. 开放问题

- 大纲 editor 完成后，`volume_outline.md` 与结构化 outline 记录谁是主事实源？
- 章节作战卡是否需要持久化到 `story/runtime/`，还是只作为候选生成的 metadata？

- 主线偏航分析是否应该作为 AI 报告保存，还是每次实时计算？

## 11. 推荐结论

长篇驾驶舱应先做成“只读聚合 + 可跳转 + 透明状态”的辅助面板，再逐步加入章节作战卡和长篇风险分析。

最小可落地路径是：

```text
已有经纬 / Truth 文件 / 候选稿 / AI metadata
  -> 只读驾驶舱
  -> Cockpit Snapshot API
  -> 章节作战卡
  -> 长篇风险分析
```

这个方向能最大化复用 NovelFork 当前已经完成的能力，同时把项目差异化聚焦到“长篇不崩、伏笔可控、人物反差可见、AI 结果可审阅”。
