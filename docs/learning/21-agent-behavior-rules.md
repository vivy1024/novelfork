# Agent 行为规范

> Agent 的自律机制，确保高效可靠地完成任务。

## 失败循环检测

同一种方法连续失败 2 次后，Agent 必须停下来：

1. 诊断根因（不再做增量修补）
2. 解释失败原因
3. 提出完全不同的方案
4. 征求用户确认后再继续

不允许同一条路反复试错超过 2 次。

## 默认行动决策

Agent 优先行动，不只是建议：

| 场景 | 行为 |
|------|------|
| 小改动（改名、修 typo、单文件调整） | 直接做，不问 |
| 中等改动（多文件、陌生代码） | 先读相关代码，再实施 |
| 大改动（架构调整、新模块） | 先说方案，等确认再动手 |
| 分析/比较类请求 | 只输出分析，不动代码 |

不确定时，推断最可能的有用行动并执行，通过读代码补充缺失信息，而非猜测。

## 验证流程

每次代码改动后必须验证：

1. **编译检查** — 运行项目构建，确保无报错
2. **功能验证** — 运行相关测试或实际操作确认行为正确
3. **前端改动** — 必须截图确认 UI 正确
4. **后端改动** — 必须 curl 实测确认 API 正确

验证发现问题时当场修复，不交给用户。

## Bash 使用规则

| 规则 | 说明 |
|------|------|
| 超时处理 | 预计超过 30 秒的命令必须 `run_in_background` |
| 禁止甩锅 | 不要让用户手动执行命令，Agent 自己执行 |
| 实时输出 | 长命令使用后台模式 + 轮询结果 |
| 路径安全 | Windows 下用 Git Bash 语法，注意路径转义 |

## 上下文压缩感知

当上下文窗口被压缩（compaction）后：

- Agent 不依赖记忆中"之前做了什么"
- 主动检查文件状态和命令输出确认当前进度
- 如果在多步任务中间被压缩，从检查点重新确认后继续

这确保压缩不会导致错误操作或重复工作。

## Agent 查阅提示

- 失败循环：2 次同类失败 → 停下诊断 → 换方法 → 征求确认
- 默认行动：小改动直接做，大改动先确认，分析类不动代码
- 验证是交付条件：未验证 = 未完成
- Bash 30 秒规则：dev server、编译等长命令必须后台运行
- 压缩后：读文件/跑命令确认状态，不靠记忆
