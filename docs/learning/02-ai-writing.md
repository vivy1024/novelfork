# AI 写作功能

> NovelFork 的 AI 写作能力覆盖从续写到审校的全流程，所有生成结果以候选稿形式呈现，由你决定是否采纳。

---

## 写作动作

工作台顶部的 WorkbenchWritingActions 提供以下快捷动作：

| 动作 | ID | 输出边界 | 说明 |
|------|-----|---------|------|
| 生成下一章 | session-native.write-next | session → candidate | 完整写作流程：PGI → Guided Plan → 生成 |
| 续写草稿 | ai.draft.async | session → draft | 快速续写，不走完整审批流程 |
| 扩写/改写 | writing-modes.preview | candidate | 对选中内容进行变换 |
| 连续性审校 | ai.audit | audit | 检查前后章节的设定一致性 |
| 去 AI 味检测 | ai.detect | audit | 检测并标记 AI 生成痕迹 |
| 伏笔建议 | hooks.generate | candidate | 生成章末钩子方案 |

### 动作触发方式

1. 工作台顶栏按钮点击
2. 叙述者对话中自然语言请求
3. Slash 命令（如 `/write`）

## 写作模式

### 续写

在当前章节末尾继续生成内容。适合：
- 写到一半需要 AI 接力
- 已有开头，需要 AI 补完

### 扩写

将简短段落扩展为详细描写。适合：
- 大纲式文字需要丰满
- 对话场景需要补充动作描写

### 过渡

在两个场景之间生成过渡段落。适合：
- 时间跳跃
- 场景切换
- 视角转换

### 对话生成

DialogueGenerator 组件，专门生成角色对话。特点：
- 基于角色设定生成符合人设的对话
- 控制对话节奏和情绪
- 支持多角色对话场景

### 变体生成

VariantCompare 组件，为同一段落生成多个变体供选择：
- 同时生成 2-4 个不同风格的版本
- 并排对比，选择最满意的

### 大纲分支

OutlineBrancher 组件，从当前节点生成多条故事走向：
- 探索不同的剧情发展可能
- 选择一条分支继续深入

## 选区变换

在编辑器中选中文本后，可执行以下变换：

| 变换 | 说明 |
|------|------|
| 润色 | 提升文字质量，保持原意 |
| 精简 | 删减冗余，保留核心信息 |
| 扩写 | 增加细节和描写 |
| 审查 | 检查选中内容的逻辑/设定问题 |

选区变换的结果以 inline diff 形式展示，你可以接受或拒绝每处修改。

## 行内续写（Tab 补全）

InlineWritePanel + ghost-text 扩展提供类似 IDE 的补全体验：

1. 光标停在段落末尾
2. 系统自动预测下一句（灰色幽灵文本）
3. 按 Tab 接受补全
4. 按 Esc 忽略

### 触发条件

- 光标在段落末尾停留 > 1 秒
- 当前段落有足够上下文
- 模型已配置且可用

## 智能大纲

### 大纲生成

基于作品设定和已有章节，生成后续章节的大纲建议。

### 大纲检查

检查现有大纲的逻辑一致性：
- 伏笔是否有回收
- 角色弧线是否完整
- 节奏是否合理

### 大纲建议

针对当前进度，给出下一步写作方向的建议。

## 候选稿流程

所有 AI 生成的正文内容都以候选稿（Candidate）形式存在，不会直接覆盖你的正文。

### 生命周期

```
AI 生成 → candidate（待处理）
                ↓
    ┌───────────┼───────────┐
    ↓           ↓           ↓
  接受        拒绝        归档
    ↓
 ┌──┼──┐
 ↓  ↓  ↓
merge replace draft
```

### 接受方式

| 方式 | 说明 |
|------|------|
| merge | 将候选稿内容合并到目标章节（追加或智能合并） |
| replace | 用候选稿完全替换目标章节内容 |
| draft | 保存为草稿，不立即写入正文 |

### CandidateActionsBar

候选稿打开后，顶部显示操作栏：
- 状态标签（待处理/已接受/已拒绝/已归档）
- 来源信息（哪个动作生成的）
- 目标章节
- 创建时间
- 操作按钮

## 上下文组装

AI 写作时，系统自动组装以下上下文：

### 组装顺序

```
1. 真相文件（Truth Files）
   └── 经纬条目中 participatesInAi=true 的内容

2. 经纬上下文（Jingwei Context）
   └── 按可见性规则过滤后的设定条目

3. POV 过滤
   └── 当前视角角色能"知道"的信息

4. 前文摘要
   └── 前 N 章的压缩摘要

5. 当前章节上下文
   └── 已有正文 + 大纲意图

6. 用户指导
   └── PGI 回答 + Guided Plan
```

### 上下文预算

每个经纬条目有 `tokenBudget` 字段，控制该条目在上下文中占用的最大 token 数。系统会在预算内智能裁剪。

## 最佳实践

- **用"生成下一章"而非"续写草稿"**：前者走完整流程（PGI + Guided Plan），质量更高
- **不要跳过 PGI**：追问帮助 AI 理解你的具体意图
- **善用变体**：不确定方向时，生成多个变体对比
- **定期审校**：每写 5-10 章跑一次连续性审校
- **控制经纬规模**：过多条目会撑爆上下文，用可见性规则精准控制

## 常见坑

| 问题 | 原因 | 解决 |
|------|------|------|
| 生成内容重复前文 | 上下文中前文摘要不够精炼 | 手动编辑前文摘要 |
| 候选稿字数不达标 | 未在 PGI 中明确字数要求 | 在追问中说明目标字数 |
| 去 AI 味后文字生硬 | 过度修改 | 只对明显 AI 痕迹处理 |
| 审校报告太多误报 | 经纬设定不完整 | 补充相关设定到经纬 |
