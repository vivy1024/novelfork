# Agent 写作管线

> 5 Agent 协作管线——规划、编排、写作、审计、修订。

## 核心概念

NovelFork 的写作不是单次 API 调用，而是多 Agent 协作的管线：

```
用户请求 → PGI 追问 → 规划 → 编排 → 写作 → 审计 → 修订 → 入库
```

**5 个 Agent 角色**：

| Agent | 职责 |
|-------|------|
| 写书 | 生成章节正文 |
| 伏笔 | 埋设/追踪/回收伏笔线索 |
| 章末钩子 | 生成章末悬念方案 |
| 审校 | 连续性审计，检测矛盾与设定冲突 |
| 大纲与经纬 | 全书架构规划 + 经纬维护 |

**确认门**：写入正式章节前暂停，用户批准后才入库。

**WorkflowProgressCard**：实时展示管线执行进度（pending / running / success / failed / approval-pending）。

## 推荐使用流程

1. 在叙述者对话中发起写作请求
2. PGI 自动触发，回答追问明确写作意图
3. Guided Plan 生成章节计划，审阅后批准
4. 管线自动执行：编排上下文 → 生成正文 → 审校 → 修订
5. 确认门暂停，审阅候选稿后批准入库

## 最佳实践

- 认真回答 PGI 追问，输入越精确输出质量越高
- 关注审校步骤发现的问题，不要盲目跳过
- 不同 Agent 可配不同模型：Writer 用强模型保质量，Auditor 用快模型省成本
- 保持经纬完整——管线质量高度依赖经纬中的设定

## 常见坑

- **管线中途失败** → API 超时或模型不可用，智能重试会自动处理 429/502/503
- **审校误报过多** → 经纬设定不完整，审校 Agent 缺乏判断基准
- **工作流卡在 approval-pending** → 确认门等待用户操作，在对话中批准或拒绝
- **上下文溢出** → 章节过多导致超限，自动恢复机制会截断历史保留关键信息

## Agent 查阅提示

- 管线入口：叙述者对话触发，PipelineRunner 调度
- 核心管线工具：`plan_chapter` / `compose_chapter` / `audit_chapter` / `revise_chapter` / `import_chapters`
- 只读工具自动并行，写入工具串行 + 确认门
- 智能重试：429/502/503 指数退避，最多 3 次
- 上下文溢出恢复：自动截断早期消息，保留系统提示 + 最近 N 轮
- 确认门触发条件：写入正式章节、修改经纬、删除资源、执行修订
- 安全原则：最小权限（默认只读）、可回退（写入前自动 Checkpoint）、透明（工具调用可见）、用户主权（随时中断）
