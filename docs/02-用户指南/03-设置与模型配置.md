## 本文定位

本文说明当前 Studio 中模型与供应商配置的真实口径。当前主线不是旧的浏览器本地模型配置，也不是只靠 CLI 写配置文件；主线是 Studio 设置页、Provider runtime store 和运行时模型池。

## 配置不再是创作前置硬门槛

未配置模型时，用户仍可完成本地创作管理工作：

- 打开 Studio。
- 创建或查看作品。
- 编辑已有章节。
- 查看和维护当前可用的故事经纬入口。
- 查看部分不依赖模型的写作工具结果。

只有触发 AI 续写、生成、审校、改写、去 AI 味、深度检测或 Agent 动作时，才需要可用模型。

## 当前模型配置主线

```text
设置页 / AI 供应商
  -> Provider runtime store
  -> 平台账号 / API key 供应商
  -> runtime model pool
  -> ChatWindow / WritingTools / AI gate 读取模型池
```

这条主线的意义是：

1. 供应商和模型不再散落在多个浏览器本地配置里。
2. 模型列表从 runtime store 派生。
3. 凭据缺失、账号禁用或模型不可用时，模型池应过滤或返回明确错误。
4. AI 调用失败时不能生成假 assistant 正文或假写作结果。

## 当前已知配置能力

| 能力 | 当前状态 | 说明 |
|---|---|---|
| Provider runtime store | 真实可用 | 供应商和模型运行时配置主线 |
| API key 接入 | 真实可用 | 通过设置页维护自定义供应商凭据 |
| Codex / Kiro 平台账号导入 | 真实可用 | 账号 JSON 持久化到 runtime store |
| Cline 平台账号导入 | 透明过渡 | 当前应返回 unsupported，不能假装已接 |
| Runtime model pool | 真实可用 | 从 runtime store 派生可用模型 |
| Legacy Model UI | 待迁移 | 旧模型选择组件只保留兼容或引导，不应扩展新功能 |

## 推荐配置流程

1. 启动 Studio。
2. 进入设置页。
3. 打开 AI 供应商相关分区。
4. 添加或启用供应商。
5. 填写 Base URL、API Key、兼容格式、模型信息等必要字段。
6. 刷新或读取模型列表。
7. 使用测试动作确认模型可用。
8. 返回创作工作台，触发写作工具或 AI 动作。

如果模型不可用，系统应该显示错误原因或 AI gate，而不是返回虚假的生成结果。

## 配置层次说明

当前文档中提到的配置大致分为三类：

| 类型 | 例子 | 说明 |
|---|---|---|
| 运行参数 | `--root`、`--port`、`NOVELFORK_STUDIO_PORT` | 控制 Studio 进程如何启动 |
| 项目配置 | `novelfork.json`、项目语言、默认模型等 | 与当前项目根目录相关 |
| 供应商运行时配置 | Provider runtime store、平台账号、API key | 当前模型池主线 |

不要把这三类混在一起写。启动参数不是模型配置，模型配置也不等同于项目结构配置。

## 错误和空态应该怎么表现

| 场景 | 正确表现 |
|---|---|
| 未配置 API Key | 提示需要配置供应商或凭据 |
| 模型列表为空 | 显示无可用模型，并提供设置入口 |
| 供应商不可用 | 显示失败原因、HTTP 状态或 provider 错误 |
| 账号被禁用 | 模型池中过滤该账号模型 |
| 当前功能未接平台 | 返回 unsupported 或禁用按钮 |
| AI 调用失败 | 返回错误 envelope，不写入假正文 |

## 安全要求

- 不要把真实 API Key 写入文档、测试数据或提交记录。
- 文档示例必须使用占位值。
- UI 展示 API Key 时必须脱敏。
- 排障时只记录错误类型和请求元数据，不记录密钥明文。
