## 本文定位

本文只描述 NovelFork 当前真实可用的启动方式。旧文档中以 `pnpm build`、全局 CLI 链接、PWA 安装或大规模自动写作 CLI 为核心的内容已经归档，不再作为当前推荐入口。

当前推荐把 NovelFork 当成本地 Studio 工作台启动：本机运行 Hono API 和前端静态资源，浏览器访问 `http://localhost:4567`。

## 前置条件

| 工具 | 用途 | 当前口径 |
|---|---|---|
| Git | 克隆仓库、查看变更、开发协作 | 必需 |
| Node.js | 运行 pnpm、部分构建脚本和兼容工具 | 推荐使用项目要求的 Node 版本 |
| pnpm | 安装依赖、运行 workspace 脚本 | 必需 |
| Bun | 运行 `main.ts` 和编译单文件产物 | 当前主运行时 |

> 当前是原生 Windows 环境可用的项目，不需要切换到 WSL。

## 获取源码

```bash
git clone https://github.com/vivy1024/novelfork.git novelfork
cd novelfork
pnpm install
```

如果只是继续已有工作区，进入仓库后先看状态：

```bash
git status --short
```

## 开发运行

开发时优先使用 Bun 主入口：

```bash
bun run main.ts --root=. --port=4567
```

启动后访问：

```text
http://localhost:4567
```

这个模式的特点：

- 直接使用当前源码。
- 适合调试 Studio API、前端页面和文档提到的当前能力。
- 如果前端静态资源不存在，服务可能以 API-only 或缺资源状态启动，需要看启动日志。

## 编译运行

当前编译交付入口是：

```bash
pnpm bun:compile
```

该命令会执行：

```text
pnpm bun:build-client
  -> 构建 Studio 前端
pnpm bun:embed-assets
  -> 生成 embedded frontend assets
bun build ./main.ts --compile --outfile dist/novelfork
  -> 编译单文件产物
```

编译后运行：

```bash
./dist/novelfork.exe --root=. --port=4567
```

如果在非 Windows 环境构建，产物文件名可能不是 `.exe`，以 `dist/` 下实际文件为准。

## 当前推荐启动路径

### 本地开发者

```text
pnpm install
  -> bun run main.ts --root=. --port=4567
  -> 打开 http://localhost:4567
```

### 验证编译产物

```text
pnpm bun:compile
  -> ./dist/novelfork.exe --root=. --port=4567
  -> 打开 http://localhost:4567
```

### 只想读文档或检查文档体系

```bash
bun run docs:verify
```

## 首次打开后应该做什么

1. 打开 Studio。
2. 查看是否能进入创作工作台。
3. 检查是否已有作品；没有作品时先创建或导入测试作品。
4. 未配置模型时仍可进行本地章节编辑和资料维护。
5. 需要 AI 能力时，再进入设置页配置供应商和模型。

## 当前边界

当前仍是 Bun-first 交付形态。以下内容不能写成已经完成：

- installer
- signing
- auto-update
- first-launch UX
- 完整 repair / migration 终态
- 完整桌面应用壳体验

## 常见启动问题

| 现象 | 优先检查 |
|---|---|
| 端口无法访问 | 是否启动到了 `4567`；是否被其他进程占用；是否设置了 `NOVELFORK_STUDIO_PORT` |
| 页面空白或资源缺失 | 是否执行过前端构建或 embedded assets 生成 |
| API 可访问但前端不可访问 | 可能是 static provider 缺失，查看启动日志中的 static provider 状态 |
| 模型调用失败 | 先检查设置页 provider/runtime model pool，再看凭据和模型是否可用 |
| 文档链接失效 | 运行 `bun run docs:verify` |
