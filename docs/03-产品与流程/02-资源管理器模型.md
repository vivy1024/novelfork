## 本文定位

本文说明创作工作台左侧“小说资源管理器”的产品模型。它不是通用文件浏览器，也不是 IDE 项目树，而是围绕长篇小说创作对象组织的资源树。

## 资源管理器的目标

资源管理器应该帮助作者快速回答四个问题：

1. 我现在在写哪本书？
2. 这本书有哪些正式章节？
3. AI 生成了哪些候选稿或草稿？
4. 写这一章时需要参考哪些大纲、资料、伏笔和发布检查？

## 资源树推荐结构

```text
作品
├── 卷
│   ├── 已有章节
│   ├── 生成章节
│   └── 草稿
├── 大纲
├── 故事经纬
│   ├── 人物
│   ├── 地点
│   ├── 势力
│   ├── 物品
│   ├── 伏笔
│   └── 世界规则
├── 素材
└── 发布报告
```

当前不要求所有节点都已经完整做实。资源管理器可以先显示已接入资源和透明过渡入口，但不能把未接入入口伪造成可用功能。

## 当前资源对象状态

| 对象 | 说明 | 当前状态 | 当前表现 |
|---|---|---|---|
| 作品 | 一本小说的根对象 | current | 可通过作品选择和书籍 API 获取 |
| 已有章节 | 正式章节正文 | current | 可在资源树中打开、编辑、保存 |
| 生成章节 | AI 候选稿或生成结果 | current | 可打开为候选稿，并执行处理动作 |
| 卷 | 章节分组 | planning | 后续补齐长篇结构管理 |
| 草稿 | 未合并正文的备选文本 | planning | 候选稿另存草稿路径仍待完整资源化 |
| 大纲 | 作品和章节规划 | planning | 当前多为入口或空态 |
| 故事经纬 | 人物、地点、伏笔、世界规则等资料 | planning/current 混合 | 面板和仓储可复用，完整工作台 CRUD 待补齐 |
| 素材 | 外部参考、片段、设定材料 | planning | 后续接入 |
| 发布报告 | 发布前检查与导出前报告 | planning | 后续接入 |

## 正式章节与候选稿的区别

| 维度 | 正式章节 | 候选稿 |
|---|---|---|
| 来源 | 用户创建、保存或确认后的正文 | AI 生成、改写或续写产生的候选内容 |
| 是否代表作品正文 | 是 | 否 |
| 是否可直接发布 | 可以进入后续发布检查 | 不能直接发布 |
| 修改方式 | 编辑并保存 | 合并、替换、另存草稿或放弃 |
| 风险 | 正式内容变更 | 不应自动影响正式正文 |

资源树必须让这两类对象看起来不同，避免用户误把候选稿当成正文。

## 节点点击后的打开规则

| 节点 | 推荐打开位置 | 说明 |
|---|---|---|
| 正式章节 | 中央正文编辑区 | 可编辑、可保存 |
| 候选稿 | 中央候选稿编辑区或对照区 | 明确标注不会自动覆盖正文 |
| 大纲 | 中央 viewer/editor | 未接入时显示透明空态 |
| 故事经纬条目 | 右侧面板或中央详情 | 未接入 CRUD 时显示未接入状态 |
| 素材 | 中央 viewer | 后续补齐 |
| 发布报告 | 中央报告视图 | 后续补齐 |

## 与 AI 输出流程的关系

AI 生成结果应该先进入“生成章节”或“草稿候选”资源，而不是直接写入“已有章节”。

```text
AI 操作
  -> 生成章节 / 草稿候选
  -> 资源管理器出现新节点
  -> 用户打开阅读
  -> 用户确认处理方式
```

## 维护原则

1. 资源管理器只展示有明确来源的数据。
2. 未接入能力必须显示为未接入、只读或透明过渡。
3. 不要把内部文件结构直接暴露成用户资源树。
4. 用户看到的是小说对象，不是工程文件。
5. 候选稿、草稿、正式章节必须清楚区分。
