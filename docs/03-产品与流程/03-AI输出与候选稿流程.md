# AI 输出与候选稿流程

**版本**: v1.0.0  
**创建日期**: 2026-05-10  
**状态**: ✅ 当前有效  
**文档类型**: current

---

## 概述

NovelFork Studio 的核心安全原则：AI 生成结果不直接覆盖正式章节。所有 AI 输出通过"输出边界"（outputBoundary）分类，决定结果的去向和用户需要执行的下一步操作。

---

## AI 输出边界（outputBoundary）

每个写作动作声明一个输出边界，决定 AI 结果如何呈现给用户。

| 输出边界 | 说明 | 用户下一步 |
|---------|------|-----------|
| `candidate-artifact` | 结果进入候选稿容器 | 在候选稿列表中审核、接受/拒绝 |
| `draft-artifact` | 结果进入草稿容器 | 在草稿列表中查看、编辑 |
| `prompt-preview` | 只展示提示词预览，不产出内容 | 复制或显式应用 |
| `analysis` | 返回分析结果（审校/检测） | 查看分析面板 |
| `gate` | 需要确认门（如伏笔建议需确认） | 查看确认门，决定是否继续 |
| `async-start` | 异步启动，后台执行 | 等待事件通知或刷新 |

### 输出边界与写作动作的对应关系

```typescript
interface WritingActionDescriptor {
  id: string;                    // 动作唯一标识
  label: string;                 // 显示标签
  entry: string;                 // 入口路径或调用链
  outputBoundary: OutputBoundary; // 输出边界
  writesFormalChapter: boolean;  // 是否直接写入正式章节（始终为 false）
  capability: BackendCapability; // 后端能力状态
  chain?: readonly string[];     // Session-native 调用链
}
```

所有写作动作的 `writesFormalChapter` 均为 `false`，确保 AI 不会绕过候选稿流程直接修改正式章节。

## 候选稿生命周期

```
AI 生成 → candidate（待处理）
              ├── accepted（已接受）→ merge / replace / draft
              ├── rejected（已拒绝）
              └── archived（已归档）
```

### 状态定义

| 状态 | 说明 | 可执行操作 |
|------|------|-----------|
| `candidate` | 待处理，等待作者审核 | 接受、拒绝、归档、删除 |
| `accepted` | 已接受并应用 | 删除（受限） |
| `rejected` | 已拒绝 | 删除 |
| `archived` | 已归档备查 | 删除 |

### 接受动作

| 动作 | 说明 | 效果 |
|------|------|------|
| `merge` | 合并到章节 | 将候选稿内容追加到目标章节末尾 |
| `replace` | 替换章节 | 用候选稿内容完全覆盖目标章节 |
| `draft` | 另存为草稿 | 将候选稿保存为独立草稿，不影响正式章节 |

### 候选稿数据结构

```typescript
interface ChapterCandidateRecord {
  id: string;
  bookId: string;
  targetChapterId?: string;       // 目标章节（接受时写入的目标）
  title: string;
  source: string;                 // 来源标识（如 "session-native"）
  createdAt: string;
  updatedAt: string;
  status: "candidate" | "accepted" | "rejected" | "archived";
  contentFileName: string;        // 正文文件名
  metadata?: Record<string, unknown>;
  acceptedAction?: "merge" | "replace" | "draft";
  draftId?: string;               // 若 action=draft，关联的草稿 ID
}
```

### 候选稿 API

| 操作 | 方法 | 路径 |
|------|------|------|
| 列表 | GET | `/api/books/:bookId/candidates` |
| 接受 | POST | `/api/books/:bookId/candidates/:candidateId/accept` |
| 拒绝 | POST | `/api/books/:bookId/candidates/:candidateId/reject` |
| 归档 | POST | `/api/books/:bookId/candidates/:candidateId/archive` |
| 删除 | DELETE | `/api/books/:bookId/candidates/:candidateId` |

## 前端组件：CandidateActionsBar

候选稿操作栏，嵌入在候选稿 viewer 中。

### 组成

| 元素 | 说明 |
|------|------|
| 状态 Badge | 显示当前状态（待处理/已接受/已拒绝/已归档） |
| 元信息 | 来源、目标章节、创建时间 |
| 接受下拉菜单 | 三个选项：替换章节、合并到章节、另存为草稿 |
| 拒绝按钮 | 标记为 rejected |
| 归档按钮 | 标记为 archived |
| 删除按钮 | 二次确认后物理删除 |

### 交互规则

- 仅 `candidate` 状态显示接受/拒绝/归档按钮
- 删除按钮所有状态可用（accepted 状态受限）
- 操作进行中禁用所有按钮，显示 loading 状态
- 删除需要二次确认（"确认删除？" + 确认/取消）
- 操作失败显示错误提示

### Props 接口

```typescript
interface CandidateActionsBarProps {
  candidateId: string;
  bookId: string;
  status: CandidateStatus;
  source?: string;
  targetChapterId?: string;
  createdAt?: string;
  onAccept: (candidateId: string, action: "merge" | "replace" | "draft") => Promise<void>;
  onReject: (candidateId: string) => Promise<void>;
  onArchive: (candidateId: string) => Promise<void>;
  onDelete: (candidateId: string) => Promise<void>;
}
```

## 写作动作适配器（writing-action-adapter.ts）

适配器将后端返回的原始结果标准化为统一的 `NormalizedWritingActionResult`，供前端 UI 决策。

### 标准化结果类型

| kind | 说明 | nextStep |
|------|------|---------|
| `prompt-preview` | 提示词预览 | copy-or-explicit-apply |
| `generated` | 生成内容（需显式应用） | explicit-apply-required |
| `candidate` | 候选稿已创建 | review-candidate |
| `draft-write` | 草稿已写入 | review-draft |
| `async-started` | 异步任务已启动 | wait-for-events-or-refresh |
| `gate-blocked` | 确认门阻塞 | show-gate |
| `confirmation-required` | 需要工具确认 | show-confirmation |
| `unsupported` | 不支持或错误 | show-provider-or-tool-error |
| `analysis` | 分析结果 | show-analysis |

### 适配器方法

```typescript
createWritingActionAdapter(input) → {
  previewWritingMode(bookId, payload)       // prompt-preview
  applyWritingMode(bookId, payload)         // candidate-artifact
  startAsyncWriteNext(bookId, payload?)     // async-start
  startAsyncDraft(bookId, payload?)         // async-start
  generateHooks(bookId, payload)            // gate
  applyHook(bookId, payload)               // draft-artifact
  auditChapter(bookId, chapterNumber)       // analysis
  detectChapter(bookId, chapterNumber)      // analysis
  listSessionNativeConfirmations(sessionId) // confirmation-required
  confirmSessionNativeStep(sessionId, toolName, decision)
}
```

### 标准化逻辑

`normalizeWritingActionResult` 按以下优先级判断结果类型：

1. 请求失败 + 包含 `gate` → `gate-blocked`
2. 请求失败 + 无 gate → `unsupported`
3. 响应包含 `gate` → `gate-blocked`
4. `mode === "prompt-preview"` → `prompt-preview`
5. `mode === "generated"` + content → `generated`
6. `status` 为 writing/drafting/queued → `async-started`
7. 包含 candidate artifact（id） → `candidate`
8. `target === "candidate"` + resourceId → `candidate`
9. `target === "draft"` + resourceId → `draft-write`
10. `persisted === true` + file → `draft-write`
11. 其他 → `analysis`

## 安全原则

| 原则 | 实现方式 |
|------|---------|
| AI 不直接写正式章节 | 所有 writesFormalChapter = false |
| 候选稿必须人工审核 | candidate 状态需要显式 accept 操作 |
| 接受需要选择动作 | merge/replace/draft 三选一，无默认 |
| 画布禁止保存候选稿 | assertSaveable 拦截 candidate 类型 |
| 确认门阻塞 | gate 类型结果需要用户确认后才能继续 |
| 异步任务不静默完成 | async-start 需要用户等待或刷新确认 |

## Artifact Surfacing（实时生成内容呈现）

AI 生成内容通过 WebSocket 实时推送到前端，用户可以在生成过程中逐行看到产出。

### 推送机制

| 工具调用 | WebSocket 事件 | 说明 |
|---------|---------------|------|
| `candidate.create_chapter` | `session:tool-stream` | 通过 `onStreamChunk` 回调实时推送生成的章节内容 |
| `Write` / `Edit` 工具 | `session:tool-input-chunk` | 流式推送工具的输入参数（即将写入的内容） |

### 前端渲染流程

```
Agent 调用 candidate.create_chapter
  → onStreamChunk 逐块产出
  → WebSocket 发送 session:tool-stream 事件
  → 前端 ToolCallCard 自动展开
  → 逐行渲染生成内容（打字机效果）
  → 生成完成
  → 切换为完整内容视图 / diff 视图
```

### ToolCallCard 行为

| 阶段 | 显示内容 |
|------|---------|
| 生成中 | 自动展开卡片，逐行渲染流式内容，显示进度指示器 |
| 生成完成 | 显示完整内容；如果是对已有章节的修改，显示 diff 视图 |
| 生成失败 | 显示错误信息和重试按钮 |

### Write/Edit 工具的流式输入

当 Agent 调用 Write 或 Edit 工具时，工具的输入参数（即要写入的内容）通过 `session:tool-input-chunk` 事件流式推送。这让用户在工具执行前就能看到 Agent 打算写入什么内容，提供透明度和可预见性。
