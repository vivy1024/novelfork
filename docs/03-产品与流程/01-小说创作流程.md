## 本文定位

本文描述 NovelFork 当前推荐的小说创作流程。它是产品层流程文档，不是逐端点 API 文档，也不是把所有规划能力都声明为已完成。

NovelFork 的创作流程围绕“正式正文”和“可确认的中间产物”展开：作者可以手写、编辑、调用 AI、检查资料，但 AI 结果必须先成为候选稿或草稿，不能直接覆盖正式章节。

## 主流程总览

```text
创建或打开作品
  -> 选择卷 / 章节 / 候选稿 / 草稿 / 资料
  -> 选择经纬模板，整理故事经纬
  -> 在正文编辑区编辑正式章节
  -> 可选：问卷中心 / 生成前追问补齐创作意图
  -> 调用 AI 生成候选稿或草稿
  -> 阅读候选稿
  -> 用户确认合并、替换、另存或放弃
  -> 写作工具、AI 味报告和故事经纬辅助检查
  -> 保存正式正文
  -> 后续发布检查或导出
```

## 关键对象

| 对象 | 含义 | 当前口径 |
|---|---|---|
| 作品 | 一本小说的根对象，承载章节和资料 | current |
| 卷 | 章节分组，适合长篇网文结构 | planning |
| 正式章节 | 用户确认后的正文，是作品内容事实源 | current |
| 候选稿 | AI 生成或改写产生的待确认内容 | current |
| 草稿 | 暂存想法、备选片段或不立即合并的文本 | planning |
| 大纲 | 作品、卷、章节的结构规划 | planning |
| 故事经纬 | 人物、地点、势力、伏笔、世界规则等连续性资料 | current/transparent 混合，见能力矩阵 |
| 问卷回答 | 用户在生成前填入的题材、卖点、目标、冲突等意图 | current |
| 生成前追问 | 系统基于经纬资料和章节位置提出的本章关键问题 | current |
| 核心设定变更 | 对主前提、主线冲突、世界规则、人物弧的可审计变更 | current |
| 写作工具结果 | 节奏、对话、健康、AI 味、冲突、语气等辅助判断 | current/transparent 混合 |

## 阶段 1：作品和章节准备

用户进入工作台后，首先选择作品。作品是资源树根节点，下面会逐步呈现：

- 已有章节。
- 生成章节候选。
- 大纲入口。
- 故事经纬入口。
- 后续规划中的草稿、素材、发布报告。

当前最稳定的写作入口是正式章节：可以从 Workspace 顶部新建章节，也可以选择已有章节，在中央编辑器中打开正文并编辑。

如果是新建作品，推荐先选择经纬模板：基础模板建立人物、事件、设定、章节摘要等栏目；增强模板追加伏笔、名场面、核心记忆等栏目；题材推荐模板只作为候选结构，不代表固定标准。

## 阶段 2：正文编辑

正式章节是用户确认后的文本，不等同于 AI 输出。正文编辑区应该展示章节标题、章节状态、正文、保存状态和错误反馈。

编辑过程的基本状态：

| 状态 | 含义 |
|---|---|
| 已保存 | 当前编辑器内容已写入章节存储 |
| 未保存 | 用户已修改正文，但尚未保存 |
| 保存中 | 正在调用章节保存路径 |
| 保存失败 | API 或存储失败，正文应保留，错误原因应显示 |

## 阶段 3：引导式生成准备

引导式生成的目标是先把本章真正需要 AI 理解的意图问清楚，再进入生成。它包含三类当前已实现的底座能力：

| 能力 | 在流程中的位置 | 结果 |
|---|---|---|
| 问卷中心 | 新书初始化或资料补齐时 | 把卖点、目标、冲突、风格等答案映射到经纬资料 |
| 生成前追问 | 调用章节生成前 | 根据冲突升级、伏笔回收窗口等规则提出本章问题 |
| 核心设定变更 | 发现主前提、世界规则、人物弧需要改动时 | 记录提议、影响范围、接受或拒绝状态 |

这些能力的共同边界：它们改变的是上下文、资料或审计记录，不直接覆盖正式正文。

```text
准备生成
  -> 读取故事经纬和章节上下文
  -> 可选：填写问卷或回答生成前追问
  -> 可选：提出/接受核心设定变更
  -> 形成本次 AI 生成上下文
```

## 阶段 4：AI 生成候选稿

当用户调用 AI 写作能力时，输出不应直接进入正式正文，而应进入候选稿或草稿候选。

```text
用户触发 AI 操作
  -> 系统读取章节上下文、模型配置、故事经纬和本章作者指示
  -> 生成候选稿
  -> 候选稿进入资源树
  -> 用户打开候选稿阅读和处理
```

这样设计的原因：

1. 保护正式正文，避免 AI 误覆盖。
2. 允许用户比较多个版本。
3. 让 AI 生成、人工判断、正式写入之间有清楚边界。
4. 方便后续做版本记录、草稿库和发布审校。

## 阶段 5：候选稿处理

候选稿不是正文。用户需要明确选择处理动作：

| 动作 | 适合场景 | 对正式正文的影响 |
|---|---|---|
| 合并到正式章节 | AI 生成的是可用补充段落 | 修改正式正文，需要保存 |
| 替换正式章节 | AI 生成的是完整重写版本 | 高风险修改，需要用户确认 |
| 另存为草稿 | 有参考价值但暂不采用 | 不影响正式正文 |
| 放弃候选稿 | 内容不可用或已过期 | 不影响正式正文 |

## 阶段 6：写作工具辅助

写作工具不是替作者做决定，而是提供辅助判断：

- 节奏分析：检查章节节奏、段落密度、推进速度。
- 对话分析：检查对白比例和说话密度。
- 书籍健康：整合作品字数、章节状态、敏感词、矛盾记录等指标。
- AI 味报告：用本地规则和可选朱雀配置扫描 AI 痕迹，给出改写建议。
- 冲突图、人物弧、语气检查：帮助发现连续性和风格问题。
- hooks：把新的伏笔或待回收信息写入 story truth 相关文件。

缺少事实源的指标必须显示 unknown、未接入或透明过渡，不能固定显示满分。

## 阶段 7：故事经纬维护

故事经纬负责长期资料：人物、地点、势力、物品、伏笔、世界规则、前文摘要等。

它在流程中的作用：

1. 写作前：提供章节上下文、人物设定和可见性裁剪。
2. 写作中：辅助检查人物、地点、伏笔和核心设定一致性。
3. 写作后：沉淀新事实、补充 hooks、更新资料库。
4. 生成前：为问卷和追问提供事实基础。

当前结构化 API、模板和上下文装配已经可用；当前工作台内完整编辑闭环仍在补齐，因此不能把所有分类详情写成已经完成的完整资料库。

## 流程中的禁止事项

- 禁止 AI 结果直接覆盖正式正文。
- 禁止把候选稿显示成正式章节。
- 禁止把 prompt preview 写成真实写入。
- 禁止把问卷、追问或变更历史写成已自动改正文。
- 禁止把未接入能力显示为成功。
- 禁止把参考资料或历史路线当成当前产品承诺。

## 当前最小可用闭环

当前能作为最小闭环理解的是：

```text
选择作品
  -> 新建章节或打开已有章节
  -> 编辑正文
  -> 保存正文
  -> 可选：问卷或生成前追问补上下文
  -> 查看候选稿
  -> 通过明确动作处理候选稿
  -> 用写作工具和 AI 味报告辅助检查
```

完整的长篇创作闭环还需要继续补齐完整候选稿状态刷新、完整大纲、故事经纬工作台 CRUD、素材、发布报告和导出。
