## 本文定位

本文定义用户侧“故事经纬 / 经纬”的产品概念、资料范围和在创作流程中的作用。它不声明工作台内完整资料库 CRUD 已经完成；具体能力状态以 [Studio能力矩阵](../01-当前状态/02-Studio能力矩阵.md) 为准。

## 用户侧命名

用户界面和用户文档统一使用“故事经纬 / 经纬”。

“经纬”表达的是故事的纵横脉络：人物关系、地点空间、势力结构、伏笔回收、世界规则和前文摘要共同支撑长篇小说连续性。

## 资料范围

| 分类 | 用途 | 例子 |
|---|---|---|
| 人物 | 维护角色身份、目标、关系、状态变化 | 主角、师父、反派、配角 |
| 地点 | 维护场景、地理关系、危险程度 | 宗门、城市、秘境、战场 |
| 势力 | 维护组织结构、利益关系、冲突关系 | 门派、公司、家族、联盟 |
| 物品 | 维护重要道具、资源、装备、线索 | 法宝、钥匙、合同、遗物 |
| 伏笔 | 维护已埋线索、待回收问题和回收章节 | 未解释异象、隐藏身份、约定事件 |
| 世界规则 | 维护力量体系、社会规则、经济规则 | 境界、科技、法律、交易规则 |
| 前文摘要 | 为后续章节提供上下文压缩 | 已发生事件、人物状态、未解决冲突 |
| 作品前提 | 维护一句话卖点、主线问题和核心冲突 | 主角为什么必须行动、作品承诺给读者的爽点 |
| 人物弧 | 维护角色长期变化和阶段目标 | 从弱到强、从复仇到守护、从逃避到承担 |
| 核心记忆 | 维护对后文影响长期有效的事实 | 师门血仇、禁术代价、不可公开的身份 |

## 在创作流程中的作用

```text
写作前
  -> 读取人物、地点、伏笔、前文摘要，构建上下文
  -> 可选：用问卷补齐作品前提、人物目标和主线冲突

生成前
  -> 根据冲突升级、伏笔回收窗口等规则提出本章追问

写作中
  -> 辅助检查章节内容是否违背既有设定

写作后
  -> 沉淀新事实，新增伏笔，更新人物和地点状态
  -> 记录核心设定变更和受影响章节
```

故事经纬不是简单的百科词条，而是服务章节生产、生成前追问、AI 上下文装配和连续性审计的资料系统。

## 与正式正文的关系

正式正文是最终作品内容；故事经纬是辅助资料。两者不应混淆：

- 正文写“发生了什么”。
- 经纬写“这件事对故事长期状态意味着什么”。
- 正文可以由用户编辑和确认。
- 经纬可以由用户维护，也可以由工具辅助提取，但需要可审查。

## 与 AI 的关系

AI 使用故事经纬时应遵循：

1. 读取经纬资料作为上下文。
2. 根据可见性规则裁剪本章真正需要的资料。
3. 可选地使用问卷回答或生成前追问答案补充作者意图。
4. 生成章节候选稿或修改建议。
5. 不自动把新设定写回资料库。
6. 需要沉淀新事实时，应形成可审查的更新建议。

当前结构化经纬底座支持栏目和条目、模板应用、上下文预览、问卷映射和核心设定变更记录。它们都属于“给 AI 更可靠上下文”的能力，不等同于自动写正文。

## 当前已知边界

| 能力 | 当前口径 |
|---|---|
| 结构化栏目和条目 API | 已有真实 route、仓储和测试覆盖 |
| 经纬模板应用 | 已可按空白、基础、增强和题材推荐写入栏目结构 |
| 经纬上下文预览 | 已可把启用栏目裁剪为 AI 上下文 |
| 问卷中心 | 已有模板、回答保存、AI 建议和映射底座；页面入口需按能力矩阵判断 |
| 生成前追问 | 已有启发式引擎和 route；当前覆盖冲突升级、伏笔回收窗口等规则 |
| 核心设定变更 | 已有提出、接受、拒绝和影响章节记录能力 |
| 工作台右侧经纬面板 | 已有基础展示入口，但部分路径仍需做实 |
| 人物/地点/势力/物品/伏笔/世界规则完整工作台 CRUD | 仍需工作台内接入 |
| 与章节编辑的深度联动 | 仍需补齐 |
| 自动事实抽取和回写 | 需要审查和确认机制，不能静默写入 |

## 维护原则

- 用户侧只说“故事经纬 / 经纬”。
- 技术兼容名只允许出现在代码、数据库、迁移说明或历史引用中。
- 问卷、追问和变更历史必须标明“影响上下文/资料”，不能写成自动改正文。
- 未接入的分类详情必须显示透明状态。
- 不得把空态、示例数据或未接入接口写成真实资料库能力。
