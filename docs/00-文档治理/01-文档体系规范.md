# 文档体系规范

**版本**: v1.0.0
**创建日期**: 2026-04-28
**更新日期**: 2026-04-28
**状态**: ✅ 当前有效
**文档类型**: current

---

## 1. 新文档体系

NovelFork 文档中心采用当前事实优先的 Hub-and-Spoke 结构：

```text
docs/
├── 00-文档治理/
├── 01-当前状态/
├── 02-用户指南/
├── 03-产品与流程/
├── 04-架构与设计/
├── 05-开发者指南/
├── 06-API与数据契约/
├── 07-运行运维/
├── 08-测试与质量/
├── 90-参考资料/
└── 99-历史归档/
```

## 2. 文档类型

| 文档类型 | 状态 | 用途 | 可进入根 README 当前事实入口 |
|---|---|---|---:|
| `current` | ✅ 当前有效 | 当前事实、当前使用方式、当前架构事实 | 是 |
| `planning` | 📋 规划中 | 已批准或正在执行的计划 | 有条件 |
| `reference` | 📚 参考资料 | 外部项目、竞品、调研、经验参考 | 否 |
| `archived` | 🗄️ 历史归档 | 历史记录和旧口径 | 否 |
| `deprecated` | ⚠️ 已废弃 | 已被替代的口径 | 否 |

## 3. Header 规范

每篇 `docs/**/*.md` 必须使用以下 header：

```markdown
# 标题

**版本**: vX.Y.Z
**创建日期**: YYYY-MM-DD
**更新日期**: YYYY-MM-DD
**状态**: ✅ 当前有效 | 📋 规划中 | 📚 参考资料 | 🗄️ 历史归档 | ⚠️ 已废弃
**文档类型**: current | planning | reference | archived | deprecated

---
```

## 4. Single Source of Truth

- 当前状态唯一入口：`../01-当前状态/01-项目当前状态.md`
- Studio 能力矩阵：`../01-当前状态/02-Studio能力矩阵.md`
- API 总览唯一入口：`../06-API与数据契约/01-Studio API总览.md`
- 运行方式唯一入口：`../07-运行运维/01-当前运行与启动方式.md`
- 测试状态唯一入口：`../08-测试与质量/01-当前测试状态.md`

其他文档引用这些事实源，不复制长正文。

## 5. 命名规则

- 顶层目录使用 `XX-名称/`。
- 当前事实和正式指南使用编号文件，例如 `01-项目当前状态.md`。
- 参考资料和历史归档可以按主题子目录分组。
- 删除旧文档前必须确认其内容已合并、归档或记录在迁移映射中。

## 6. 与旧规则的关系

旧规则原文保留下方，供追溯；当前执行以本文件前半部分的新体系为准。

## 1. 目录结构

```text
docs/
├── 00-文档命名整理规则.md
├── 01-快速开始/
├── 02-核心架构/
├── 03-代码参考/
├── 04-开发指南/
├── 05-API文档/
├── 06-部署运维/
├── 07-测试报告/
└── README.md
```

## 2. 命名规则

- 一级目录：`XX-名称/`
- 二级目录：`XX-名称/`
- 文件：`XX-标题.md`
- 使用中文标题，避免括号、状态后缀、版本号后缀
- 新增文档时编号接当前最大值 +1
- 删除文档后不强制重排旧编号，避免大规模 git diff

## 3. Header 规范

每篇文档开头必须包含：

```markdown
# 标题

**版本**: vX.Y.Z
**创建日期**: YYYY-MM-DD
**更新日期**: YYYY-MM-DD
**状态**: ✅ 已完成 | 🔄 持续更新 | 📋 规划中 | ⚠️ 已废弃

---
```

## 4. 分类原则

| 内容类型 | 目录 | 判断标准 |
|----------|------|----------|
| 快速上手 | 01-快速开始 | 回答“怎么开始” |
| 架构设计 | 02-核心架构 | 回答“是什么/为什么” |
| 代码实现/外部参考 | 03-代码参考 | 回答“怎么实现” |
| 使用、调研、迁移方案 | 04-开发指南 | 回答“怎么做/怎么迁” |
| HTTP/SSE/WebSocket 接口 | 05-API文档 | 接口定义 |
| 运行、部署、启动、版本变更 | 06-部署运维 | 回答“怎么运行/怎么排查” |
| 测试、评审、历史归档 | 07-测试报告 | 测试结果、评估、归档 |

## 5. README 规范

每个目录必须有 `README.md`，至少包含：
- 目录说明（1-2 句）
- 文件列表（与实际一致）

根 `docs/README.md` 额外包含：
- 项目概述
- 完整目录树
- 快速导航

## 6. 清理原则

- 已整合旧文档：直接删除，不保留“已整合”跳转页
- 历史分析类：移入 `07-测试报告/02-历史归档/`
- 过渡期文档：明确标注“过渡文档”，不能冒充最终口径
- 根目录中的重要历史文档，优先迁入 docs 体系统一管理
